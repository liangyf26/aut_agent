# 规范执行审计细则

## 提取：什么算「可机械核验的约定」

读层级规则文件时，找**谈论文件、目录、命名、必备内容的祈使句**。判断标准：能不能写成一条 shell 命令来核验？能 → 提取；不能（如"沟通要结论先行"）→ 跳过。

| 类别 | 规则句式特征 | 核验手段 |
|---|---|---|
| 命名约定 | "文件夹名必须 X"、"禁止 Y 前缀/后缀" | `ls` + 逐个名字比对 |
| 必备文件 | "每个项目必须有 X"、"顶部必须声明 Y" | 存在性检查 + 读文件头 |
| 同源约束 | "A 必须软链到 B"、"永远只编辑 B" | `readlink`、`file` |
| 红线 | ".gitignore 必须含 X"、"密钥不进代码" | `grep` |
| 目录纪律 | "根目录不允许裸放文件"、"X 只能放在 Y 下" | `ls` + 类型判断 |
| 声明一致 | "CLAUDE.md 里写的启动命令/URL/路径必须真实" | 对照代码与配置核验 |

## 核验范围

- 默认：当前项目 + 直接上级工作空间的同级项目**名字层面**（名字核验是 `ls` 一眼的事，成本为零；同级项目的**内容**不审）
- 用户说「审全部」/「整个工作空间」：逐项目跑存在性与同源检查
- 全局配置文件：只做方向二（死引用 / 矛盾），不做新增

## 处置分级（完整版）

### 直接修（安全、可逆、纯补齐）

- 补 `AGENTS.md → CLAUDE.md` 软链
- 给缺 CLAUDE.md 且已有可运行代码的项目建最小脚手架
- `.gitignore` 补红线条目（`.env`、`.env.local` 等）
- 规则文件里指向**确认已删除**项目的引用：清掉
- 规则文件内的相对时间、明显笔误

### 待你拍板（破坏性 / 有外部影响 / 需要权威判断）

- **目录重命名**（命名违规的修复）：会破坏 git remote 对应关系、部署脚本路径、其他文档里的引用
- **删除文件 / 目录**：除非规则明文授权
- **合并两份内容不一致的 CLAUDE.md / AGENTS.md**：需要人工确认哪边是权威
- **规则漂移**：规则说 X，但所有项目实际做 Y 且运转良好
- **规则矛盾**：上下级打架且无法从现状判断哪边现行

### 摘要格式

```
### 规范审计
- 自动修复：
  - xxx 补 AGENTS.md 软链（依据：全局 CLAUDE.md「同源」节）
- 待你拍板：
  - xxx 目录名违反 kebab-case（依据：code/CLAUDE.md 命名约定）。
    建议改为 xxx；影响：...
```

每条违规都带**依据出处**和**影响说明**。

## 常见误判提醒

- **`work/` 下沿用公司既定名称的项目不算命名违规**
- **sandbox / 一次性目录**通常被工作空间规则明确豁免
- **规则引用的路径在另一台机器上**（多设备同步场景）：`ls` 不在 ≠ 已删除
- 报告违规前**重读一遍规则原文**
