# 测试手册第二阶段 v3

更新时间：2026-06-26

本文说明第二阶段 v3 运行中心的本地运行、真实浏览器接入、run 创建、自动评测、复盘、下一轮、报告和排错方法。

第二阶段 v3 的推荐入口是图形界面：

```text
http://localhost:4173/stage2
```

旧的 Python CLI、系统地图向导和模板脚本仍可用于排查或专项验证，但日常测试应优先从“第二阶段运行中心”创建和操作 run。

## 1. 运行前准备

### 1.1 启动运行中心服务

在仓库根目录执行：

```powershell
npm run dev
```

启动后访问：

```text
http://localhost:4173/stage2
```

如果只打开 `http://localhost:4173`，页面可能仍显示第一阶段内容；第二阶段 v3 建议直接访问 `/stage2`。

### 1.2 启动已登录 Chrome 调试端口

第二阶段真实浏览器模式需要连接一个已经登录被测系统的 Chrome。推荐先关闭已有 Chrome，再用独立用户目录启动：

```powershell
chrome.exe --remote-debugging-port=9222 --user-data-dir="D:\Temp\stage2-chrome-profile"
```

如果 `chrome.exe` 不在 PATH 中，可使用常见安装路径：

```powershell
& "C:\Program Files\Google\Chrome\Application\chrome.exe" --remote-debugging-port=9222 --user-data-dir="D:\Temp\stage2-chrome-profile"
```

启动后在这个 Chrome 中人工登录被测系统，并停留在系统首页或登录后的入口页。

用浏览器或 PowerShell 验证 CDP 是否可用：

```text
http://localhost:9222/json/version
```

可用时应看到包含 `Browser`、`Protocol-Version`、`webSocketDebuggerUrl` 的 JSON。

### 1.3 可选：配置大模型 profiles

运行中心会读取：

```text
config/stage2-model-profiles.json
```

如果文件不存在，界面会显示未配置模型 profiles。此时仍可创建 run、执行真实浏览器链路和规则复盘，但不会有多模型 AI 参与遍历、分析和报告点评。

最小示例：

```json
{
  "schema_version": "stage2_model_profiles.v1",
  "profiles": [
    {
      "id": "model_a",
      "label": "Model A",
      "provider": "openai_compatible",
      "baseUrl": "https://api.example.com/v1",
      "apiKeyEnv": "MODEL_A_API_KEY",
      "model": "model-a-name",
      "browserUseMode": "chatopenai"
    },
    {
      "id": "model_b",
      "label": "Model B",
      "provider": "openai_compatible",
      "baseUrl": "https://api.example.com/v1",
      "apiKeyEnv": "MODEL_B_API_KEY",
      "model": "model-b-name",
      "browserUseMode": "chatopenai"
    }
  ]
}
```

启动服务前设置环境变量：

```powershell
$env:MODEL_A_API_KEY="..."
$env:MODEL_B_API_KEY="..."
npm run dev
```

也可以通过环境变量指定配置文件位置：

```powershell
$env:STAGE2_MODEL_PROFILES_PATH="D:\Project\702-aut_agent\config\stage2-model-profiles.json"
```

## 2. 推荐主流程

### 2.1 进入第二阶段运行中心

打开：

```text
http://localhost:4173/stage2
```

左侧是 run 创建表单和 run 列表，右侧是当前 run 的操作区、指标卡、阶段状态、复盘、人工处理、报告和 artifacts。

### 2.2 填写 run 信息

左侧表单建议这样填写：

| 字段 | 填写建议 |
| --- | --- |
| 系统名称 | 被测系统名称，例如 `追本溯源管理平台` |
| 首页 URL | 已登录后可访问的首页 URL，例如 `https://www.zbsykj.com:19096/index` |
| CDP URL | 默认 `http://localhost:9222` |
| 大模型预检 | 勾选需要参与测试的大模型；可多选，多选时依次测试 |
| 执行模式 | 正式测试选 `真实浏览器：连接 CDP/Playwright 后执行` |
| 执行安全策略 | 默认 `低风险只读`；测试环境可选 `测试环境全权限` |
| 测试账号与环境 | 写账号角色、权限范围、登录方式、验证码/扫码说明 |
| 探索范围 | 写优先页面、排除页面、禁止动作、前置数据限制 |
| 优先目标页面 | 每行一个，例如 `线上备案申请` |
| 人工豁免目标 | 对确认无需追踪的目标，每行一个 |
| 最大页面 | 第一轮/本轮最多探索页面数，建议先 30 |
| 最大轮次 | 建议先 2 到 5 |

如果选择 `测试环境全权限`，必须勾选确认框。该模式会允许提交、保存、删除、审批等副作用动作，只应在可重置、可污染的测试环境使用。

### 2.3 创建 run

点击：

```text
创建 run
```

成功后，run 会出现在左侧 `Run 列表`。右侧顶部会显示：

新建 run 时，右侧 `操作反馈` 会清空旧 run 的反馈，并以“新 run 开始”标识当前操作上下文，避免把上一轮错误误认为当前 run 状态。

- run 类型：可操作 v3 run / 只读 v3 run / 本地草稿
- 执行模式：契约占位 / 真实浏览器
- 安全策略：低风险只读 / 测试环境全权限
- 当前阶段、阻塞原因、下一步动作

只有“可操作 v3 run”可以启动、复盘、进入下一轮和生成报告。本地草稿表示后端 v3 API 创建失败，只能作为界面临时记录，不能执行。

### 2.4 启动真实浏览器自动评测

确认右侧执行模式选择为：

```text
真实浏览器
```

点击：

```text
开始自动评测
```

运行中心会先检查 CDP，再调用 Python v3 执行链路。成功后会逐步写入以下结构化产物：

```text
artifacts/stage2/runs/<run_id>/
```

常见核心产物：

| 产物 | 说明 |
| --- | --- |
| `run_manifest.json` | run 元信息、状态、轮次 |
| `input_config.json` | 创建 run 时的输入配置 |
| `progress_events.jsonl` | 进度事件 |
| `current_status.json` | 当前阶段、阻塞原因、下一步动作 |
| `menu_entries.json` | 菜单入口发现结果 |
| `page_entries.json` | 页面入口发现结果 |
| `feature_points.json` | 功能点识别结果 |
| `generated_test_cases.json` | 执行型测试用例 |
| `execution_results.json` | 执行结果、功能点标识、证据、失败原因 |
| `screenshots_index.json` | 截图索引，包含截图 ID、阶段、标签和文件路径 |
| `screenshots/` | 实际截图文件目录 |
| `round_analysis.json` | 本轮复盘 |
| `next_round_plan.json` | 下一轮计划 |
| `human_tasks.json` | 需要人工处理的任务 |
| `model_comparison.json` | 多模型对比结果 |
| `promotion_candidates.json` | 项目级沉淀候选 |
| `reports/run_report.json` | 结构化总体报告 |
| `reports/run_report.md` | 可阅读总体报告 |

## 3. 如何理解运行状态

### 3.1 契约占位

`契约占位` 表示只生成 v3 产物契约和最小闭环，不连接真实浏览器，不代表完成了真实测试。

用途：

- 验证运行中心 API 是否可用
- 验证产物结构和报告生成链路
- 前后端联调

正式测试不要停留在契约占位，应使用 `真实浏览器` 模式。

### 3.2 真实浏览器

`真实浏览器` 表示运行中心会通过 CDP 连接本机 Chrome，并由 Python/Playwright 执行真实页面发现、功能点识别和用例执行。

如果界面显示“真实浏览器未就绪”，优先检查：

1. `http://localhost:9222/json/version` 是否能打开。
2. Chrome 是否用 `--remote-debugging-port=9222` 启动。
3. 运行中心表单里的 CDP URL 是否是 `http://localhost:9222`。
4. 被测系统是否已在该 Chrome 中完成登录。
5. 是否启动了多个 Chrome，实际登录窗口不是 9222 这个调试实例。

### 3.3 页面入口数和浏览器 target 数

界面里可能同时看到：

- 浏览器 target 数：Chrome 当前可调试 tab/frame 数。
- 菜单入口数：第一轮遍历发现的菜单节点数。
- 页面入口数：确认可进入、可记录为业务页面的入口数。

这三者不是同一个概念。浏览器 target 数不代表发现了多少业务页面。

### 3.4 当前阶段“已计划”

`已计划` 通常表示下一轮计划已经生成，但尚未真正执行，或正在等待运行中心提交下一轮动作。

处理方式：

1. 查看右侧 `下一步动作`。
2. 如果下一步动作要求继续，点击 `进入下一轮`。
3. 如果按钮不可用或返回阻塞，查看 `阻塞原因`。
4. 如果阻塞原因是 `human_approval_required` 或 `wait_human_review`，先到 `人工处理` tab 完成任务或审批，再继续。
5. 如果下一轮计划是 `stop_goal_completed`，通常说明当前范围已完成；但若 `round_analysis.json` 显示用户优先目标已发现但页面不可达、白屏或缺少功能点覆盖，应按“未覆盖目标”处理并继续下一轮。

## 4. 每轮结束后怎么做

### 4.1 复盘

点击：

```text
规则复盘
```

当前按钮名为“规则复盘”，表示运行中心至少会执行确定性复盘。如果本 run 配置了可用模型 profiles，相关模型结果会进入 `model_comparison.json` 和报告里的模型效果对比。

复盘重点看：

- 覆盖摘要：发现多少菜单入口、页面入口、功能点、用例。
- 失败摘要：失败、跳过、阻断、需人工复核的原因。
- 证据质量：是否有截图、动作日志、页面反馈。
- 未命中目标：例如用户写了“线上备案申请”，但本轮未发现。
- 下一轮计划：是否继续、继续找什么、是否需要人工审批。

### 4.2 人工处理

进入：

```text
人工处理
```

可能出现的人工任务：

- 选择优先测试页面。
- 审核功能点。
- 录制实际操作路径。
- 更正测试数据。
- 批准下一轮。
- 豁免确认找不到或不再追踪的页面。

人工处理应在界面完成，不要直接修改 JSON。界面提交后会写入 `human_tasks.json` 或对应处理产物，后续轮次会消费这些结构化结果。

### 4.3 进入下一轮

点击：

```text
进入下一轮
```

进入下一轮不是简单改状态，而是根据 `next_round_plan.json` 再次启动执行链路。

如果提示当前目标已完成：

- 先生成报告。
- 如果用户优先目标已经出现在菜单或页面入口中，但状态是不可达、白屏、未生成可验证功能点或未产生执行结果，继续下一轮而不是结束 run。
- 如果还要继续扩大范围，创建新 run，并增加 `最大页面`、`最大轮次` 或补充 `优先目标页面`。

如果提示需要人工审批：

- 到 `人工处理` tab 完成任务。
- 再点击 `进入下一轮`。

## 5. 生成和阅读报告

点击：

```text
生成报告
```

报告位置：

```text
artifacts/stage2/runs/<run_id>/reports/run_report.md
artifacts/stage2/runs/<run_id>/reports/run_report.json
```

报告应包含：

- 运行概况。
- 覆盖与结果矩阵。
- 未执行、阻断、需复核区域。
- 页面入口、功能点、执行型测试用例和执行结果。
- 多模型效果对比。
- 项目级沉淀候选。
- 人工确认项。

`promotion_candidates.json` 中的项目级候选可以作为后续模板、定位器、字段映射、等待策略、测试数据建议的来源。平台级候选必须人工审核后才能晋升，不会自动应用。

## 6. 新系统接入向导和 v3 运行中心的关系

页面中还有“三段式向导”，用于系统地图和模板收敛：

1. 系统地图。
2. 模板收敛。
3. 验证与汇总。

它适合做新系统首次接入、模板排查或手工收敛，不是 v3 run 主闭环的替代品。

推荐使用方式：

- 日常自动评测：优先使用 `/stage2` 的第二阶段运行中心创建 v3 run。
- 首次接入陌生系统：可先用“三段式向导”探索系统地图，再回到 v3 运行中心创建真实 run。
- 定位复杂页面问题：可使用向导生成模板修订清单，再让 v3 run 消费修订后的经验。

## 8. 常见问题

### 8.1 点击“创建 run”没反应

先看页面顶部或操作反馈区是否有错误。

常见原因：

- 没填系统名称或首页 URL。
- 选择了测试环境全权限，但未勾选确认。
- 后端服务未启动或不是当前仓库的服务。
- 浏览器打开的是旧缓存页面，刷新后重试。

### 8.2 点击“开始自动评测”后没有真实执行

检查：

- run 是否是“可操作 v3 run”，不是本地草稿。
- 执行模式是否选择 `真实浏览器`。
- CDP 是否可用。
- Chrome 是否已登录被测系统。
- Python 是否可用。
- 运行中心操作反馈区是否显示执行器错误。

如果执行模式是 `契约占位`，不会真实打开页面。

### 8.3 第一轮没有发现二级菜单

预期目标是：第一轮先完成菜单项入口遍历，包含一级菜单和可展开的二级菜单；第二轮再逐个菜单页面深入探索。

若没有发现二级菜单：

1. 确认登录账号确实能看到该菜单。
2. 在 `优先目标页面` 中写入目标，例如 `线上备案申请`。
3. 增大 `最大页面` 和 `最大轮次` 后新建 run。
4. 查看 `menu_entries.json`、`page_exploration_log.jsonl`、`round_analysis.json` 中的失败原因。
5. 与 `demo/suyuan_test3.py` 的发现行为对照，判断是运行中心桥接问题还是页面交互策略不足。

### 8.3.1 第一轮出现重复首页或白屏页面

第一轮可能从菜单文本、首页标题、URL 或页面 chrome 中得到多个看似不同的入口。运行中心应按规范化 URL 去重，例如 `/` 与 `/index` 只保留一个可信入口。

如果进入某个入口后页面全白：

1. 先打开 `screenshots_index.json`，找到对应截图 ID。
2. 再打开 `screenshots/<screenshot_id>.png` 或索引中的绝对路径确认页面状态。
3. 查看 `page_exploration_log.jsonl` 中该入口的 `failure_reason`。
4. 若该入口来自顶部工具、字号选择、个人中心、退出登录等非业务 chrome，应排查菜单扫描过滤规则。
5. 若该入口是用户优先目标但仍不可达，应在 `round_analysis.json` 中标记为已发现但未覆盖，并进入下一轮继续处理。

### 8.4 AI 复盘没有输出

先区分两件事：

- `规则复盘`：不依赖大模型，也应产生 `round_analysis.json` 和 `next_round_plan.json`。
- AI/多模型复盘：依赖 `config/stage2-model-profiles.json` 和模型连通性。

如果没有配置模型，报告里不会出现有效的多模型点评。配置后刷新“大模型预检”，确认至少一个 profile 可用。

### 8.5 “进入下一轮”后仍是已计划

先查看：

```text
artifacts/stage2/runs/<run_id>/next_round_plan.json
artifacts/stage2/runs/<run_id>/current_status.json
```

如果需要人工审批，先处理人工任务。

如果 `decision` 是 `stop_goal_completed`，当前范围已结束，不应继续点下一轮；需要新建更大范围 run。

如果 `decision` 是 `auto_continue` 但仍无法执行，记录操作反馈区错误和 `current_status.json`，再排查 Python/真实浏览器链路。

### 8.6 如何新建更大探索范围的 run

新建 run 时调整：

- `最大页面`：例如从 30 提到 60。
- `最大轮次`：例如从 2 提到 5。
- `优先目标页面`：写入一个或多个关键页面。
- `探索范围`：补充“必须遍历全部一级菜单和二级菜单”等约束。
- `人工豁免目标`：只填写确认无需追踪的页面，避免误豁免关键入口。

### 8.7 如何允许提交、删除、审批等副作用动作

在创建 run 时：

1. `执行安全策略` 选择 `测试环境全权限`。
2. 勾选 `我确认这是可执行副作用动作的测试环境`。
3. 在 `测试账号与环境` 写明这是可污染、可重置的测试环境。

该模式会把允许动作写入 run 配置，并传给后端执行链路。仍建议先用小范围页面验证，再扩大到全系统。

## 9. 验收一个 v3 run 是否有效

一个有效的真实浏览器 v3 run 至少应满足：

- `preflight_result.json` 显示真实浏览器链路可用。
- `menu_entries.json` 记录菜单遍历结果，且菜单入口数与界面说明一致。
- `page_entries.json` 至少包含实际业务页面入口。
- `feature_points.json` 至少包含可解释的功能点。
- `execution_results.json` 有执行状态、动作日志、页面反馈或截图证据。
- `round_analysis.json` 说明覆盖、失败、证据质量和下一轮方向。
- `next_round_plan.json` 明确继续、停止或人工处理原因。
- `reports/run_report.md` 可直接阅读，并能解释未执行、阻断和风险区域。

如果这些产物只有占位内容，说明该 run 没有完成真实测试，应优先排查真实浏览器和 Python 执行链路。
