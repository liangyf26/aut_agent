# 第二阶段新系统接入测试手册

更新时间：2026-06-24

## 1. 文档目标

本文档用于指导团队成员把一个新的真实 Web 系统接入到第二阶段原型中，并完成第一轮可复核的自动化验证。

本文档面向第一次参与该工作的同事，目标是让团队按统一步骤完成以下事情：

1. 先让平台自动探索目标系统，生成菜单/页面入口清单。
2. 读取系统地图树和页面类型初分，减少人工盲猜。
3. 再选择首批要验证的页面，收敛成具体模板。
4. 用统一命令执行 discovery、人工补录、模板修订和连机验证。
5. 读取结构化产物，判断本轮是否通过、失败还是需要人工处理。
6. 在模板稳定后，把新系统样本接入统一验证汇总。

## 2. 适用范围

本文档适用于以下场景：

1. 新接入一个真实 Web 系统。
2. 已有可用测试账号，且可手工登录。
3. 团队准备先验证 1 个低风险、可重复进入的功能点。
4. 团队希望尽量让平台先做系统探索，减少人工判断成本。

本文档不适用于以下场景：

1. 一开始就做高风险真实提交、审批、支付、外部通知触发。
2. 还没有拿到测试账号或登录后首页。
3. 一开始就接复杂多步骤提交流程，并要求第一轮直接自动通过。

## 3. 团队分工建议

建议至少安排 3 个角色：

1. 接入负责人：负责确定测试对象、命名新模板、组织评审。
2. 执行测试人：负责连机执行命令、保存产物、记录结果。
3. 结果复核人：负责确认系统地图、页面类型初分、验证结论和风险边界。

如果团队人数较少，也可以由 1 人兼任前两项，但结果复核最好由另一个人完成。

## 4. 完成标准

一次“新系统已接入”至少应满足以下条件：

1. 已完成 1 次系统地图探索，并产出 `navigation_tree.json`、`page_semantic_summary.json`、`page_entries.json`、`feature_points.json`。
2. 团队已经基于系统地图选定 1 个首批验证页面和 1 个首批功能点。
3. 新系统已经有 1 个模板目录，包含 `template.json`、`baseline.json`、`data_schema.json`、`locator_hints.json`。
4. 至少完成 1 次连机验证命令执行。
5. 已产出 `validation_result.json`、`verification_result.json`、`network_events.json` 等结构化产物。
6. 团队能够说明本轮结论是“通过”“失败”还是“需要人工处理”。
7. 新系统样本已经进入统一验证汇总配置，或已经形成明确的待接入改动清单。

## 5. 测试前准备

开始前，请逐项确认：

1. 已拉取当前仓库代码，并能进入仓库根目录 `C:\project\aut_agent`。
2. 已准备 Python 虚拟环境，能运行 `.\.venv\Scripts\python.exe --version`。
3. 已准备 Chrome，并能使用远程调试端口连接当前测试会话。
4. 已获取测试账号，并能手工登录目标系统。
5. 已确认本轮不涉及高风险真实提交，或已获得明确许可。

建议先执行下面两个命令确认环境：

```powershell
.\.venv\Scripts\python.exe --version
npm run check
```

如果需要查看运行中心页面，可以启动：

```powershell
npm run dev
```

默认地址：

```text
http://localhost:4173
```

如果端口 `4173` 已被占用，说明本机可能已经有一个运行中心实例在运行。此时不需要重复启动，只要确认现有服务是否可用即可。

如果确认需要查出旧进程并结束它，推荐在 Windows 终端执行：

```powershell
netstat -ano | findstr :4173
```

看到最后一列 PID 后，可继续执行：

```powershell
taskkill /PID <PID> /F
```

如果想先确认 PID 对应的进程名，可以再执行：

```powershell
Get-Process -Id <PID>
```

说明：

1. 某些环境里 `Get-NetTCPConnection -LocalPort 4173` 可能返回 `Access is denied`。
2. 因此优先推荐 `netstat -ano | findstr :4173` 这条兼容性更高的命令。
3. 如果 `http://localhost:4173` 已能正常打开，通常不必重启服务。

## 6. 如何准备浏览器会话

第二阶段连机验证依赖 Chrome 的 CDP 连接。最常见做法是先启动一个带远程调试端口的 Chrome，再用人工方式登录系统。

示例命令如下，Chrome 路径请按本机安装位置调整：

```powershell
& "C:\Program Files\Google\Chrome\Application\chrome.exe" --remote-debugging-port=9222 --user-data-dir="C:\tmp\stage2-chrome"
```

启动后：

1. 在这个 Chrome 窗口中手工登录目标系统。
2. 打开系统首页或你准备探索的入口页。
3. 不要关闭该窗口，后续所有 `--cdp-url http://localhost:9222` 命令都会复用它。

可用下面的命令确认本地 CDP 端口可访问：

```powershell
Invoke-WebRequest http://localhost:9222/json/version | Select-Object -ExpandProperty Content
```

如果这条命令返回一段 JSON，说明 CDP 连接正常。

## 7. 如何选择新系统的首批测试对象

第一次接入新系统时，建议优先选择以下类型的功能点：

1. 查询 / 筛选 / 重置。
2. 详情查看 / 抽屉 / 弹窗查看。
3. 只读导航切换。
4. 简单表单填写但不提交。

不建议第一次就选择：

1. 新增并真实提交。
2. 上传并提交。
3. 支付、审批、发消息、发短信。
4. 依赖复杂前置业务数据的页面。

选择标准如下：

1. 测试账号长期可访问。
2. 页面入口稳定，容易重复进入。
3. 成功判定清晰，例如“按钮出现”“字段清空”“详情抽屉打开”。
4. 不会污染真实业务数据。

## 8. 模板命名与目录规范

每个新系统功能点最终都需要一个模板目录，放在：

```text
prototype/stage2/templates/<template_name>/
```

推荐命名格式：

```text
<system_key>_<scenario_key>
```

示例：

```text
bus_station_query_reset
bus_station_detail_view
license_portal_search
license_portal_detail_view
```

补充说明：

1. 直接在 CLI 里执行 `--explore-system-map` 时，`--template` 请写完整模板名，例如 `bus_system_map`。
2. 如果从运行中心页面的 `system key/template` 输入框发起系统地图探索，只填基础 key，例如 `bus` 或 `suyuan` 即可；页面会自动归一化为 `<system_key>_system_map`。
3. 不要在运行中心表单里手工输入重复后缀，例如 `bus_system_map_system_map`。

模板目录最少包含 4 个文件：

1. `template.json`
2. `baseline.json`
3. `data_schema.json`
4. `locator_hints.json`

现在不建议第一次接入时就从空白状态手工创建这 4 个文件。

推荐顺序是：

1. 先探索系统地图。
2. 再根据系统地图选首批要验证的页面。
3. 再为那个页面生成模板骨架。
4. 最后基于 discovery 和人工录制修订模板。

建议优先参考以下现有模板：

1. 查询 / 重置类：`prototype/stage2/templates/suyuan_online_query_reset/`
2. 详情查看类：`prototype/stage2/templates/suyuan_online_detail_view/`
3. 复杂提交类：`prototype/stage2/templates/suyuan_online_apply/`

第一次接入新系统时，不建议直接从复杂提交模板开始复制。优先从查询或详情模板开始。

## 9. 详细操作步骤

### 先说明推荐流程

新系统第一次接入时，推荐按下面三段执行，不建议一开始就手写完整模板：

1. 第一段：先探索系统地图，让平台生成菜单/页面入口树和页面类型初分。
2. 第二段：再从系统地图中选 1 个低风险页面，生成最小模板骨架并补首轮线索。
3. 第三段：最后人工修订模板并做深入验证。

这样做的原因是：

1. 新手第一次接入时，通常并不知道系统里哪些入口最稳定。
2. 先看系统地图和页面类型初分，能明显减少“该测哪一页、该从哪里起模板”的人工判断。
3. 第二轮再修模板，工作量会明显小于从零手写四个文件。

### 第 1 步：先探索系统地图

第一次接入新系统时，先不要急着为具体功能点创建模板。优先执行系统地图探索：

```powershell
.\.venv\Scripts\python.exe -m prototype.stage2.main --explore-system-map --target-name "<system_name>" --template <template_name> --page-url <start_url> --cdp-url http://localhost:9222
```

示例：

```powershell
.\.venv\Scripts\python.exe -m prototype.stage2.main --explore-system-map --target-name "公交业务系统" --template bus_system_map --page-url https://example.com/bus/home --cdp-url http://localhost:9222
```

参数说明：

1. `--target-name`
   写团队能看懂的系统名称，用于系统地图探索输出。
2. `--template`
   本次系统地图探索使用的模板名。CLI 下建议直接使用 `<system_key>_system_map` 这样的完整模板名；如果从运行中心页面发起，只填基础 key 即可，页面会自动补成完整模板名。
3. `--page-url`
   系统首页或本轮要探索的入口页地址。
4. `--cdp-url`
   当前 Chrome CDP 地址。

这条命令会自动完成两件事：

1. 先为本次探索生成一个最小导航模板骨架。
2. 再立即执行 live discovery，产出系统地图结果。

重复执行说明：

1. 如果同一个系统地图模板已经存在，且模板里记录的入口 URL 与本次 `--page-url` 相同，命令会复用现有模板，不会因为目录已存在而直接失败。
2. 如果模板已存在，但绑定的是另一个入口 URL，请改用新的模板名，或显式加 `--bootstrap-overwrite` 重新生成。

### 第 2 步：检查系统地图产物

系统地图探索完成后，先不要立刻做模板修订，优先检查以下产物：

1. `navigation_tree.json`
2. `page_semantic_summary.json`
3. `navigation_nodes.json`
4. `page_entries.json`
5. `feature_points.json`
6. `discovery_result.json`
7. 页面截图

这几个文件的用途分别是：

1. `navigation_tree.json`
   系统地图树。适合快速查看有哪些一级/二级入口，以及哪些节点已经成功映射到页面入口。
2. `page_semantic_summary.json`
   页面类型初分摘要。当前会给出例如“导航页”“查询列表页”“详情展示页”“新增录入页”“编辑处理页”“提交流程页”“混合工作台页”等初分结果。
3. `navigation_nodes.json`
   原始导航节点清单。适合需要细看节点状态、路径、定位器和触发文本时使用。
4. `page_entries.json`
   去重后的页面入口清单。
5. `feature_points.json`
   当前发现到的功能点候选。

如果你是从运行中心页面点击“01 先探索系统地图”，完成后卡片会直接显示三个核心直链：

1. `navigation_tree.json`
2. `page_semantic_summary.json`
3. `page_entries.json`

其中“步骤结果”链接主要用于查看该步命令返回的调试 JSON，适合排查报错时使用，不是本轮首要审阅产物。

重点先回答三个问题：

1. 系统地图树是否基本符合真实菜单结构。
2. 页面类型初分是否大致合理。
3. 哪个页面最适合作为首批低风险验证目标。

### 第 3 步：确认系统地图里的首批目标

基于 `navigation_tree.json` 和 `page_semantic_summary.json`，建议优先选择：

1. `查询列表页`
2. `详情展示页`
3. `导航页`

不建议第一轮就从以下页面开始：

1. `提交流程页`
2. `新增录入页` 且带真实提交
3. `混合工作台页` 中最复杂的主流程

推荐的选择方法：

1. 先找 `confidence=high` 或 `medium` 的页面。
2. 再看该页面下的 `feature_points.json` 候选是否足够清晰。
3. 再由结果复核人确认“这一页是否值得先沉淀成模板”。

### 第 4 步：查看模型路由摘要

在正式跑目标页面的 discovery 或 validation 前，先看路由摘要，确认当前模板会被怎么处理。

命令：

```powershell
.\.venv\Scripts\python.exe -m prototype.stage2.main --routing-summary --template <template_name>
```

建议关注：

1. 当前模型能力标签。
2. discovery 推荐模式。
3. verification 推荐模式。
4. 是否存在明显阻断原因。

注意：

1. 手册里的 `AI-tester` 只是示例 profile 名称。
2. `--model` 的值必须与本机已加载的模型 profile 一致。
3. 如果不确定，可先省略 `--model`，让系统使用当前默认加载的第一个 profile。

### 第 5 步：为目标页面生成最小模板骨架

当你已经从系统地图里选定一个首批目标页面后，再为这个具体页面生成模板骨架：

```powershell
.\.venv\Scripts\python.exe -m prototype.stage2.main --bootstrap-template --template <template_name> --page-url <page_url> --page-name "<page_name>" --scenario-kind <scenario_kind>
```

常见示例：

查询类页面：

```powershell
.\.venv\Scripts\python.exe -m prototype.stage2.main --bootstrap-template --template bus_station_query_reset --page-url https://example.com/bus/query --page-name "班线查询页" --scenario-kind query
```

详情类页面：

```powershell
.\.venv\Scripts\python.exe -m prototype.stage2.main --bootstrap-template --template license_portal_detail_view --page-url https://example.com/license/list --page-name "许可列表页" --scenario-kind detail
```

这一步的目标不是生成最终模板，而是：

1. 固定模板目录结构。
2. 让后续 `--live-discovery`、`--capture-human-recording`、`--template-revision-checklist` 有稳定输入。
3. 把系统地图里选中的页面正式收敛成一个具体模板目标。

### 第 6 步：执行目标页面的第一轮自主探索

模板骨架创建完成后，建议先让系统对这个目标页面跑一轮 discovery：

```powershell
.\.venv\Scripts\python.exe -m prototype.stage2.main --live-discovery --template <template_name> --cdp-url http://localhost:9222
```

如果你明确知道本机要用的模型 profile，也可以显式指定：

```powershell
.\.venv\Scripts\python.exe -m prototype.stage2.main --live-discovery --template <template_name> --model <your_model_profile> --cdp-url http://localhost:9222
```

这一轮的目标不是直接得到最终模板，而是补下列线索：

1. 当前页面入口名称是否合理。
2. 页面上有哪些候选功能点。
3. 哪些按钮文字、标题文字、字段名更稳定。
4. 当前页面是否还有关联页面入口、弹层动作、行级动作。

建议重点查看：

1. `page_entries.json`
2. `feature_points.json`
3. `discovery_review_queue.json`
4. `page_semantic_summary.json`
5. 页面截图

如果通过运行中心页面执行这一步，卡片会直接提供 `page_entries.json`、`feature_points.json`、`discovery_review_queue.json` 的直链。

### 第 7 步：需要时执行人工录制，补首轮线索

如果 discovery 还不足以说明页面行为，或者页面是表格 + 弹窗 + 复杂交互组合，建议补一轮人工录制：

```powershell
.\.venv\Scripts\python.exe -m prototype.stage2.main --capture-human-recording --template <template_name> --cdp-url http://localhost:9222
```

这一轮适合解决的问题：

1. 页面里哪些按钮是真正有用的业务按钮。
2. 某个查询、重置、详情动作是怎么触发的。
3. 某些定位器单靠 discovery 不够稳定时，需要人工演示路径补充上下文。

建议重点看以下产物：

1. `recording_summary.json`
2. `candidate_template_draft.json`
3. `candidate_template_review.json`
4. `key_screenshots.json`

### 第 8 步：生成模板修订清单

当目标页面的第一轮 discovery 或人工录制完成后，先执行：

```powershell
.\.venv\Scripts\python.exe -m prototype.stage2.main --template-revision-checklist --template <template_name>
```

如果你希望显式指定 discovery 目录或 candidate review 文件，也可以这样写：

```powershell
.\.venv\Scripts\python.exe -m prototype.stage2.main --template-revision-checklist --template <template_name> --discovery-dir <live_discovery_dir> --candidate-review <candidate_template_review.json>
```

这条命令会生成两份结果：

1. `template_revision_checklist.json`
2. `template_revision_checklist.md`

默认输出目录在：

```text
prototype/stage2/templates/<template_name>/_revision_checklist/
```

这份清单当前还会额外提醒：

1. discovery 对该页面的页面类型初分是什么。
2. 当前模板是否真的收敛在正确的页面语义上。
3. 本模板是否锁定了一个明确功能点，而不是把多个候选混在一起。

### 第 9 步：按修订清单人工修改模板文件

建议按下面顺序修订：

1. `template.json`
2. `locator_hints.json`
3. `baseline.json`
4. `data_schema.json`

优先关注：

1. `page_entry.name`
2. `page_entry.url`
3. `feature_point.name`
4. `feature_point.type`
5. `steps`
6. `success_rules`
7. discovery 给出的页面类型初分是否与你的模板目标一致

建议把第二轮的第一版 `steps` 控制在 3 到 8 步以内。

### 第 10 步：执行单模板连机验证

这是新系统接入时最重要的一步。

命令：

```powershell
.\.venv\Scripts\python.exe -m prototype.stage2.main --validate-connected-template <template_name> --cdp-url http://localhost:9222
```

说明：

1. 这个命令会复用当前已登录的 Chrome 会话。
2. 它适合验证新系统的单模板、单功能点最小闭环。
3. 它比 `--run-sample` 更适合新系统首测。

### 第 11 步：检查验证产物

单模板连机验证完成后，请检查输出目录下的结构化结果。

当前重点关注：

1. `validation_result.json`
2. `verification_result.json`
3. `network_events.json`
4. 截图文件

对 `--validate-connected-template` 而言，当前结果主要看：

1. `status=passed`
   说明该模板已具备初步回归能力。
2. `status=failed`
   说明模板、定位器、页面状态或前置条件至少有一项未满足。

`status=skipped` 更常见于统一验证汇总场景，不应作为单模板 connected validation 的常见预期结果。

### 第 12 步：失败后修订模板并重跑

如果本轮失败，不要直接改成大脚本，优先按顺序排查：

1. 当前页面 URL 是否正确。
2. 目标按钮或字段文本是否已变化。
3. `locator_hints.json` 是否能提供更稳定的定位线索。
4. 当前登录态是否已失效。
5. 当前功能点是否依赖前置数据。
6. 系统地图里对该页面的页面类型初分是否本来就选错了。

建议每次只改一类问题，再重新执行：

```powershell
.\.venv\Scripts\python.exe -m prototype.stage2.main --validate-connected-template <template_name> --cdp-url http://localhost:9222
```

### 第 13 步：需要人工补齐时的处理方式

当系统必须人工补齐登录、切换页面、补前置数据或确认某个页面状态时，可以使用以下辅助方式。

如果要录制人工演示过程：

```powershell
.\.venv\Scripts\python.exe -m prototype.stage2.main --capture-human-recording --template <template_name> --cdp-url http://localhost:9222
```

如果已有完整 run 并进入人工处理阶段，恢复续跑命令为：

```powershell
.\.venv\Scripts\python.exe -m prototype.stage2.main --resume-human-takeover <run_dir> --cdp-url http://localhost:9222
```

需要注意：

1. “人工已处理”不等于“系统自动通过”。
2. 人工处理记录只表示人已经补齐条件或允许继续。
3. 原始阻塞原因和人工处理记录会分开保存。

### 第 14 步：把新系统样本接入统一验证汇总

当单模板连机验证已经基本稳定后，需要把它接入统一验证汇总。

当前最关键的代码点通常是：

1. `prototype/stage2/app/verification/validation_matrix.py`
   把新模板加入默认验证目标列表。
2. `prototype/stage2/tests/test_g4_validation_matrix.py`
   为新模板补验证矩阵回归测试。

如果你还希望单模板 validation payload 里的系统元信息不再落到默认 `generic_template / Generic Template`，再额外检查：

1. `prototype/stage2/main.py`
   当前这里仍主要按模板名前缀推断元信息，而不是维护一张完整的新系统映射表。

当前 `ValidationMatrixTarget` 重点信息包括：

1. `target_id`
2. `template_name`
3. `family`
4. `system_name`
5. `mode`
6. `description`

注意：

1. 当前矩阵 target 里没有独立的 `system_id` 字段。
2. 当前默认矩阵仍以 `lab_*` 本地模板族和 `suyuan_*` 单真实系统样本为主。
3. 它代表“统一汇总骨架已成立”，不代表仓库已经默认覆盖多个真实业务系统。

完成代码接线后，再运行统一验证汇总命令：

```powershell
.\.venv\Scripts\python.exe -m prototype.stage2.main --validation-matrix --cdp-url http://localhost:9222
```

### 第 15 步：查看统一验证汇总结果

统一验证汇总执行完成后，请检查：

1. `artifacts/stage2/validation_matrix/latest_validation_matrix.json`
2. `artifacts/stage2/validation_matrix/latest_validation_matrix.md`

重点确认：

1. 新模板是否出现在结果列表中。
2. `system_name` 是否正确。
3. `mode` 是否正确。
4. 结果是否为 `passed`、`failed` 或 `skipped`。
5. `reason` 是否清楚表达失败原因。

## 10. 推荐的最小接入顺序

为了提高成功率，建议团队按下面顺序接入新系统：

1. 先跑系统地图探索。
2. 再从系统地图里选一个 `查询列表页` 或 `详情展示页`。
3. 再为这个页面起模板。
4. 再跑目标页面 discovery 或人工录制。
5. 最后做 connected validation。

如果第一轮就从复杂提交页开始，往往会同时遇到定位器、前置数据、业务规则和风险边界四类问题，排查成本会明显更高。

## 11. 常见问题

### 11.1 系统地图探索能跑，但没有形成像样的树

优先检查：

1. 当前登录页是否真的是系统首页或核心入口页。
2. 该系统菜单是否主要依赖前端框架动态切换，而不是普通 href 跳转。
3. `navigation_tree.json` 里是否已有静态可见节点，只是还没有全部映射成页面入口。
4. 当前 discovery 深度是否只够看首层入口。

### 11.2 连机验证一开始就报错

优先检查：

1. `http://localhost:9222` 是否可访问。
2. 模板目录是否完整。
3. `template_name` 是否和目录名一致。
4. 当前页面是否需要额外前置跳转。

### 11.3 页面打开了，但断言失败

优先检查：

1. 按钮文字是否与模板一致。
2. 定位器是否过于脆弱。
3. 断言内容是否选得太业务化。
4. 页面是否依赖某条测试数据记录。
5. 本模板是否收敛到了错误的页面类型。

### 11.4 想把新系统加入统一验证汇总，但命令里看不到它

请检查：

1. `prototype/stage2/app/verification/validation_matrix.py`
2. `prototype/stage2/tests/test_g4_validation_matrix.py`
3. 如需更准确的系统元信息，再检查 `prototype/stage2/main.py` 的模板元信息推断逻辑

### 11.5 不知道该用哪个命令

建议按用途区分：

1. 先探索系统地图：`--explore-system-map`
2. 看路由信息：`--routing-summary`
3. 看目标页面入口和功能点：`--live-discovery`
4. 录人工演示：`--capture-human-recording`
5. 生成模板修订清单：`--template-revision-checklist`
6. 测单个真实模板：`--validate-connected-template`
7. 跑本地 headless 的 generic-template 验证入口：`--validate-template`
8. 跑统一验证汇总：`--validation-matrix`
9. 恢复人工处理后的 run：`--resume-human-takeover`

## 12. 团队提交物清单

每完成一个新系统样本，建议至少提交以下内容：

1. 系统地图探索命令记录。
2. `navigation_tree.json` 与 `page_semantic_summary.json` 的关键结论。
3. 模板目录 4 个文件的改动。
4. 连机验证命令记录。
5. 关键产物路径。
6. 本轮结论。
7. 风险说明。
8. 下一步建议。

建议团队在评审时至少回答以下问题：

1. 系统地图树是否足够让别人看懂这个系统的大致入口结构。
2. 这个模板是否可以重复执行两次以上。
3. 这个模板是否会污染真实业务数据。
4. 失败时能否从结构化产物中看出原因。
5. 是否已经具备进入统一验证汇总的条件。
6. 是否还需要人工录制或人工补齐。
