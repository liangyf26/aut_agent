# 第二阶段新系统接入测试手册

更新时间：2026-06-23

## 1. 文档目标

本文档用于指导团队成员把一个新的真实 Web 系统接入到第二阶段原型中，并完成第一轮可复核的自动化验证。

本文档的目标不是讲架构设计，而是让不熟悉前期过程的同事也能按步骤完成以下工作：

1. 为新系统选择合适的首批测试对象。
2. 创建新系统的模板目录和最小配置。
3. 用统一命令执行发现、验证和结果汇总。
4. 读取结构化产物，判断本轮是否通过。
5. 在需要时记录人工处理和恢复续跑。
6. 把新系统样本接入统一验证汇总结果。

## 2. 适用范围

本文档适用于以下场景：

1. 新接入一个真实 Web 系统。
2. 该系统已经有可用测试账号。
3. 团队计划先验证 1 个低风险、可重复进入的功能点。
4. 团队需要把结果沉淀为可共享、可回归的模板样本。

本文档不适用于以下场景：

1. 直接做高风险真实提交、审批、支付、外部通知触发。
2. 尚未拿到测试账号或登录后首页。
3. 一开始就接复杂多步骤提交流程。

## 3. 团队分工建议

建议至少安排 3 个角色：

1. 接入负责人：负责选测试对象、建模板、修改配置。
2. 执行测试人：负责连机执行命令、保存产物、记录结果。
3. 结果复核人：负责确认测试结论、风险边界和后续动作。

如果团队人数较少，也可以由 1 人兼任前两项，但结果复核最好由另一个人完成。

## 4. 完成标准

一次“新系统已接入”至少应满足以下条件：

1. 新系统已经有 1 个模板目录，包含 `template.json`、`baseline.json`、`data_schema.json`、`locator_hints.json`。
2. 至少完成 1 次连机验证命令执行。
3. 已产出 `validation_result.json`、`verification_result.json`、`network_events.json` 等结构化产物。
4. 团队能够说明本轮结论是“通过”“失败”还是“需要人工处理”。
5. 新系统样本已经进入统一验证汇总配置，或已经形成明确的待接入改动清单。

## 5. 测试前准备

开始前，请逐项确认：

1. 已拉取当前仓库代码，并能进入仓库根目录 `C:\project\aut_agent`。
2. 已准备 Python 虚拟环境，能运行 `.\.venv\Scripts\python.exe --version`。
3. 已准备 Chrome，并能使用远程调试端口连接当前测试会话。
4. 已获取测试账号，并能手工登录目标系统。
5. 已确认本轮不涉及高风险真实提交，或已获得明确许可。

建议先执行下面两个命令确认环境：

```powershell
.\.venv\Scripts\python.exe --version
npm run check
```

如果需要查看运行中心页面，可以启动：

```powershell
npm run dev
```

默认地址：

```text
http://localhost:4173
```

## 6. 如何准备浏览器会话

第二阶段连机验证依赖 Chrome 的 CDP 连接。最常见做法是先启动一个带远程调试端口的 Chrome，再用人工方式登录系统。

示例命令如下，Chrome 路径请按本机安装位置调整：

```powershell
& "C:\Program Files\Google\Chrome\Application\chrome.exe" --remote-debugging-port=9222 --user-data-dir="C:\tmp\stage2-chrome"
```

启动后：

1. 在这个 Chrome 窗口中手工登录目标系统。
2. 打开要测试的页面或系统首页。
3. 不要关闭该窗口，后续所有 `--cdp-url http://localhost:9222` 命令都会复用它。

可用下面的命令确认本地 CDP 端口可访问：

```powershell
Invoke-WebRequest http://localhost:9222/json/version | Select-Object -ExpandProperty Content
```

如果这条命令返回一段 JSON，说明 CDP 连接正常。

## 7. 如何选择新系统的首批测试对象

第一次接入新系统时，建议优先选择以下类型的功能点：

1. 查询 / 筛选 / 重置。
2. 详情查看 / 抽屉 / 弹窗查看。
3. 只读导航切换。
4. 简单表单填写但不提交。

不建议第一次就选择：

1. 新增并真实提交。
2. 上传并提交。
3. 支付、审批、发消息、发短信。
4. 依赖复杂前置业务数据的页面。

选择标准如下：

1. 测试账号长期可访问。
2. 页面入口稳定，容易重复进入。
3. 成功判定清晰，例如“按钮出现”“字段清空”“详情抽屉打开”。
4. 不会污染真实业务数据。

## 8. 模板命名与目录规范

每个新系统功能点都需要一个模板目录，放在：

```text
prototype/stage2/templates/<template_name>/
```

推荐命名格式：

```text
<system_key>_<scenario_key>
```

示例：

```text
bus_station_query_reset
bus_station_detail_view
license_portal_search
license_portal_detail_view
```

模板目录最少包含 4 个文件：

1. `template.json`
2. `baseline.json`
3. `data_schema.json`
4. `locator_hints.json`

现在不建议新手第一次就手工创建这 4 个文件。

推荐做法是：

1. 先用 `--bootstrap-template` 生成最小骨架。
2. 用系统先跑一轮 discovery 或人工录制，拿到初步线索。
3. 再由测试人员人工修改这 4 个文件，进入第二轮深入验证。

建议优先参考以下现有模板：

1. 查询 / 重置类：`prototype/stage2/templates/suyuan_online_query_reset/`
2. 详情查看类：`prototype/stage2/templates/suyuan_online_detail_view/`
3. 复杂提交类：`prototype/stage2/templates/suyuan_online_apply/`

第一次接入新系统时，不建议直接从复杂提交模板开始复制。优先从查询或详情模板开始。

## 9. 详细操作步骤

### 先说明推荐流程

新系统第一次接入时，推荐按下面两轮执行，不建议一开始就手写完整模板：

1. 第一轮：先生成最小模板骨架，让系统自主探索并产出初步线索。
2. 第二轮：由测试人员基于第一轮产物，人工修订模板，再做深入验证。

这样做的原因是：

1. 新手一开始通常还不知道页面上哪些文案、按钮、字段名最稳定。
2. 先跑 discovery 或人工录制，系统可以先帮你把页面入口、功能点和候选定位线索落出来。
3. 测试人员第二轮再修改模板，工作量会明显小于从零手写四个文件。

---

### 第 1 步：先生成最小模板骨架

第一次接入新系统时，先不要手工创建 4 个 JSON 文件，优先执行：

```powershell
.\.venv\Scripts\python.exe -m prototype.stage2.main --bootstrap-template --template <template_name> --page-url <page_url> --page-name "<page_name>" --scenario-kind <scenario_kind>
```

常见示例：

查询类页面：

```powershell
.\.venv\Scripts\python.exe -m prototype.stage2.main --bootstrap-template --template bus_station_query_reset --page-url https://example.com/bus/query --page-name "班线查询页" --scenario-kind query
```

详情类页面：

```powershell
.\.venv\Scripts\python.exe -m prototype.stage2.main --bootstrap-template --template license_portal_detail_view --page-url https://example.com/license/list --page-name "许可列表页" --scenario-kind detail
```

参数说明：

1. `--template`
   模板目录名，建议使用 `<system_key>_<scenario_key>`。
2. `--page-url`
   当前准备接入的页面地址。
3. `--page-name`
   页面入口名称，写人能看懂的名字即可。
4. `--scenario-kind`
   用于告诉系统这是哪类初始草稿。推荐值：
   - `query`
   - `detail`
   - `navigation`
   - `create`
   - `edit`
   - `generic`

执行后，系统会自动生成：

1. `template.json`
2. `baseline.json`
3. `data_schema.json`
4. `locator_hints.json`

这些文件一开始是“最小草稿”，用途是：

1. 让 `--routing-summary`、`--live-discovery`、`--capture-human-recording` 能立刻使用。
2. 让测试人员不必第一次接触就手写四个模板文件。
3. 先把模板目录结构和基本字段固定下来。

### 第 2 步：查看模型路由摘要

在正式跑 discovery 或 validation 前，先看路由摘要，确认当前模板会被怎么处理。

命令：

```powershell
.\.venv\Scripts\python.exe -m prototype.stage2.main --routing-summary --template <template_name>
```

建议关注：

1. 当前模型能力标签。
2. discovery 推荐模式。
3. verification 推荐模式。
4. 是否存在明显阻断原因。

如果这里已经显示明显阻断，先不要继续连机验证，先排查模型配置或模板基础信息。

### 第 3 步：执行第一轮自主探索

模板骨架创建完成后，建议先让系统自己跑一轮 discovery。

命令：

```powershell
.\.venv\Scripts\python.exe -m prototype.stage2.main --live-discovery --template <template_name> --model AI-tester --cdp-url http://localhost:9222
```

这一轮的目标不是直接得到最终模板，而是先拿到以下线索：

1. 当前页面入口名称是否合理。
2. 页面上有哪些候选功能点。
3. 哪些按钮文字、标题文字、字段名更稳定。
4. 当前页面是否还有关联页面入口、弹层动作、行级动作。

建议重点查看这些产物：

1. `page_entries.json`
2. `feature_points.json`
3. `discovery_review_queue.json`
4. 页面截图
5. `routing_summary.json`
6. `discovery_strategy.json`

如果这一步能正常跑完，说明新系统至少已经具备：

1. 已登录页面可被复用。
2. 模板目录结构正确。
3. discovery 主流程可以工作。

### 第 4 步：需要时执行人工录制，补首轮线索

如果 discovery 还不足以说明页面行为，或者页面是表格 + 弹窗 + 复杂交互组合，建议补一轮人工录制：

```powershell
.\.venv\Scripts\python.exe -m prototype.stage2.main --capture-human-recording --template <template_name> --cdp-url http://localhost:9222
```

这一轮适合解决的问题：

1. 页面里哪些按钮是真正有用的业务按钮。
2. 某个查询、重置、详情动作是怎么触发的。
3. 某些定位器单靠 discovery 不够稳定时，需要人工演示路径补充上下文。

建议重点看以下产物：

1. `recording_summary.json`
2. `candidate_template_draft.json`
3. `candidate_template_review.json`
4. `key_screenshots.json`

其中最重要的是：

1. `candidate_template_draft.json`
   系统根据人工操作生成的候选步骤草稿。
2. `candidate_template_review.json`
   系统整理出的候选字段映射、alias 草案和定位线索，适合第二轮人工修模板时参考。

### 第 5 步：先生成“半自动修订清单”

当第一轮 discovery 或人工录制完成后，不建议直接人工打开 4 个 JSON 文件逐个比对。

先执行：

```powershell
.\.venv\Scripts\python.exe -m prototype.stage2.main --template-revision-checklist --template <template_name>
```

如果你希望显式指定 discovery 目录或 candidate review 文件，也可以这样写：

```powershell
.\.venv\Scripts\python.exe -m prototype.stage2.main --template-revision-checklist --template <template_name> --discovery-dir <live_discovery_dir> --candidate-review <candidate_template_review.json>
```

这条命令会生成两份结果：

1. `template_revision_checklist.json`
2. `template_revision_checklist.md`

默认输出目录在：

```text
prototype/stage2/templates/<template_name>/_revision_checklist/
```

这份清单会自动把第一轮产物整理成 4 组建议：

1. `template.json` 建议改什么
2. `locator_hints.json` 建议补什么
3. `baseline.json` 建议保留哪些页面上下文
4. `data_schema.json` 建议回填哪些字段规则

建议优先看 `template_revision_checklist.md`，因为它已经把重点按“文件动作 + 建议片段 + 待确认项”整理好了。

### 第 6 步：按修订清单人工修改模板文件

这一步才是第二轮真正开始人工修模板。现在回头修改，难度会比从零开始低很多。

建议按下面顺序修订：

1. `template.json`
2. `locator_hints.json`
3. `baseline.json`
4. `data_schema.json`

#### 先改 `template.json`

`template.json` 是第二轮最核心的文件。优先确认这些字段：

1. `template_name`
2. `page_entry.name`
3. `page_entry.url`
4. `feature_point.name`
5. `feature_point.type`
6. `steps`
7. `success_rules`
8. `notes`

建议把第二轮的第一版 `steps` 控制在 3 到 8 步以内。

推荐写法：

1. 先保留“打开页面”和“截图留证”。
2. 再加 1 到 3 个关键交互步骤。
3. 最后补 1 到 2 个成功判定。

查询 / 重置类常见思路：

1. 填一个查询条件。
2. 点击搜索。
3. 点击重置。
4. 断言字段已清空，或结果区域恢复默认状态。

详情查看类常见思路：

1. 点击详情按钮。
2. 等待详情抽屉或弹窗出现。
3. 断言某个标题、按钮或字段可见。

#### 再改 `locator_hints.json`

第一轮后，这个文件通常比一开始更容易写，因为你已经有了 discovery 和录制产物。

优先补：

1. 关键按钮定位线索。
2. 关键输入框定位线索。
3. 详情按钮、抽屉标题、弹窗标题等稳定标识。

#### 再改 `baseline.json`

如果当前模板需要默认页面上下文、默认筛选值或少量参考值，再补到这里。

第一批低风险模板通常只需要：

1. 页面 URL。
2. 默认查询条件。
3. 某些成功后应出现的稳定上下文说明。

#### 最后改 `data_schema.json`

如果模板暂时只做查询、详情、导航类验证，可以先保持简单。

只有在你确认确实需要输入字段、生成规则或唯一值策略时，再补 `field_rules`。

如果 `template_revision_checklist.json` 里已经给出了可用的 `steps` 建议和 `field_rules` 建议，优先复制这些建议，再做少量人工修正，不要从空白状态重新写。

### 第 7 步：执行单模板连机验证

这是新系统接入时最重要的一步。

命令：

```powershell
.\.venv\Scripts\python.exe -m prototype.stage2.main --validate-connected-template <template_name> --cdp-url http://localhost:9222
```

说明：

1. 这个命令会复用当前已登录的 Chrome 会话。
2. 它适合验证新系统的单模板、单功能点最小闭环。
3. 它比 `--run-sample` 更适合新系统首测。

注意：

1. 当前 `--run-sample` 是仓库内现有样本的完整流程入口，不适合直接拿来做新系统第一次接入测试。
2. 新系统首测应优先使用 `--validate-connected-template`。

### 第 8 步：检查验证产物

单模板连机验证完成后，请检查输出目录下的结构化结果。

当前重点关注：

1. `validation_result.json`
2. `verification_result.json`
3. `network_events.json`
4. 截图文件

应重点查看的字段：

1. `status`
2. `success`
3. `rule_summary`
4. `failed_step_ids`
5. `feature_point_name`
6. `page_entry_url`

建议按下面的规则判断本轮结果：

1. `status=passed`
   说明该模板已具备初步回归能力。
2. `status=failed`
   说明模板、定位器、页面状态或前置条件至少有一项未满足。
3. `status=skipped`
   说明当前模式未执行，例如 CDP 未提供。

### 第 9 步：失败后修订模板并重跑

如果本轮失败，不要直接改成大脚本，优先按顺序排查：

1. 当前页面 URL 是否正确。
2. 目标按钮或字段文本是否已变化。
3. `locator_hints.json` 是否能提供更稳定的定位线索。
4. 当前登录态是否已失效。
5. 当前功能点是否依赖前置数据。

建议每次只改一类问题，再重新执行：

```powershell
.\.venv\Scripts\python.exe -m prototype.stage2.main --validate-connected-template <template_name> --cdp-url http://localhost:9222
```

如果已经连续两轮失败，建议把失败原因写进模板 `notes` 或单独记录到团队测试记录中，避免重复试错。

### 第 10 步：需要人工补齐时的处理方式

当系统必须人工补齐登录、切换页面、补前置数据或确认某个页面状态时，可以使用以下辅助方式。

如果要录制人工演示过程：

```powershell
.\.venv\Scripts\python.exe -m prototype.stage2.main --capture-human-recording --template <template_name> --cdp-url http://localhost:9222
```

如果已有完整 run 并进入人工处理阶段，恢复续跑命令为：

```powershell
.\.venv\Scripts\python.exe -m prototype.stage2.main --resume-human-takeover <run_dir> --cdp-url http://localhost:9222
```

需要注意：

1. “人工已处理”不等于“系统自动通过”。
2. 人工处理记录只表示人已经补齐条件或允许继续。
3. 原始阻塞原因和人工处理记录会分开保存。

### 第 11 步：把新系统样本接入统一验证汇总

当单模板连机验证已经基本稳定后，需要把它接入统一验证汇总。

当前需要修改的代码点通常包括：

1. `prototype/stage2/main.py`
   更新模板元信息映射，让新模板不再落到默认的 `generic_template`。
2. `prototype/stage2/app/verification/validation_matrix.py`
   把新模板加入默认验证目标列表。
3. `prototype/stage2/tests/test_g4_validation_matrix.py`
   为新模板补验证矩阵回归测试。

团队需要明确填写的新信息包括：

1. `system_id`
2. `system_name`
3. `family`
4. `mode`
5. `description`

完成代码接线后，再运行统一验证汇总命令：

```powershell
.\.venv\Scripts\python.exe -m prototype.stage2.main --validation-matrix --cdp-url http://localhost:9222
```

需要注意：

1. 这个命令只会执行当前“默认目标列表”中的模板。
2. 如果新系统模板没有被加入默认目标列表，执行该命令时不会自动测到你的新模板。

### 第 12 步：查看统一验证汇总结果

统一验证汇总执行完成后，请检查：

1. `artifacts/stage2/validation_matrix/latest_validation_matrix.json`
2. `artifacts/stage2/validation_matrix/latest_validation_matrix.md`

重点确认：

1. 新模板是否出现在结果列表中。
2. `system_name` 是否正确。
3. `mode` 是否正确。
4. 结果是否为 `passed`、`failed` 或 `skipped`。
5. `reason` 是否清楚表达失败原因。

## 10. 推荐的最小接入顺序

为了提高成功率，建议团队按下面顺序接入新系统：

1. 查询 / 重置类模板
2. 详情查看类模板
3. 导航类模板
4. 简单表单填写但不提交
5. 提交、上传、审批类模板

如果第一轮就从提交类模板开始，往往会同时遇到定位器、前置数据、业务规则和风险边界四类问题，排查成本会明显更高。

## 11. 常见问题

### 11.1 命令能跑，但没有发现页面

优先检查：

1. Chrome 是否用远程调试端口启动。
2. 当前登录态是否还在。
3. 模板中的 `page_entry.url` 是否正确。
4. 当前活动页是否就是目标系统页面。

### 11.2 连机验证一开始就报错

优先检查：

1. `http://localhost:9222` 是否可访问。
2. 模板目录是否完整。
3. `template_name` 是否和目录名一致。
4. 当前页面是否需要额外前置跳转。

### 11.3 页面打开了，但断言失败

优先检查：

1. 按钮文字是否与模板一致。
2. 定位器是否过于脆弱。
3. 断言内容是否选得太业务化。
4. 页面是否依赖某条测试数据记录。

### 11.4 想把新系统加入统一验证汇总，但命令里看不到它

说明新模板还没有接入默认验证目标列表。

请检查：

1. `prototype/stage2/app/verification/validation_matrix.py`
2. `prototype/stage2/main.py` 中的系统元信息映射
3. 对应的测试是否已经补齐

### 11.5 不知道该用哪个命令

建议按用途区分：

1. 看路由信息：`--routing-summary`
2. 看页面入口和功能点：`--live-discovery`
3. 生成半自动修订清单：`--template-revision-checklist`
4. 测单个真实模板：`--validate-connected-template`
5. 跑本地模板：`--validate-template`
6. 跑统一验证汇总：`--validation-matrix`
7. 录人工演示：`--capture-human-recording`
8. 恢复人工处理后的 run：`--resume-human-takeover`

## 12. 团队提交物清单

每完成一个新系统样本，建议至少提交以下内容：

1. 模板目录 4 个文件的改动。
2. 本轮执行命令记录。
3. 关键产物路径。
4. 本轮结论。
5. 风险说明。
6. 下一步建议。

建议团队在评审时至少回答以下问题：

1. 这个模板是否可以重复执行两次以上。
2. 这个模板是否会污染真实业务数据。
3. 失败时能否从结构化产物中看出原因。
4. 是否已经具备进入统一验证汇总的条件。
5. 是否还需要人工录制或人工补齐。
