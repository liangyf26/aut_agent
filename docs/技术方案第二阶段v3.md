# AI Agent 软件自动化评测平台技术方案（第二阶段 v3）

更新时间：2026-06-24

## 1. v3 技术方案目标

本文档是《技术方案第二阶段.md》的纠偏版，不覆盖旧文档。

v3 技术方案的目标是把第二阶段从“模板接入与样本验证工程”拉回到“已登录 Web UI 系统的自动发现、自动用例生成、低风险自动验证、证据报告输出”的主线。

v3 的核心技术命题是：

**模板可以继续存在，但不能再是用户主入口。系统必须以 run 为中心，从目标系统页面自动生成页面入口、功能点、执行型测试用例、执行结果和报告。**

## 2. 当前实现技术审计

### 2.1 已有可复用能力

当前代码中已有以下可复用能力：

- `prototype.stage2.main` 统一 CLI 入口
- `--explore-system-map`
- `--live-discovery`
- `--capture-human-recording`
- `--template-revision-checklist`
- `--validate-connected-template`
- `--validation-matrix`
- `DiscoveryArtifactWriter`
- `plan_live_discovery`
- `TemplateFlowExecutor`
- `TemplateActionRegistry`
- `ProgressManager`
- `run_report` / `platform_daily_report` 报告模型
- Node.js 运行中心 API 与 operation session

这些能力说明底层工程并非完全缺失。

### 2.2 当前主要技术偏差

当前偏差在于编排中心和数据模型：

1. **主实体错位**  
   当前用户主入口是模板和 operation step；v3 应以 `run` 为主实体。

2. **发现结果没有驱动后续执行**  
   `page_entries.json`、`feature_points.json` 可以生成，但缺少从功能点自动生成执行型测试用例并执行的稳定链路。

3. **模板暴露过早**  
   `template.json`、`baseline.json`、`data_schema.json`、`locator_hints.json` 被用户当作必填配置。这些应转为内部中间资产。

4. **运行中心是命令面板，不是评测工作流**  
   运行中心目前能触发多个命令，但没有把“发现-审核-生成-执行-报告”表达成一个连续 run。

5. **报告不是完整 run 的自然产物**  
   报告能力存在，但没有由新系统自动评测主流程稳定产出。

## 3. v3 总体架构

v3 采用以下结构：

```text
运行中心 UI
  ├─ 新建运行
  ├─ 运行状态
  ├─ 页面入口树
  ├─ 功能点审核
  ├─ 执行型测试用例
  ├─ 执行证据
  └─ 总体报告

Node.js 平台 API
  ├─ run 创建与查询
  ├─ run 阶段推进
  ├─ artifact 安全访问
  └─ 人工审核结果保存

Python Stage-2 Orchestrator
  ├─ preflight
  ├─ login/session check
  ├─ discovery
  ├─ feature analysis
  ├─ review gate
  ├─ test case generation
  ├─ low-risk execution
  ├─ evidence collection
  └─ report generation

执行器层
  ├─ Playwright CDP connector
  ├─ DOM scanner
  ├─ Navigation explorer
  ├─ Light interaction explorer
  ├─ Action executor
  ├─ Screenshot collector
  └─ Network/event collector

AI/规则层
  ├─ 页面语义分类
  ├─ 功能点归类
  ├─ 测试用例生成
  ├─ 失败归因
  └─ 报告摘要

产物层
  ├─ run_manifest.json
  ├─ page_entries.json
  ├─ feature_points.json
  ├─ generated_test_cases.json
  ├─ execution_results.json
  ├─ screenshots_index.json
  └─ reports/run_report.md
```

### 3.1 运行中心是总控平面

v3 技术实现必须把运行中心作为总控平面，而不是把运行中心做成脚本按钮集合。

运行中心是第二阶段的查询、操作、监控、人工介入和报告查看的集中平台。

运行中心负责：

- run 创建、配置、启动、暂停、恢复、终止。
- run 列表、历史轮次和 artifact 查询。
- 实时进度监控和事件流展示。
- 页面入口树、功能点、测试用例、执行结果和报告展示。
- 人工审核、人工录制、数据更正、策略授权。
- AI 复盘、下一轮计划和沉淀候选审阅。

Python CLI 仍可作为后台执行器入口，但普通用户不应通过 CLI 推进常规流程。

### 3.2 控制流与数据流

v3 的控制流由运行中心发起：

```text
运行中心
  -> Node.js Run API
  -> Python Stage-2 Orchestrator
  -> Playwright / AI / Artifact Writer
  -> 稳定 run artifacts
  -> 运行中心轮询或订阅状态
```

v3 的数据流必须以 run artifact contract 为准：

```text
input_config
  -> preflight_result
  -> page_entries + feature_points
  -> discovery_review
  -> generated_test_cases
  -> execution_results
  -> round_analysis + next_round_plan
  -> run_report
```

任何阶段不得只把结果打印到终端。

### 3.3 后台脚本的定位

现有 CLI 能力按三类处理：

1. **后台执行入口**  
   由 Node.js API 调用，用户不直接接触。例如 discovery、execution、reporting。

2. **高级调试入口**  
   对开发者可见。例如单独运行 `--live-discovery`、`--validate-connected-template`。

3. **内部迁移资产**  
   可保留但不进入主界面。例如 `--bootstrap-template`、`--template-revision-checklist`。

运行中心主流程不得要求用户复制命令。

## 4. v3 核心运行模型

### 4.1 Run 是唯一主实体

每次用户点击“开始自动评测”后，系统创建一个 run：

```json
{
  "run_id": "stage2_20260624_101500_xxxxxxxx",
  "system_name": "追本溯源管理系统",
  "entry_url": "https://example.com/home",
  "cdp_url": "http://localhost:9222",
  "status": "discovering",
  "created_at": "2026-06-24T10:15:00+08:00"
}
```

所有 discovery、review、test case、execution、report 都归属这个 run。

### 4.2 Template 是内部派生资产

v3 中模板分为两类：

1. **临时执行计划**  
   从功能点自动生成，用于本 run 执行，不要求用户编辑。

2. **可沉淀模板**  
   当某个功能点多次验证稳定后，系统可建议沉淀为项目级模板。

用户默认不需要打开模板文件。

### 4.3 Artifact 是稳定接口

运行中心不应读取零散临时文件，而应读取稳定 artifact contract：

```text
artifacts/stage2/runs/<run_id>/
  run_manifest.json
  input_config.json
  progress_events.jsonl
  current_status.json
  system_map.json
  navigation_tree.json
  page_entries.json
  feature_points.json
  discovery_review.json
  generated_test_cases.json
  execution_results.json
  screenshots_index.json
  failure_summary.json
  reports/run_report.json
  reports/run_report.md
```

### 4.4 Round 是自进化循环单元

一个 run 可以包含多个 round。

每个 round 至少包含：

```json
{
  "round_id": "round_001",
  "goal": "覆盖首批低风险查询和详情功能点",
  "started_at": "string",
  "finished_at": "string|null",
  "input_artifacts": [],
  "output_artifacts": [],
  "status": "completed|failed|waiting_human|stopped"
}
```

round 的生命周期：

```text
plan
  -> execute
  -> collect evidence
  -> AI analyze
  -> propose improvements
  -> decide next round
```

如果 AI 判断可以自动继续，且所有动作都属于低风险范围，则 orchestrator 可创建下一轮。否则进入运行中心人工确认。

### 4.5 自进化分层模型

自进化产物分三层：

1. **运行时修正**  
   只作用于当前 run 或下一轮，例如替换 locator 候选、调整等待时间、重试查询动作。

2. **项目级沉淀**  
   作用于同一被测系统后续 run，例如稳定页面入口、功能点别名、字段映射、测试数据建议。

3. **平台级晋升候选**  
   作用于多个系统的通用规则，例如表格行级动作识别、查询表单识别策略。必须人工批准。

AI 可以生成三类候选，但不能自行批准平台级晋升。

## 5. 阶段设计

### 5.1 Preflight 阶段

输入：

- 系统名称
- 首页 URL
- CDP URL
- 测试账号说明
- 探索范围
- 安全策略

检查：

- Python 可执行
- Playwright 可用
- CDP 可连接
- 当前浏览器有可用页面
- 当前页面是否疑似登录态
- 页面 DOM 是否可读取

输出：

- `preflight_result.json`
- `current_status.json`
- `progress_events.jsonl`

失败时进入：

- `login_required`
- `preflight_failed`

### 5.2 Discovery 阶段

Discovery 不再依赖模板先行。

核心步骤：

1. 连接 CDP。
2. 打开或定位首页。
3. 扫描可见导航候选。
4. 对候选导航按安全规则过滤。
5. 对同源候选执行有限深度访问。
6. 每个页面采集标题、URL、主要内容区、表格、表单、按钮、链接、弹窗触发器。
7. 捕获页面入口截图。
8. 生成页面入口树和页面入口列表。

默认限制：

- 最大探索深度：1
- 最大页面入口数：可配置，默认 30
- 单页最大功能点候选：可配置，默认 30
- 跳过退出、删除、提交、审批、发布等高风险动作

输出：

- `system_map.json`
- `navigation_tree.json`
- `page_entries.json`
- `screenshots_index.json`

### 5.3 Feature Analysis 阶段

对每个页面入口执行功能点识别。

识别来源：

- button
- link
- input + button 组合
- form
- table row action
- modal trigger
- tab
- toolbar
- dropdown menu

分类方式：

1. 规则先行  
   根据文本、DOM role、表单结构、URL、按钮位置做初分。

2. AI 辅助  
   使用结构化 schema 让模型判断业务语义、功能点类型和风险等级。

3. 置信度合并  
   规则与 AI 输出不一致时标记 `needs_review`。

输出：

- `feature_points.json`
- `feature_analysis_trace.json`

### 5.4 Review Gate 阶段

发现后进入人工审核门。

运行中心展示：

- 页面入口树
- 每页功能点
- 类型
- 风险等级
- 自动验证建议
- 截图证据

用户操作：

- include / exclude 页面入口
- include / exclude 功能点
- 修改名称
- 修改类型
- 标记前置数据
- 对特定动作授权

输出：

- `discovery_review.json`

如果用户选择自动继续，则系统基于默认安全策略生成 review。

### 5.5 Test Case Generation 阶段

输入：

- `page_entries.json`
- `feature_points.json`
- `discovery_review.json`
- 安全策略

生成：

- 导航类测试用例
- 查询类测试用例
- 重置类测试用例
- 详情类测试用例
- Tab 切换测试用例
- 表单前端校验测试用例
- 高风险动作入口确认测试用例

输出：

- `generated_test_cases.json`

测试用例不是需求验收用例，而是执行型测试用例。

### 5.6 Execution 阶段

执行器只执行低风险或已授权动作。

执行策略：

- 每个用例从所属页面入口标准起点开始。
- 执行前截图。
- 关键动作后截图。
- 出错时截图。
- 网络事件可选采集。
- 每步记录动作、locator、结果、耗时。

输出：

- `execution_results.json`
- `action_log.jsonl`
- `network_events.json`
- `screenshots_index.json`

### 5.7 Reporting 阶段

报告生成器读取稳定产物：

- `run_manifest.json`
- `page_entries.json`
- `feature_points.json`
- `generated_test_cases.json`
- `execution_results.json`
- `screenshots_index.json`
- `failure_summary.json`

输出：

- `reports/run_report.json`
- `reports/run_report.md`

报告章节：

1. 运行概况
2. 输入范围
3. 页面入口发现
4. 功能点识别
5. 自动执行结果
6. 未执行与跳过原因
7. 失败原因初分
8. 人工确认项
9. 证据索引
10. 后续建议

### 5.8 AI Round Analysis 阶段

每轮 execution 或 discovery 完成后，orchestrator 必须调用 AI 分析。

输入：

- `page_entries.json`
- `feature_points.json`
- `generated_test_cases.json`
- `execution_results.json`
- `action_log.jsonl`
- `screenshots_index.json`
- `network_events.json`
- `discovery_review.json`
- 人工录制和人工处理记录

输出：

- `round_analysis.json`
- `failure_clusters.json`
- `improvement_candidates.json`
- `next_round_plan.json`
- `promotion_candidates.json`

`round_analysis.json` 至少包含：

- 本轮目标
- 覆盖摘要
- 失败摘要
- 未执行原因
- 证据完整性检查
- AI 置信度
- 需要人工确认项

`next_round_plan.json` 至少包含：

- 是否建议继续
- 下一轮目标
- 计划页面入口
- 计划功能点
- 计划修正项
- 风险等级
- 是否需要人工批准

### 5.9 Next Round Decision 阶段

系统根据 AI 分析和 policy gate 做下一轮决策。

决策结果：

- `auto_continue`
- `wait_human_review`
- `wait_human_takeover`
- `stop_goal_completed`
- `stop_no_progress`
- `stop_budget_exhausted`
- `stop_policy_blocked`

自动继续条件：

- 下一轮只包含低风险动作。
- 不需要业务正确性判断。
- 不需要平台级沉淀批准。
- 上一轮没有连续重复失败。
- 仍在预算内。

否则运行中心必须展示确认任务。

### 5.10 Human Task 阶段

人工介入通过结构化任务表达。

任务类型：

- `select_priority_pages`
- `review_feature_points`
- `record_human_path`
- `correct_test_data`
- `approve_risk_policy`
- `review_next_round_plan`
- `review_project_promotion`
- `review_platform_promotion`

每个任务至少包含：

```json
{
  "task_id": "string",
  "task_type": "review_feature_points",
  "title": "string",
  "reason": "string",
  "input_refs": [],
  "ui_schema": {},
  "status": "pending|completed|cancelled",
  "result_artifact": "string|null"
}
```

运行中心根据 `ui_schema` 渲染表格、勾选框、表单、录制按钮或确认按钮。用户提交后生成结构化结果，但不直接编辑 JSON 文件；方案不能要求用户直接修改 JSON 或模板文件作为正常操作。

## 7. AI 使用边界

### 7.1 AI 负责

- 页面语义归类
- 功能点类型判断
- 用例步骤草案生成
- 失败原因摘要
- 报告自然语言摘要

### 7.2 AI 不负责

- 直接自由点击真实系统
- 绕过安全策略
- 决定高风险动作是否最终提交
- 直接把业务正确性判为验收通过
- 单独生成不可复现的执行结果

### 7.3 结构化输出要求

AI 输出必须经过 schema 校验。

不通过 schema 校验时：

- 回退规则分类
- 标记 `needs_review`
- 不阻断已发现页面入口的落盘

## 8. Playwright 执行边界

Playwright 是唯一执行层。

允许动作：

- goto
- click low-risk element
- fill safe field
- clear field
- select
- wait
- screenshot
- read text
- listen network
- open/close modal

禁止动作：

- 未授权 final submit
- delete confirm
- approve
- publish
- payment
- inventory mutation
- external notification trigger

所有动作必须经过 policy gate。

```text
POST /api/stage2/runs/:runId/analyze-round
POST /api/stage2/runs/:runId/continue-next-round
POST /api/stage2/runs/:runId/generate-report
GET  /api/stage2/runs/:runId/human-tasks
POST /api/stage2/runs/:runId/human-tasks/:taskId/complete
GET  /api/stage2/runs/:runId/artifacts/:artifactKey
```

operation API 可以保留为高级调试入口，但不应作为运行中心主流程。

### 9.1 API 约束

- 所有 API 返回 run 当前状态和下一步建议。
- 所有阶段推进必须写入 `progress_events.jsonl`。
- 所有用户提交必须生成结构化 artifact。
- artifact 访问必须走白名单 key，不暴露任意路径。
- 不允许 API 要求前端传入 raw JSON 文件路径作为常规输入。

## 12. 分阶段实施计划

### P0：恢复最初需求的最小闭环

目标：

用户不写模板，从已登录首页启动自动评测。

交付：

- `POST /api/stage2/runs`
- preflight
- discovery
- page_entries
- feature_points
- review 默认自动通过
- generated_test_cases
- low-risk execution
- round_analysis
- next_round_plan
- run_report.md

验收：

- 在追本溯源管理系统上至少发现 1 层页面入口。
- 至少识别 5 个功能点或明确说明为什么不足。
- 至少生成 3 条执行型测试用例。
- 至少执行 1 条低风险用例。
- 至少生成 1 份 AI 复盘和下一轮计划。
- 输出报告。

### P1：人工审核与可控执行

目标：

发现结果可人工审核，执行策略可控。

交付：

- 页面入口树审核 UI
- 功能点审核 UI
- 高风险策略 UI
- `discovery_review.json`
- 审核后重新生成测试用例

### P2：内部模板沉淀

目标：

把稳定功能点自动沉淀为项目级模板，但不要求用户手写。

交付：

- 自动模板候选
- locator hint 自动建议
- baseline 自动采样
- 项目级沉淀审阅

### P3：自动循环与自进化

目标：

系统能在低风险范围内自动进行多轮测试，并把 AI 分析转化为下一轮动作。

交付：

- 多 round run 管理
- 自动继续策略
- stop conditions
- failure cluster 驱动重试
- 项目级沉淀候选审阅
- 平台级晋升候选审阅

验收：

- 同一 run 至少能执行两轮。
- 第二轮计划来自第一轮 AI 分析。
- 人工可在运行中心批准或拒绝下一轮计划。

## 13. 风险与修正

### 13.1 风险：自动发现过浅

修正：

- 显示探索深度和跳过原因。
- 允许用户扩大范围后续跑。
- 对未进入页面保留候选状态。

### 13.2 风险：功能点误分类

修正：

- 规则与 AI 双通道判断。
- 低置信度进入人工审核。
- 用户修改结果写入 review patch。

### 13.3 风险：执行动作造成副作用

修正：

- 执行前经过 policy gate。
- 高风险动作只验证入口和确认弹窗。
- 最终提交默认禁止。

### 13.4 风险：模型不可用

修正：

- 模型失败不阻断 DOM 规则发现。
- AI 分类失败时标记 unknown/needs_review。
- 报告明确记录模型能力状态。

### 13.5 风险：报告只是一堆日志

修正：

- 报告按页面入口和功能点组织。
- 每个结论带截图和执行证据。
- 明确列出未执行原因。

### 13.6 风险：运行中心退化为脚本面板

修正：

- 所有主流程动作必须以 run 状态机表达。
- 前端只展示业务动作，不展示 CLI 命令作为主操作。
- operation session 只作为高级调试视图。

### 13.7 风险：AI 循环无限重试

修正：

- 设置最大轮次、最大连续失败次数、最大无改进次数和预算上限。
- 每轮必须比较覆盖率、通过率、失败簇变化和新增证据。
- 无改进时停止并给出原因。

### 13.8 风险：AI 自进化污染资产

修正：

- 所有沉淀候选带适用范围、证据和置信度。
- 项目级沉淀需要运行中心确认。
- 平台级晋升必须人工批准。

### 13.9 风险：人工处理结果不可追踪

修正：

- 所有人工提交都生成 artifact。
- 人工处理记录必须关联任务、操作者、时间、输入和结果。
- 后续 round 必须引用人工处理 artifact。

### 13.10 风险：界面更正与底层产物不一致

修正：

- 运行中心提交的是 patch artifact。
- Orchestrator 每次进入下一阶段前先合并 patch。
- 报告同时展示原始识别和人工修正。

## 14. 方案漏洞审查循环

### 14.1 第一轮：总控入口审查

问题：如果保留 CLI 为常规入口，用户体验会再次碎片化。

修正：运行中心为唯一普通用户入口；CLI 仅后台和高级调试。

### 14.2 第二轮：闭环完整性审查

问题：只做 discovery 或只做 report 都不足以满足 v3。

修正：P0 必须串通 `preflight -> discovery -> feature analysis -> case generation -> execution -> AI analysis -> report`。

### 14.3 第三轮：AI 价值审查

问题：AI 只写总结，不驱动后续动作。

修正：AI 输出必须生成 `next_round_plan.json`，并被状态机消费。

### 14.4 第四轮：人工介入审查

问题：人工仍需编辑 JSON。

修正：以 `human_tasks.json` + UI renderer 承载所有人工介入。

### 14.5 第五轮：安全审查

问题：自动循环放大副作用风险。

修正：每轮执行前经过 policy gate；高风险动作必须人工授权；默认只验证入口或前端校验。

### 14.6 第六轮：资产沉淀审查

问题：低质量 AI 建议污染模板和全局策略。

修正：沉淀分层，平台级晋升强制人工批准。

### 14.7 第七轮：可观测性审查

问题：用户不知道系统在跑什么。

修正：运行中心显示阶段、轮次、当前对象、事件流、截图、阻塞原因和下一步动作。

### 14.8 第八轮：验收审查

问题：实现完成组件但用户仍跑不通。

修正：验收以真实系统端到端 run 为准，不以模块数量为准。

### 14.9 当前信心结论

经过上述审查，v3 技术方案已覆盖当前已知的关键漏洞：入口碎片化、自动化不足、AI 不闭环、人工介入文件化、自进化失控、安全副作用、沉淀污染和报告不可复核。后续实现时应把本节作为设计审查清单逐项打勾。

## 15. v3 成功判据

v3 技术方案成功的标志不是模板数量增加，而是：

1. 新系统首次接入时，用户只填少量业务字段即可启动。
2. 系统自动生成页面入口和功能点清单。
3. 系统自动生成执行型测试用例。
4. 系统自动执行安全用例。
5. 系统自动输出报告。
6. 模板文件默认不出现在用户主流程里。
7. 每轮执行后 AI 自动生成复盘和下一轮计划。
8. 人工介入全部通过运行中心界面完成。
9. 至少一个真实系统能完成两轮低风险自动测试循环。
