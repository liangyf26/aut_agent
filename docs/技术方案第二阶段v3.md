# AI Agent 软件自动化评测平台技术方案（第二阶段 v3）

更新时间：2026-06-25

## 1. v3 技术方案目标

本文档是《技术方案第二阶段.md》的纠偏版，不覆盖旧文档。

v3 技术方案的目标是把第二阶段从“模板接入与样本验证工程”和“raw CDP 当前页快照”拉回到“Browser-use + Playwright 驱动的自动探索、自动用例生成、低风险自动验证、证据报告输出和自进化闭环”的主线。

v3 的核心技术命题是：

**模板可以继续存在，但不能再是用户主入口。系统必须以 run 为中心，先完整发现菜单入口树，再逐菜单探索页面和功能点，随后生成执行型测试用例、执行结果、复盘分析、沉淀候选和报告。**

技术路线的纠偏原则：

- Browser-use + Playwright 是探索和测试主链路。
- raw CDP 仅用于连接、预检、target 枚举、低层截图或兼容兜底，不能作为完整发现能力。
- 第一轮必须完成菜单入口完全遍历。
- 第二轮才逐个菜单入口进入页面探索。
- 后续轮次通过 AI/规则复盘扩大成功范围，而不是只做一次性扫描。
- 运行中心界面化必须服务于上述闭环，不能先做按钮面板再补能力。

## 2. 当前实现技术审计

### 2.1 已有可复用能力

当前代码中已有以下可复用能力：

- `prototype.stage2.main` 统一 CLI 入口
- `--explore-system-map`
- `--live-discovery`
- `--capture-human-recording`
- `--template-revision-checklist`
- `--validate-connected-template`
- `--validation-matrix`
- `DiscoveryArtifactWriter`
- `plan_live_discovery`
- `TemplateFlowExecutor`
- `TemplateActionRegistry`
- `ProgressManager`
- `run_report` / `platform_daily_report` 报告模型
- Node.js 运行中心 API 与 operation session

这些能力说明底层工程并非完全缺失。

### 2.2 当前主要技术偏差

当前偏差在于编排中心和数据模型：

1. **主实体错位**  
   当前用户主入口是模板和 operation step；v3 应以 `run` 为主实体。

2. **发现结果没有驱动后续执行**  
   `page_entries.json`、`feature_points.json` 可以生成，但缺少从功能点自动生成执行型测试用例并执行的稳定链路。

3. **模板暴露过早**  
   `template.json`、`baseline.json`、`data_schema.json`、`locator_hints.json` 被用户当作必填配置。这些应转为内部中间资产。

4. **运行中心是命令面板，不是评测工作流**  
   运行中心目前能触发多个命令，但没有把“发现-审核-生成-执行-报告”表达成一个连续 run。

5. **报告不是完整 run 的自然产物**  
   报告能力存在，但没有由新系统自动评测主流程稳定产出。

6. **探索技术路线跑偏**
   v3 run 的真实浏览器路径仍主要依赖 raw CDP 对当前页做一次 DOM 快照，未使用 Browser-use 进行页面语义理解和多步导航，也未使用 Playwright 做完整菜单展开、逐项点击和状态恢复。

7. **菜单遍历缺失**
   当前实现没有把“第一轮完整菜单入口遍历”作为独立阶段，导致首页存在多个一级菜单时也可能只产出 1 个首页页面入口。

8. **下一轮状态机不完整**
   `进入下一轮` 可能只创建 planned round，而不是执行下一轮探索/测试；运行中心和 orchestrator 的动作语义需要重新对齐。

## 3. v3 总体架构

v3 采用以下结构：

```text
运行中心 UI
  ├─ 新建运行
  ├─ 运行状态
  ├─ 菜单入口树
  ├─ 页面入口树
  ├─ 功能点审核
  ├─ 执行型测试用例
  ├─ 执行证据
  └─ 总体报告

Node.js 平台 API
  ├─ run 创建与查询
  ├─ run 阶段推进
  ├─ artifact 安全访问
  └─ 人工审核结果保存

Python Stage-2 Orchestrator
  ├─ preflight
  ├─ login/session check
  ├─ round 1 menu discovery
  ├─ round 2 page exploration
  ├─ feature analysis
  ├─ review gate
  ├─ test case generation
  ├─ low-risk execution
  ├─ round analysis
  ├─ knowledge promotion
  ├─ evidence collection
  └─ report generation

执行器层
  ├─ Browser-use Agent
  ├─ Browser-use tools
  ├─ Playwright browser/session connector
  ├─ Playwright menu explorer
  ├─ Playwright page explorer
  ├─ Playwright action executor
  ├─ Screenshot collector
  └─ Network/event collector

AI/规则层
  ├─ 菜单语义理解
  ├─ 页面语义分类
  ├─ 功能点归类
  ├─ 测试用例生成
  ├─ 失败归因
  ├─ 经验沉淀建议
  ├─ 下一轮计划
  └─ 报告摘要

产物层
  ├─ run_manifest.json
  ├─ menu_tree.json
  ├─ menu_entries.json
  ├─ page_entries.json
  ├─ feature_points.json
  ├─ generated_test_cases.json
  ├─ execution_results.json
  ├─ screenshots_index.json
  └─ reports/run_report.md
```

### 3.1 运行中心是总控平面

v3 技术实现必须把运行中心作为总控平面，而不是把运行中心做成脚本按钮集合。

运行中心是第二阶段的查询、操作、监控、人工介入和报告查看的集中平台。

运行中心负责：

- run 创建、配置、启动、暂停、恢复、终止。
- run 列表、历史轮次和 artifact 查询。
- 实时进度监控和事件流展示。
- 页面入口树、功能点、测试用例、执行结果和报告展示。
- 人工审核、人工录制、数据更正、策略授权。
- AI 复盘、下一轮计划和沉淀候选审阅。

Python CLI 仍可作为后台执行器入口，但普通用户不应通过 CLI 推进常规流程。

### 3.2 控制流与数据流

v3 的控制流由运行中心发起：

```text
运行中心
  -> Node.js Run API
  -> Python Stage-2 Orchestrator
  -> Browser-use + Playwright + AI/规则分析 + Artifact Writer
  -> 稳定 run artifacts
  -> 运行中心轮询或订阅状态
```

v3 的数据流必须以 run artifact contract 为准：

```text
input_config
  -> preflight_result
  -> menu_tree + menu_entries
  -> page_entries + feature_points
  -> discovery_review
  -> generated_test_cases
  -> execution_results
  -> round_analysis + next_round_plan
  -> run_report
```

任何阶段不得只把结果打印到终端。

### 3.3 后台脚本的定位

现有 CLI 能力按三类处理：

1. **后台执行入口**  
   由 Node.js API 调用，用户不直接接触。例如 discovery、execution、reporting。

2. **高级调试入口**  
   对开发者可见。例如单独运行 `--live-discovery`、`--validate-connected-template`。

3. **内部迁移资产**  
   可保留但不进入主界面。例如 `--bootstrap-template`、`--template-revision-checklist`。

运行中心主流程不得要求用户复制命令。

## 4. v3 核心运行模型

### 4.1 Run 是唯一主实体

每次用户点击“开始自动评测”后，系统创建一个 run：

```json
{
  "run_id": "stage2_20260624_101500_xxxxxxxx",
  "system_name": "追本溯源管理系统",
  "entry_url": "https://example.com/home",
  "cdp_url": "http://localhost:9222",
  "status": "discovering",
  "created_at": "2026-06-24T10:15:00+08:00"
}
```

所有 discovery、review、test case、execution、report 都归属这个 run。

### 4.2 Template 是内部派生资产

v3 中模板分为两类：

1. **临时执行计划**  
   从功能点自动生成，用于本 run 执行，不要求用户编辑。

2. **可沉淀模板**  
   当某个功能点多次验证稳定后，系统可建议沉淀为项目级模板。

用户默认不需要打开模板文件。

### 4.3 Artifact 是稳定接口

运行中心不应读取零散临时文件，而应读取稳定 artifact contract：

```text
artifacts/stage2/runs/<run_id>/
  run_manifest.json
  input_config.json
  progress_events.jsonl
  current_status.json
  system_map.json
  navigation_tree.json
  menu_tree.json
  menu_entries.json
  menu_traversal_log.jsonl
  page_entries.json
  page_exploration_log.jsonl
  feature_points.json
  discovery_review.json
  generated_test_cases.json
  execution_results.json
  screenshots_index.json
  failure_summary.json
  reports/run_report.json
  reports/run_report.md
```

### 4.4 Round 是自进化循环单元

一个 run 可以包含多个 round。

v3 默认轮次语义：

1. `round_001`：菜单入口完全遍历。目标是得到可信菜单树，不做深层功能测试。
2. `round_002`：按菜单入口逐页探索。目标是进入菜单叶子或用户指定入口，沉淀页面入口和功能点候选。
3. `round_003+`：执行型用例生成、测试执行、失败归因、范围扩大和项目级沉淀。

每个 round 至少包含：

```json
{
  "round_id": "round_001",
  "round_type": "menu_discovery|page_exploration|test_execution|review_only",
  "goal": "完成首页菜单入口完全遍历",
  "started_at": "string",
  "finished_at": "string|null",
  "input_artifacts": [],
  "output_artifacts": [],
  "status": "completed|failed|waiting_human|stopped"
}
```

round 的生命周期：

```text
plan
  -> execute
  -> collect evidence
  -> AI analyze
  -> propose improvements
  -> decide next round
```

如果 AI 判断可以自动继续，且所有动作都属于低风险范围，则 orchestrator 可创建下一轮。否则进入运行中心人工确认。

`continue-next-round` 不能只把状态写成 `planned` 后停住。它必须二选一：

- 直接提交后台执行任务，并把状态变为 `queued|running`。
- 明确返回 `waiting_human|executor_unavailable`，在运行中心展示阻塞原因和可操作修复项。

### 4.5 自进化分层模型

自进化产物分三层：

1. **运行时修正**  
   只作用于当前 run 或下一轮，例如替换 locator 候选、调整等待时间、重试查询动作。

2. **项目级沉淀**  
   作用于同一被测系统后续 run，例如稳定页面入口、功能点别名、字段映射、测试数据建议。

3. **平台级晋升候选**  
   作用于多个系统的通用规则，例如表格行级动作识别、查询表单识别策略。必须人工批准。

AI 可以生成三类候选，但不能自行批准平台级晋升。

## 5. 阶段设计

### 5.1 Preflight 阶段

输入：

- 系统名称
- 首页 URL
- CDP URL
- 测试账号说明
- 探索范围
- 安全策略

检查：

- Python 可执行
- Playwright 可用
- CDP 可连接
- 当前浏览器有可用页面
- 当前页面是否疑似登录态
- 页面 DOM 是否可读取

输出：

- `preflight_result.json`
- `current_status.json`
- `progress_events.jsonl`

失败时进入：

- `login_required`
- `preflight_failed`

### 5.2 Round 1 Menu Discovery 阶段

Round 1 的唯一目标是完整遍历菜单入口树，不做功能点测试。

技术主链路：

1. Playwright 连接已登录浏览器会话。
2. Playwright 定位当前可操作 page target；如存在多个 target，逐个读取 title、URL、可见文本，选择与入口 URL 或测试目标最匹配的 target。
3. Playwright 执行确定性菜单候选扫描。
4. Browser-use Agent 在候选不足或菜单语义不确定时参与理解和目标寻找。
5. Playwright 按候选菜单逐级展开、点击和恢复状态。
6. 每次展开、点击、URL/route 变化都写入菜单遍历日志。
7. 每个一级菜单展开后截图。
8. 输出菜单树和菜单项列表。

菜单候选扫描必须过滤非业务 chrome。顶部工具栏、字号/布局选择、个人中心、退出登录、Logo、搜索框提示、浏览器或组件状态提示等元素不得作为业务菜单入口；若误入这些入口导致白屏，应记录截图证据并在下一轮修正扫描策略。

菜单候选扫描至少覆盖：

- `.el-menu`
- `.el-menu-item`
- `.el-sub-menu`
- `[role=menu]`
- `[role=menuitem]`
- `[class*=menu]`
- `[class*=sidebar]`
- `[class*=nav]`
- `aside nav ul li`
- 文本可见且有 click handler 的 `div`、`span`、`li`

参考 `demo/suyuan_test3.py`，菜单探索工具至少包括：

- `get_page_info`：获取标题和 URL。
- `get_page_text`：获取当前页面完整可见文本。
- `get_clickable_elements`：获取所有可点击元素，必须包含菜单类选择器。
- `forced_screenshot`：关键节点截图。
- `script_close_popups`：关闭弹窗和遮罩。

Round 1 输出：

- `menu_tree.json`
- `menu_entries.json`
- `menu_traversal_log.jsonl`
- `navigation_tree.json`
- `screenshots_index.json`

Round 1 判定：

- 如果首页可见 3 个一级菜单，则必须发现 3 个一级菜单。
- 如果指定目标如“线上备案申请”是二级菜单，则必须在菜单树中出现。
- 如果只产出首页 1 个页面入口，则状态为 `menu_discovery_incomplete`，不能进入 completed。

### 5.3 Round 2 Page Exploration 阶段

Round 2 基于 Round 1 的 `menu_tree.json` 和 `menu_entries.json`，逐个进入叶子菜单或被选中的菜单入口，探索页面内容。

核心步骤：

1. 读取已审核或默认通过的菜单入口。
2. Playwright 进入菜单入口并等待页面稳定。
3. Browser-use 在页面结构复杂或功能语义不明确时辅助理解。
4. 采集标题、URL、route、主内容区、表格、表单、按钮、链接、弹窗触发器。
5. 识别默认可见功能点。
6. 仅执行轻量、可逆、低风险交互以暴露更多显式功能点，如展开筛选区、切换 Tab、打开详情弹窗。
7. 捕获页面入口截图和关键交互截图。
8. 生成页面入口清单和页面级探索日志。

页面入口写入前应按规范化 URL 和菜单路径去重，`/` 与 `/index` 这类首页别名只能保留一个可信入口。进入入口后如果 DOM 快照无有效可见文本或页面呈现白屏，应将页面标记为不可达/白屏状态并保留截图证据，不能把它当成已覆盖业务页面。

默认限制：

- 最大菜单入口数：可配置，默认 30。
- 单页最大功能点候选：可配置，默认 30。
- 轻量交互深度：默认 1。
- 跳过退出、删除、提交、审批、发布等高风险动作。

Round 2 输出：

- `page_entries.json`
- `page_exploration_log.jsonl`
- `feature_points.json`
- `screenshots_index.json`

### 5.4 Feature Analysis 阶段

对每个页面入口执行功能点识别。

识别来源：

- button
- link
- input + button 组合
- form
- table row action
- modal trigger
- tab
- toolbar
- dropdown menu

分类方式：

1. 规则先行  
   根据文本、DOM role、表单结构、URL、按钮位置做初分。

2. AI 辅助  
   使用结构化 schema 让模型判断业务语义、功能点类型和风险等级。

3. 置信度合并  
   规则与 AI 输出不一致时标记 `needs_review`。

输出：

- `feature_points.json`
- `feature_analysis_trace.json`

### 5.5 Review Gate 阶段

发现后进入人工审核门。

运行中心展示：

- 页面入口树
- 每页功能点
- 类型
- 风险等级
- 自动验证建议
- 截图证据

用户操作：

- include / exclude 页面入口
- include / exclude 功能点
- 修改名称
- 修改类型
- 标记前置数据
- 对特定动作授权

输出：

- `discovery_review.json`

如果用户选择自动继续，则系统基于默认安全策略生成 review。

### 5.6 Test Case Generation 阶段

输入：

- `page_entries.json`
- `feature_points.json`
- `discovery_review.json`
- 安全策略

生成：

- 导航类测试用例
- 查询类测试用例
- 重置类测试用例
- 详情类测试用例
- Tab 切换测试用例
- 表单前端校验测试用例
- 高风险动作入口确认测试用例

输出：

- `generated_test_cases.json`

测试用例不是需求验收用例，而是执行型测试用例。

### 5.7 Execution 阶段

执行器只执行低风险或已授权动作。

执行策略：

- 每个用例从所属页面入口标准起点开始。
- 执行前截图。
- 关键动作后截图。
- 出错时截图。
- 网络事件可选采集。
- 每步记录动作、locator、结果、耗时。

输出：

- `execution_results.json`
- `action_log.jsonl`
- `network_events.json`
- `screenshots_index.json`

### 5.8 Reporting 阶段

报告生成器读取稳定产物：

- `run_manifest.json`
- `page_entries.json`
- `feature_points.json`
- `generated_test_cases.json`
- `execution_results.json`
- `screenshots_index.json`
- `failure_summary.json`

输出：

- `reports/run_report.json`
- `reports/run_report.md`

报告章节：

1. 运行概况
2. 输入范围
3. 页面入口发现
4. 功能点识别
5. 自动执行结果
6. 未执行与跳过原因
7. 失败原因初分
8. 人工确认项
9. 证据索引
10. 后续建议

### 5.9 AI Round Analysis 阶段

每轮 execution 或 discovery 完成后，orchestrator 必须调用 AI 分析。

输入：

- `page_entries.json`
- `feature_points.json`
- `generated_test_cases.json`
- `execution_results.json`
- `action_log.jsonl`
- `screenshots_index.json`
- `network_events.json`
- `discovery_review.json`
- 人工录制和人工处理记录

输出：

- `round_analysis.json`
- `failure_clusters.json`
- `improvement_candidates.json`
- `next_round_plan.json`
- `promotion_candidates.json`

`round_analysis.json` 至少包含：

- 本轮目标
- 覆盖摘要
- 失败摘要
- 未执行原因
- 证据完整性检查
- AI 置信度
- 需要人工确认项

`next_round_plan.json` 至少包含：

- 是否建议继续
- 下一轮目标
- 计划页面入口
- 计划功能点
- 计划修正项
- 风险等级
- 是否需要人工批准

### 5.10 Next Round Decision 阶段

系统根据 AI 分析和 policy gate 做下一轮决策。

决策结果：

- `auto_continue`
- `wait_human_review`
- `wait_human_takeover`
- `stop_goal_completed`
- `stop_no_progress`
- `stop_budget_exhausted`
- `stop_policy_blocked`

自动继续条件：

- 下一轮只包含低风险动作。
- 不需要业务正确性判断。
- 不需要平台级沉淀批准。
- 上一轮没有连续重复失败。
- 仍在预算内。
- 用户指定优先目标仍未覆盖，包括未发现、已发现但页面不可达、白屏、未识别功能点或未形成执行结果。

`stop_goal_completed` 只能在当前范围内没有缺失目标、没有已发现但未覆盖目标、没有需继续的低风险修正时使用。运行中心在继续下一轮时，如果发现旧的 `next_round_plan.json` 误写为 `stop_goal_completed`，但持久化产物仍显示存在未覆盖优先目标，应重新计算下一轮计划并继续执行。

否则运行中心必须展示确认任务。

### 5.11 Human Task 阶段

人工介入通过结构化任务表达。

任务类型：

- `select_priority_pages`
- `review_feature_points`
- `record_human_path`
- `correct_test_data`
- `approve_risk_policy`
- `review_next_round_plan`
- `review_project_promotion`
- `review_platform_promotion`

每个任务至少包含：

```json
{
  "task_id": "string",
  "task_type": "review_feature_points",
  "title": "string",
  "reason": "string",
  "input_refs": [],
  "ui_schema": {},
  "status": "pending|completed|cancelled",
  "result_artifact": "string|null"
}
```

运行中心根据 `ui_schema` 渲染表格、勾选框、表单、录制按钮或确认按钮。用户提交后生成结构化结果，但不直接编辑 JSON 文件；方案不能要求用户直接修改 JSON 或模板文件作为正常操作。

## 7. AI 使用边界

### 7.1 AI 负责

- 通过 Browser-use 参与菜单语义理解、目标页面寻找和多步探索规划
- 对 Playwright 采集到的菜单、页面、文本和截图做页面语义归类
- 功能点类型判断
- 用例步骤草案生成
- 失败原因摘要
- 每轮结果复盘、错误归因、经验沉淀和下一轮计划生成
- 报告自然语言摘要

### 7.2 AI 不负责

- 脱离 Browser-use 工具约束直接自由点击真实系统
- 绕过安全策略
- 决定高风险动作是否最终提交
- 直接把业务正确性判为验收通过
- 单独生成不可复现的执行结果
- 在没有截图、DOM、动作日志或网络证据时声称已发现/已测试

### 7.3 结构化输出要求

AI 输出必须经过 schema 校验。

不通过 schema 校验时：

- 回退规则分类
- 标记 `needs_review`
- 不阻断已发现页面入口的落盘
- 在运行中心展示模型不可用、输出不合格或证据不足的明确原因

## 8. Browser-use 与 Playwright 执行边界

v3 的真实发现和测试主链路是 Browser-use + Playwright：

- Browser-use Agent 负责在受控工具集合中进行页面理解、菜单寻找、目标定位和复杂路径规划。
- Playwright 负责连接真实浏览器会话、执行确定性动作、采集截图和 DOM/文本/网络证据，并回放稳定路径。
- raw CDP 仅用于浏览器连接检查、target 枚举、低层截图或 Playwright 失败后的诊断兜底，不作为菜单发现主执行器。

允许动作：

- goto
- expand menu
- click low-risk menu/page element
- fill safe field
- clear field
- select
- wait
- screenshot
- read text
- listen network
- open/close modal

禁止动作：

- 未授权 final submit
- delete confirm
- approve
- publish
- payment
- inventory mutation
- external notification trigger

所有动作必须经过 policy gate。Browser-use 产生的动作计划必须被记录为可审计 action log，并由 Playwright 或等价受控工具执行出可复核证据。

## 12. 分阶段实施计划

### P0：接通 Browser-use + Playwright 菜单发现

目标：

用户不写模板，从已登录首页启动第一轮探索，完整发现菜单入口树。

交付：

- `POST /api/stage2/runs`
- preflight
- Browser-use + Playwright 连接真实浏览器
- 一级/二级/三级菜单候选扫描
- 菜单展开、点击、恢复和截图
- `menu_tree.json`
- `menu_entries.json`
- `menu_traversal_log.jsonl`
- 第一轮 `round_analysis.json`
- 第二轮 `next_round_plan.json`

验收：

- 在追本溯源管理系统首页可见 3 个一级菜单时，第一轮必须发现 3 个一级菜单或明确标记不可达原因。
- “线上备案申请”作为二级菜单时，必须进入 `menu_tree.json` / `menu_entries.json`，否则标记 `menu_discovery_incomplete`。
- 运行中心显示菜单树、遍历进度、截图和缺失/失败原因。
- `进入下一轮` 能真正提交第二轮页面探索任务，不能停留在无动作的 `已计划`。

### P1：基于菜单树的页面探索

目标：

第二轮按菜单入口逐页探索，生成页面入口和功能点候选。

交付：

- 页面入口树
- 功能点候选
- `page_exploration_log.jsonl`
- 页面截图索引
- 页面入口树审核 UI
- 功能点审核 UI
- `discovery_review.json`

验收：

- 第二轮至少尝试进入所有可达菜单叶子或用户指定菜单。
- 对失败菜单给出权限、弹窗、路由、加载、选择器等明确失败原因。
- 页面入口数量与菜单叶子数量的差异必须可解释。

### P2：测试用例生成与可控执行

目标：

基于已审核功能点生成执行型测试用例，并按安全策略执行。

交付：

- generated_test_cases

- 测试环境全权限模式 policy gate
- 高风险策略 UI
- execution_results
- run_report.md

### P3：内部模板沉淀

目标：

目标：

## 13. 风险与修正

### 13.1 风险：自动发现过浅

修正：

- 第一轮强制以菜单入口完全遍历为目标，而不是只扫当前 DOM。
- 使用 Browser-use + Playwright 完成菜单语义识别、展开、点击、恢复和截图。
- 显示探索深度、菜单覆盖率、跳过原因和疑似缺失菜单。
- 对未进入菜单保留候选状态，并在下一轮计划中优先处理。

### 13.1.1 风险：raw CDP 伪装成真实发现

修正：

- raw CDP 只能作为连接检查、target 枚举和诊断兜底。
- 若本轮没有 Browser-use/Playwright 动作日志和截图证据，不得标记 `menu_discovery completed`。
- 运行中心必须区分“浏览器 target 数”和“已发现菜单/页面入口数”。

### 13.2 风险：功能点误分类

修正：

- 规则与 AI 双通道判断。
- 低置信度进入人工审核。
- 用户修改结果写入 review patch。

### 13.3 风险：执行动作造成副作用

修正：

- 执行前经过 policy gate。
- 高风险动作只验证入口和确认弹窗。
- 最终提交默认禁止。

### 13.4 风险：模型不可用

修正：

- 模型失败不阻断 DOM 规则发现。
- AI 分类失败时标记 unknown/needs_review。
- 报告明确记录模型能力状态。
- Browser-use 不可用时可跑 Playwright 规则遍历，但必须把能力降级显示在运行中心。

### 13.4.1 风险：Browser-use 幻觉式完成

修正：

- Browser-use 的每个发现结论必须关联 Playwright 证据：截图、DOM 文本、URL/route 或动作日志。
- 没有关联证据的条目只能作为候选，不能作为已发现页面或已通过用例。
- AI 复盘必须显示 `model_status`、`evidence_quality` 和置信度。

### 13.5 风险：报告只是一堆日志

修正：

- 报告按页面入口和功能点组织。
- 每个结论带截图和执行证据。
- 明确列出未执行原因。

### 13.6 风险：运行中心退化为脚本面板

修正：

- 所有主流程动作必须以 run 状态机表达。
- 前端只展示业务动作，不展示 CLI 命令作为主操作。
- operation session 只作为高级调试视图。

### 13.7 风险：AI 循环无限重试

修正：

- 设置最大轮次、最大连续失败次数、最大无改进次数和预算上限。
- 每轮必须比较覆盖率、通过率、失败簇变化和新增证据。
- 无改进时停止并给出原因。

### 13.7.1 风险：进入下一轮后停在已计划

修正：

- `next_round_plan` 是计划产物，不等同于执行。
- `continue-next-round` 必须提交后台任务并写入 `progress_events.jsonl`。
- 如果执行器不可用、Browser-use/Playwright 未连接、需要人工确认或预算不足，必须进入明确阻塞状态并给出修复动作。

### 13.8 风险：AI 自进化污染资产

修正：

- 所有沉淀候选带适用范围、证据和置信度。
- 项目级沉淀需要运行中心确认。
- 平台级晋升必须人工批准。

### 13.9 风险：人工处理结果不可追踪

修正：

- 所有人工提交都生成 artifact。
- 人工处理记录必须关联任务、操作者、时间、输入和结果。
- 后续 round 必须引用人工处理 artifact。

### 13.10 风险：界面更正与底层产物不一致

修正：

- 运行中心提交的是 patch artifact。
- Orchestrator 每次进入下一阶段前先合并 patch。
- 报告同时展示原始识别和人工修正。

## 14. 方案漏洞审查循环

### 14.1 第一轮：总控入口审查

问题：如果保留 CLI 为常规入口，用户体验会再次碎片化。

修正：运行中心为唯一普通用户入口；CLI 仅后台和高级调试。

### 14.2 第二轮：闭环完整性审查

问题：只做 discovery 或只做 report 都不足以满足 v3。

修正：P0/P1 必须先串通 `preflight -> menu discovery -> page exploration -> round analysis -> next_round_plan`，随后 P2/P4 再串通 `feature analysis -> case generation -> execution -> AI/rule analysis -> report -> next round execution`。

### 14.3 第三轮：AI 价值审查

问题：AI 只写总结，不驱动后续动作。

修正：AI 输出必须生成 `next_round_plan.json`，并被状态机消费。

### 14.3.1 第三轮补充：技术主线审查

问题：方案若停留在 raw CDP 或当前页 DOM 扫描，会再次无法发现二级菜单和跨菜单页面。

修正：真实发现主链路必须是 Browser-use + Playwright；raw CDP 只做连接、target 枚举和诊断兜底。第一轮验收以菜单树完整性为准。

### 14.4 第四轮：人工介入审查

问题：人工仍需编辑 JSON。

修正：以 `human_tasks.json` + UI renderer 承载所有人工介入。

### 14.5 第五轮：安全审查

问题：自动循环放大副作用风险。

修正：每轮执行前经过 policy gate；高风险动作必须人工授权；默认只验证入口或前端校验。

### 14.6 第六轮：资产沉淀审查

问题：低质量 AI 建议污染模板和全局策略。

修正：沉淀分层，平台级晋升强制人工批准。

### 14.7 第七轮：可观测性审查

问题：用户不知道系统在跑什么。

修正：运行中心显示阶段、轮次、当前菜单/页面/功能点、事件流、截图、Browser-use/Playwright/模型状态、阻塞原因和下一步动作。

### 14.8 第八轮：验收审查

问题：实现完成组件但用户仍跑不通。

修正：验收以真实系统端到端 run 为准，不以模块数量为准。

### 14.9 当前信心结论

经过上述审查，v3 技术方案已覆盖当前已知的关键漏洞：入口碎片化、自动化不足、raw CDP 伪发现、菜单遍历缺失、AI 不闭环、人工介入文件化、自进化失控、安全副作用、沉淀污染和报告不可复核。后续实现时应把本节作为设计审查清单逐项打勾。

## 15. v3 成功判据

v3 技术方案成功的标志不是模板数量增加，而是：

1. 新系统首次接入时，用户只填少量业务字段即可启动。
2. 第一轮使用 Browser-use + Playwright 完成菜单入口完全遍历，输出可复核 `menu_tree.json` 和截图证据。
3. 第二轮按菜单入口逐页探索，自动生成页面入口和功能点清单。
4. 系统自动生成执行型测试用例。
5. 系统按安全策略自动执行用例。
6. 每轮完成后产出错误分析、经验沉淀候选和下一轮计划。
7. `进入下一轮` 会实际驱动下一轮后台任务或给出明确阻塞原因。
8. 系统自动输出可直接阅读且可追溯证据的报告。
6. 模板文件默认不出现在用户主流程里。
7. 每轮执行后 AI 自动生成复盘和下一轮计划。
8. 人工介入全部通过运行中心界面完成。
9. 至少一个真实系统能完成两轮低风险自动测试循环。
