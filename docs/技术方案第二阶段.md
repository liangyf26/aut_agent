# AI Agent 软件自动化评测平台技术方案（第二阶段）

## 1. 文档目标

本文档用于定义第二阶段 Python 执行子系统原型的总体技术方案、模块边界、执行流程、循环优化机制和人工介入机制。

第二阶段的目标不是替换现有 Node.js 平台，而是在独立原型中验证以下能力：

- 当前测试账号权限范围内的页面入口发现
- 页面入口下功能点识别
- 单功能点基础路径验证
- 结构化测试证据输出
- 基于失败簇的循环优化和项目级技能沉淀

## 2. 总体设计原则

### 2.1 总原则

第二阶段采用以下总原则：

**AI 负责发现、规划、分析与修正建议；Playwright 负责确定性执行；Browser Use 主要用于受控探索。**

执行边界如下：

- 发现阶段可使用 Browser Use + Playwright 协同
- 验证阶段默认采用 Playwright only
- 仅对明确标记为视觉兜底的场景，允许 Browser Use 参与验证

### 2.2 不让模型承担全部执行责任

不采用“让本地小模型自由端到端完成全部浏览器操作”的方案，原因如下：

- 本地模型在结构化输出、复杂页面理解和长链推理方面稳定性不足
- 纯 Agent 自由执行难以保证结果可复现
- Prompt 修正和工具调用自由度过高时，失败模式难以收敛

因此第二阶段采用：

- 上层：多 Agent 协作规划与分析
- 下层：受控执行器负责导航、点击、输入、截图、等待、下载和状态恢复

## 3. 技术栈选型

### 3.1 Python 原型栈

第二阶段原型采用 Python 技术栈，推荐核心依赖：

- `browser-use`
- `playwright`
- `pydantic`
- `fastapi` 或轻量本地服务层（仅在需要人工介入界面时启用）
- `jinja2` 或同类模板工具用于 HTML 报告生成

### 3.2 LLM 接入策略

执行子系统统一通过 OpenAI 兼容接口接入 LLM Provider，优先支持：

- 本地或内网模型：
  - qwen35B
  - deepseek v4 flash
  - 定制模型
- 云端模型：
  - 仅作为补充或对照验证，不作为第二阶段默认依赖

### 3.3 模型能力预检

第二阶段原型启动前，必须先执行模型能力预检，不能只凭模型名称或单次聊天成功就进入主流程。

预检至少要区分以下能力：

- 普通 chat completion
- `response_format.type=json_object`
- `response_format.type=json_schema`
- tool calling
- Browser Use 结构化输出兼容性
- Browser Use 专用封装兼容性

预检结果必须落盘，并作为后续任务路由的输入。

### 3.4 模型能力分层

不同模型不强行承担相同任务，建议按能力分层：

- 文本规划模型：负责步骤规划、功能点归类、失败归因、报告总结
- 视觉探索模型：负责页面语义理解、菜单识别、页面入口和功能点推断
- 确定性执行器：负责浏览器低层动作，不依赖模型实时生成每一个具体动作

如果项目中只有单一小模型可用，则优先保证：

- 文本规划能力
- 基础页面理解能力

把复杂视觉判断和深层语义推理降级为人工接管或人工审核。

### 3.5 任务路由原则

模型能力预检通过后，主代理必须基于能力标签进行任务路由：

- 仅通过普通 chat 的模型，只能进入摘要、分类、简单归纳任务
- 通过 tool calling 但不通过 `json_schema` 的模型，不得进入 Browser Use 的 `ChatOpenAI` 结构化输出路径
- 通过 `json_schema` 的模型，才可进入 Browser Use 结构化输出路径
- 若模型在 Browser Use 的专用封装中表现不同，必须按封装结果打标签，不得混用

### 3.6 路由与发现策略分离

第二阶段当前已经把“能力路由”和“发现策略”明确拆开：

- **能力路由（capability routing）**：决定 discovery / verification / reporting 三个阶段分别允许采用什么模式
- **发现策略（discovery strategy）**：在当前 run 中，决定 discovery 是阻断、仅模板播种、模板播种后 live enrich，还是复用上一轮已完成的 discovery 结果

当前原型中，发现策略至少区分以下状态：

- `blocked`
- `template_seed_only`
- `live_enrich`
- `skip_completed_discovery`

这样做的目的，是避免把“模型支持某种 Browser Use 结构化路径”直接误写成“本轮一定执行 Browser Use discovery 主线”。

## 4. 总体架构

第二阶段原型建议采用如下架构：

```text
项目配置层
  ├─ 测试账号 / 登录模式 / 风险白名单 / 前置数据说明
  ├─ LLM Provider 配置
  └─ 运行预算与停止条件

主代理调度层
  ├─ 发现阶段调度
  ├─ 验证阶段调度
  ├─ 失败簇分析与循环调度
  ├─ 人工接管调度
  └─ 日报与沉淀调度

Agent 协作层
  ├─ 页面探索代理
  ├─ 功能点分析代理
  ├─ 失败分析代理
  ├─ 人机交互代理
  ├─ 技能沉淀代理
  └─ 报告整理代理

执行器层
  ├─ Playwright 确定性执行器
  ├─ Browser Use 受控探索器
  ├─ 截图/下载/日志采集器
  └─ 状态恢复与回放器

产物层
  ├─ page_entries.json
  ├─ feature_points.json
  ├─ executions.json
  ├─ failure_clusters.json
  ├─ reports/
  ├─ screenshots/
  └─ project_knowledge/
```

### 4.1 原型实现策略

第二阶段原型阶段采用：

**单主进程 + 多逻辑代理 + 受控 worker**

含义如下：

- 主代理是单一调度器，负责阶段切换、循环判停、失败簇调度和人工介入路由
- 页面探索代理、功能点分析代理、失败分析代理、技能沉淀代理、报告整理代理在原型阶段均作为逻辑角色存在，不要求一开始拆分为独立操作系统进程
- 浏览器执行、截图、下载、报告渲染等高成本或强副作用能力，可视需要采用少量受控 worker 承担
- 原型阶段不引入完整的多进程消息总线或分布式编排系统

采用该策略的原因是：

- 当前阶段优先验证执行子系统能力边界，而不是验证复杂进程编排
- 单主进程更利于追踪状态、调试循环和复现实验结果
- 受控 worker 可在不引入全量多进程复杂度的前提下隔离高副作用任务

## 5. 多 Agent 角色设计

### 5.1 主代理（Orchestrator Agent）

主代理负责：

- 为每个阶段和子代理设定目标
- 检查循环是否卡住
- 判断是否进入下一轮
- 判断是否触发人工接管
- 判断哪些经验只保留项目级，哪些需要提交为平台级候选

主代理不直接执行页面低层操作。
主代理在原型阶段运行于单一主进程内。

### 5.2 页面探索代理（Explorer Agent）

负责：

- 遍历当前测试账号可见的菜单和页面入口
- 生成页面地图
- 识别页面入口清单
- 对页面进行轻量交互以暴露显式功能点

执行方式：

- Browser Use 负责高层页面理解
- Playwright 负责底层确定性执行

页面探索代理在原型阶段作为逻辑代理存在，由主代理按任务调度。
页面探索代理只能在发现阶段常规使用 Browser Use；进入验证阶段后，Browser Use 只能作为视觉兜底能力被显式调用。

### 5.3 功能点分析代理（Feature Analyst Agent）

负责：

- 基于页面入口、页面元素和操作路径推断功能点
- 按固定模板分类功能点类型
- 为每个功能点生成执行型测试用例

### 5.4 失败分析代理（Failure Analyst Agent）

负责：

- 对失败的功能点进行归因
- 区分环境因素、技术因素、页面识别问题、执行器问题、模型能力问题、前置数据问题
- 将失败归并为失败簇
- 提出下一轮修正策略

失败分析代理不直接修正平台级基线，只能生成修正建议和沉淀候选项。

### 5.5 人机交互代理（Human-in-the-Loop Agent）

负责：

- 暂停并提示人工介入
- 支持人工审核页面入口清单和功能点清单
- 支持人工登录接管
- 支持人工确认需人工判断的结论
- 支持人工审批平台级晋升候选项

### 5.6 技能沉淀代理（Knowledge Curator Agent）

负责：

- 提取成功路径中的可复用技能
- 生成项目级脚本、规则、定位器策略和功能点模板
- 汇总平台级晋升候选项
- 生成“建议提升到平台级”的待审材料

平台级基线晋升必须经过人工审批，不由该代理直接写入共享基线。

### 5.7 报告整理代理（Report Writer Agent）

负责：

- 汇总页面入口清单、功能点清单、执行结果、截图索引和失败原因
- 生成 Markdown/HTML 报告
- 生成运行日报

## 6. 两段式执行流程

### 6.1 阶段一：发现阶段

发现阶段流程：

1. 完成自动登录或人工登录接管
2. 从首页开始遍历页面入口
3. 为每个页面入口建立页面地图节点
4. 在每个页面入口上执行有限深度探索
5. 识别功能点并增量写入功能点清单
6. 允许人工审核开关介入

输出：

- `page_entries.json`
- `feature_points.json`
- 页面入口截图

### 6.2 阶段二：验证阶段

验证阶段流程：

1. 按功能点逐个调度
2. 每个功能点验证前回到所属页面入口标准起点
3. 按功能点类型模板执行最小验证路径
4. 记录执行型测试用例、过程日志和截图
5. 写入探索式状态分类结果

验证阶段默认执行策略：

- 默认使用 Playwright 确定性执行
- 不允许 Browser Use 自由端到端执行功能点验证
- 如果 Playwright 定位或页面理解失败，可将该功能点标记为视觉兜底候选
- 视觉兜底候选需经规则命中或人工确认后，才允许 Browser Use 辅助验证

输出：

- `executions.json`
- 截图索引
- 功能点执行日志

### 6.3 验证层共享动作族分层

当前原型中的验证层，已经不再只依赖单一项目脚本，而是明确分成三层：

1. **通用模板动作**
   - 例如导航、基础输入、基础点击、基础断言
   - 由通用 verification 执行器直接复用
2. **项目级复用动作族**
   - 从复杂真实流程中抽出的稳定动作簇
   - 当前 `suyuan_online_apply` 已形成两组样本：
     - wizard / drawer 动作族
     - upload / submit dialog 动作族
3. **少量项目级胶水**
   - 仍保留项目内部 API 注入、特殊字段映射或高风险提交策略桥接
   - 当前典型残留点是 `fill_success_template` 与最终提交前的 policy gate 组合

这样分层的目的，是让第二阶段先把“最稳定、最可复用”的动作沉到 verification 层，再逐步缩小项目专用脚本，而不是一次性把所有项目逻辑强行抽成伪通用层。

当前节奏约束：

- 先补共享动作与模板 contract 的回归护栏
- 再继续拆剩余项目级耦合点，例如 `fill_success_template`
- 最后再引入新的真实业务系统做跨系统验证

## 7. 循环优化机制

### 7.1 循环目标

循环的目标不是无限尝试，而是：

- 修复可修复的技术问题
- 提升项目级自动化覆盖
- 提炼项目级经验
- 识别哪些经验值得晋升为平台级能力

### 7.2 循环粒度

采用三层粒度：

- 页面入口：驱动发现阶段
- 功能点：驱动验证阶段
- 失败簇：驱动优化阶段

### 7.3 单轮循环

建议单轮循环如下：

1. 页面探索代理完成一轮发现
2. 功能点分析代理完成功能点推断和分类
3. 验证阶段执行单功能点基础路径验证
4. 失败分析代理对失败项聚类成失败簇
5. 主代理决定哪些修正仅作用于当前 run，哪些写入项目级沉淀
6. 下一轮探索和验证优先覆盖：
   - 上轮失败簇对应路径
   - 上轮未验证完功能点
   - 新暴露出的页面入口或功能点

### 7.4 停止条件

主代理必须遵守硬停止条件。

停止条件包括：

1. 目标完成
2. 改进无收益
3. 安全边界命中
4. 人工接管命中
5. 资源预算耗尽

#### 目标完成

- 页面入口发现完成
- 功能点验证完成
- 报告已生成

#### 改进无收益

- 连续 N 轮无新增页面入口
- 连续 N 轮无新增功能点
- 连续 N 轮相同失败簇修正后无改进

#### 安全边界

- 命中高风险操作边界
- 命中异常循环
- 命中重复重试上限
- 页面状态污染严重

#### 人工接管

- 进入人工审核开关
- 进入人工登录接管
- 命中人工确认结论
- 命中平台级沉淀审核

当前原型进一步将“人工接管”细分为两类程序语义：

- `needs_review`：本轮不再盲目自动续跑，而是先生成人工接管恢复包，等待人工补齐后恢复
- `scheduled`：仅当 `should_start_next_round=true` 时，主流程才自动进入下一轮

#### 资源预算

- 达到最大运行时长
- 达到最大模型调用预算
- 达到最大截图/日志/下载产物上限

## 8. 沉淀分层策略

第二阶段必须采用分层沉淀策略。

### 8.1 运行时临时修正

仅作用于当前 run，例如：

- 当前任务专用 prompt patch
- 当前页面临时修复策略
- 当前会话的重试参数

不进入项目级或平台级长期存储。

### 8.2 项目级沉淀

允许自动写入当前项目目录，例如：

- 页面入口规则
- 页面特定定位器策略
- 功能点模板
- 回放脚本
- 失败修复笔记

### 8.3 平台级基线沉淀

必须人工审核后才能晋升。

晋升对象包括：

- 通用提示词策略
- 通用定位器修复规则
- 通用失败修复策略
- 通用技能模板
- 可跨项目复用的自动执行脚本模板

## 9. 人工介入机制

### 9.1 触发场景

以下场景允许人工介入：

- 登录无法自动完成
- 页面入口或功能点识别结果需要人工审核
- 业务正确性无法自动判断
- 模型长时间无法收敛
- 命中高风险路径
- 平台级知识晋升审批

### 9.2 人工介入入口

原型阶段可接受的介入入口形式包括：

- CLI 暂停并等待输入
- 本地轻量 Web 页面
- 结构化清单审核页
- 手工确认和继续执行按钮

原型阶段不要求完整多用户工作流系统，但必须提供可实际使用的人机交互入口。

### 9.3 人工接管恢复机制

人工介入不能只停留在“暂停一下”。第二阶段原型应把人工接管固化为结构化恢复流程：

1. 当主流程命中人工接管或人工审核条件时，写出 `human_takeover.json`
2. 恢复包中记录：
   - 来源 run
   - 当前轮次
   - 下一轮目标阶段
   - 待处理失败簇和待处理动作
   - 已计算出的执行提示
   - 标准恢复命令
3. 人工完成补齐后，通过统一入口恢复续跑
4. 恢复后的下一轮继续沿用同一套 `next_round_decision`、失败簇、报告和日报链路

这样可以避免人工接管后只剩零散终端日志，导致主流程丢失状态。

## 10. 受控 worker 设计

原型阶段只为高副作用、高成本或强隔离需求任务引入受控 worker。

建议第一批 worker 包括：

### 10.1 浏览器执行 worker

负责：

- Playwright 浏览器生命周期管理
- 导航、点击、输入、等待
- 页面状态恢复

默认情况下，浏览器执行 worker 是验证阶段的唯一低层动作执行入口。

### 10.2 证据采集 worker

负责：

- 截图
- 下载监听
- 页面快照
- 执行日志归档

### 10.3 视觉兜底 worker

负责：

- 在 Playwright 无法稳定定位或理解页面时，调用 Browser Use 做受控辅助
- 输出可被 Playwright 回放或复核的候选动作路径
- 标记视觉兜底是否成功，以及是否需要人工确认

视觉兜底 worker 不应直接写入平台级技能基线，只能生成项目级证据和晋升候选。

### 10.4 报告渲染 worker

负责：

- Markdown/HTML 报告生成
- 日报渲染
- 大体积产物整理

这些 worker 在原型阶段不要求彼此独立部署，可先实现为受主代理调度的本地任务执行单元。

### 11.6 progress_events.jsonl / current_status.json / phase_summary.json

这组三个文件用于运行态流程可视化：

- `progress_events.jsonl`：按时间顺序记录阶段开始、阶段完成、步骤开始、步骤完成、等待人工、命中停止条件等事件
- `current_status.json`：对外提供当前阶段、当前步骤、当前轮次、当前对象、统计信息和阻塞原因的最新快照
- `phase_summary.json`：按阶段汇总完成情况，便于后续前端看板消费

它们是运行时监控层，不替代最终报告。

### 11.6.1 routing_summary.json / discovery_strategy.json

这两个文件用于把“为什么这样跑”也显式落盘，而不只记录“跑成了什么样”：

- `routing_summary.json`：面向操作者和后续前端，汇总当前模型的预检状态、discovery / verification / reporting 路由模式、run policy 装载状态、能力标签与说明备注
- `discovery_strategy.json`：记录本轮 discovery 最终采用的策略，例如 `blocked`、`template_seed_only`、`live_enrich`、`skip_completed_discovery`

这两个产物的作用是：

- 让 CLI 和运行态视图能直接展示当前模型为什么被路由到某条路径
- 让 stop / next-round / human takeover 的判断可以回溯到当时的 discovery 策略
- 避免把“Browser Use readiness”与“当前 discovery 主执行器”混为一谈

## 14. 主要漏洞与修正建议

### 14.1 漏洞：模型虽然可用，但结构化输出、视觉能力和工具调用能力不稳定

修正建议：

- 建立 LLM 预检
- 建立模型能力标签
- 将模型能力和任务类型做显式映射

### 14.1.1 已验证的兼容性分歧

对本地 `AI-tester`、本地 Qwen、火山引擎 `deepseek-v4-flash-260425`、DeepSeek 原厂 `deepseek-v4-flash` 的实际探测显示：

- 本地 `AI-tester` 当前连通性不稳定，基础 chat、`json_object`、`json_schema`、tool calling 均超时
- 本地 `Qwen3.6-35B-A3B-UD-Q5_K_M-MTP` 当前连通性不稳定，基础 chat、`json_object`、`json_schema`、forced tool calling、auto tool calling、Browser Use 封装路径均超时或连接失败
- 火山引擎 `deepseek-v4-flash-260425` 普通 chat 可用，但 `json_object`、`json_schema` 均不可用，tool calling 结果不稳定
- DeepSeek 原厂 `deepseek-v4-flash` 普通 chat、`json_object`、自动 tool calling 可用，但 `json_schema` 不可用，强制 `tool_choice` 不可用
- Browser Use 的 `ChatOpenAI` 结构化输出路径依赖 `response_format.type=json_schema`，对上述 DeepSeek 配置均不可用
- Browser Use 的 `ChatDeepSeek` 兼容路径可能绕开 `json_schema`，但是否稳定取决于是否强制 `tool_choice` 和具体模型模式

这说明第二阶段不能把“DeepSeek 可聊天”直接等同于“DeepSeek 可支持 Browser Use 的结构化输出路径”。

### 14.2 漏洞：Browser Use 自由度过高，容易偏离模板

修正建议：

- 缩小 Browser Use 的职责
- 对可用工具实行白名单
- 关键动作由 Playwright 执行器承接
- 验证阶段默认 Playwright only
- 视觉兜底必须显式标记，不作为默认路径

### 14.3 漏洞：项目级沉淀和平台级晋升混淆，会污染基线

修正建议：

- 采用沉淀分层
- 平台级晋升必须人工审核
- 运行日报显式列出候选项

### 14.4 漏洞：循环可能形成长时间无收益重试

修正建议：

- 用失败簇驱动循环
- 用停止条件硬切断
- 用改进无收益阈值判停

### 14.5 漏洞：人工介入没有统一入口，会破坏闭环

修正建议：

- 在原型阶段就设计标准化人工介入入口
- 所有人工介入都要有结构化记录
- 人工接管后必须能恢复续跑，而不是重新从头手动拼接上下文

### 14.6 漏洞：执行器状态污染会导致验证结果不稳定

修正建议：

- 每个功能点验证前回到标准起点
- 对关键页面建立状态恢复逻辑

### 14.7 漏洞：日志、截图、下载、回放脚本分散，后期平台难集成

修正建议：

- 先定义统一 artifacts 目录
- 结构化产物优先于展示层优化

### 14.8 漏洞：原型阶段过早引入完整多进程架构，会放大基础设施复杂度

修正建议：

- 原型阶段采用单主进程 + 多逻辑代理
- 只为高副作用任务引入受控 worker
- 等执行能力稳定后再评估是否拆分独立进程或服务

### 15.4 泛化闭环当前专项顺序（2026-06-21）

这是当前围绕“泛化闭环”的专门执行计划，不替代 P0-P6 或 W1-W7，而是作为验证模板泛化过程中的落地顺序：

1. **G1：把“定义了但没接上”的抽象层接通**（已完成）
   - 已接通共享 locator hints 解析链
   - 已接通共享规则评估器与共享验证结果路径
   - 已把录制草稿 DSL 对齐到可执行模板动作
2. **G2：把复杂真实流程从“项目专用流”抽成“复用动作族”**（已完成）
   - `suyuan_online_apply` 已抽出 wizard / drawer 与 upload / submit dialog 两组动作族
3. **G3：补泛化回归测试护栏**（已完成）
   - 已护住模板动作名到 registry 的 contract、共享动作 handler 输入输出、policy bridge、复杂 flow 的失败传播
   - 已补 `fill_success_template` 的输入契约边界，避免非 mapping runtime 数据悄然透传
4. **G4：引入新的真实业务系统做跨系统验证**（已启动）
   - 当前已先落地 validation matrix 骨架，用统一汇总链路承接 `lab_*` 本地模板族与 `suyuan_*` 样本
   - 默认矩阵目标已补入同一真实系统内的第三个模板样本 `suyuan_online_detail_view`
   - 该样本当前含义是“已接线、已纳入矩阵和回归护栏”，不是“已拿到新的 connected live 实证”
   - 当前目标是验证“统一执行/验证/汇总骨架”成立，不应误写成“已完成第二真实业务系统验证”
   - 下一步是在该骨架上接入新的真实业务系统，验证当前复用动作族和护栏不是只对单一溯源样本成立

当前判断：

- `fill_success_template` 是 G2 之后剩余的项目级耦合点
- 它属于“继续泛化”的下一批候选，但当前不高于 G4
- 也就是说，当前正确节奏应是“继续拆剩余耦合，并把 G4 从样本矩阵推进到新真实业务系统接入”
