# AI Agent 软件自动化评测平台需求分析书（第二阶段 v4）

更新时间：2026-07-02

## 1. 文档目的

本文档是在第二阶段 v3 需求基础上的进一步收敛稿，目标是把第二阶段从“按轮次推进的自动评测”进一步收敛为“围绕明确目标的自进化循环”。

本文档只定义需求边界、目标体系、分阶段任务和验收标准，不展开技术实现。

## 2. 需求背景

第二阶段当前已经具备一定的发现、验证、复盘和报告能力，但实际开发过程中，最容易出现两类偏差：

1. 测试大模型和运行时流程发生耦合，导致系统围绕单一界面持续补丁化。
2. 编程大模型介入过深，把本应由运行时自进化解决的问题，提前变成代码修补问题。

这会带来两个后果：

- 人工介入变深，闭环成本上升。
- 程序过度适配某个具体界面，泛化能力下降。

因此，第二阶段需要重新明确一个更稳定的主线：

**围绕单个菜单、页面或功能点建立目标循环，让系统自己在失败中收敛，在成功中沉淀。**

## 3. 总体目标

第二阶段的总体目标仍然不变：

1. 自动发现当前测试账号权限范围内的 Web UI 导航结构。
2. 基于页面入口、页面元素和操作路径推断可测功能点。
3. 按固定模板对单个功能点执行基础路径验证。
4. 输出可复核的结构化测试证据和可阅读的测试报告。

在此基础上，目标循环版要求进一步做到：

1. 以单个菜单、页面或功能点为最小目标单元。
2. 每个目标拥有明确的成功条件和停止条件。
3. 每个目标的失败都必须被分类。
4. 每个失败分类都要对应固定套路动作。
5. 每次循环都要沉淀成功经验和失败修正。
6. 直到目标达成或达到最大轮数停止。

## 4. 设计原则

### 4.1 目标优先

系统的主要关注点不是“整轮任务是否跑完”，而是“当前目标是否达成”。

目标可以是以下三类之一：

- 菜单目标：例如发现某个一级或二级菜单。
- 页面目标：例如进入某个页面并确认可达。
- 功能点目标：例如验证某个查询、详情、导出、Tab 或表单交互。

### 4.2 证据优先

任何成功或失败都必须可回溯到证据。

证据至少应覆盖：

- 目标对象
- 执行动作
- 页面反馈
- 截图
- 失败原因或成功判定

### 4.3 自动化优先

常规目标循环应尽量由系统自动完成，不把人工作为默认步骤。

人工只处理以下情况：

- 目标被安全策略阻断
- 目标缺少前置数据
- 目标需要人工确认
- 目标连续失败且系统无法继续收敛

### 4.4 自进化优先

系统必须在运行时持续学习：

- 哪些失败分类高频出现。
- 哪些套路动作有效。
- 哪些定位器或页面策略更稳定。
- 哪些经验只适合当前项目，哪些值得继续沉淀。

### 4.5 双模型分工

第二阶段明确区分两类模型：

- 测试大模型：负责 Browser Use 驱动、目标理解、失败归因、成功经验沉淀、下一轮计划建议。
- 编程大模型：负责程序缺陷分析、架构重构、通用能力提升和代码修改。

测试大模型不应直接承担程序修改职责。  
编程大模型不应介入每一次普通目标失败的日常循环。

### 4.6 优先深化能力

从第一性原理出发，最值得做深做强的能力，不是继续堆模板，而是把“发现 - 识别 - 验证 - 复盘 - 继续”的闭环做厚。

系统应优先深化以下能力：

1. 页面入口发现能力。
   - 能稳定发现菜单树、层级路径和页面入口。
   - 能去重、归一化和恢复折叠菜单壳。
   - 能区分“可见但未覆盖”和“真实不可达”。

2. 功能点识别能力。
   - 能从页面中识别真实功能点，而不是只记录页面可见性。
   - 能识别查询、重置、详情、导出、Tab、弹窗、行级动作、表单交互等高价值类型。
   - 能把功能点类型和风险等级稳定落盘。

3. 低风险执行能力。
   - 能把功能点生成可执行的基础路径用例。
   - 能优先覆盖查询、重置、详情、Tab、导出入口确认等低风险高价值路径。
   - 能在必要时保留新增、编辑、提交等高风险入口，但默认不做最终副作用执行。

4. 证据绑定能力。
   - 能把目标、动作、页面反馈、截图、网络事件和断言绑定到同一条证据链。
   - 能让每个失败或成功都可以被人工复核。
   - 能避免把 run 级截图误当成 case 级证据。

5. 目标接管能力。
   - 能在 Playwright 不足时，通过 Browser Use 做语义恢复和目标寻找。
   - 能把 Browser Use 的接管结果转化为可复核的结构化证据。
   - 能防止 Browser Use 变成无约束的自由执行器。

6. 跨系统验证能力。
   - 能把同一套目标循环复用到新的真实业务系统。
   - 能检验失败分类、套路动作和经验沉淀是否真的具有泛化性。
   - 能避免把某个具体界面的修补误当成平台能力。

## 5. 目标循环

### 5.1 目标定义

每个目标至少包含以下信息：

- 目标名称
- 目标类型
- 成功条件
- 最大轮数
- 当前状态
- 证据要求
- 是否允许人工介入

同一个 run 可以按顺序推进多个目标，但在任一时刻只允许存在一个 active goal。  
这样可以避免多目标并行导致的状态混写、证据串线和停止条件冲突。

### 5.2 成功条件

不同目标类型的成功条件不同，但都必须可验证。

示例：

- 菜单目标：菜单文本、层级路径和截图证据存在。
- 页面目标：页面可进入、状态可达、非白屏、主内容区可见。
- 功能点目标：功能点可识别、用例可生成、基础路径可执行并产生反馈。

### 5.3 停止条件

目标循环只在以下条件之一满足时停止：

1. 目标达成。
2. 达到最大轮数。
3. 被安全策略阻断。
4. 需要人工确认。
5. 多次重复失败且无进展。
6. 执行器不可用。

### 5.4 状态要求

每个目标至少应处于以下状态之一：

- planned
- running
- succeeded
- failed_max_rounds
- waiting_human
- blocked_by_policy
- blocked_by_executor
- stopped_no_progress

## 6. 失败分类与套路动作

### 6.1 失败分类要求

为了保证结果稳定，失败原因必须提前分类，不允许每次自由写长段分析后再临时决策。

建议的最小失败分类包括：

- menu_not_found
- menu_expand_failed
- menu_click_failed
- page_blank
- page_load_timeout
- permission_blocked
- login_required
- target_discovered_but_uncovered
- feature_not_identified
- locator_unstable
- action_not_observed
- assertion_failed
- missing_prerequisite_data
- blocked_by_safety_policy
- browser_use_unavailable
- evidence_incomplete
- no_progress_repeated
- unknown

### 6.2 套路动作要求

每个失败分类都必须绑定至少一个套路动作。

套路动作的要求是：

- 固定
- 可复用
- 可回放
- 可审计

例如：

- `menu_not_found` 对应扩大菜单候选、归一化菜单壳、启用语义查找。
- `page_blank` 对应重新进入、等待稳定、回到起点重试。
- `feature_not_identified` 对应轻量交互暴露更多功能点、补充页面语义分析。
- `evidence_incomplete` 对应强制补齐动作日志和步骤级截图。

### 6.3 经验沉淀要求

每次目标尝试后，系统必须沉淀两类结果：

1. 成功经验：哪些动作、定位、等待和反馈判断有效。
2. 失败修正：哪些错误分类、套路动作和重试条件有效。

沉淀结果必须区分三层：

- 当前 run 临时经验
- 项目级经验
- 平台级候选

平台级候选必须经过人工审核。

## 7. 分阶段任务

### 7.1 阶段A：菜单目标

目标：让系统稳定发现菜单入口树。

要求：

- 发现首页可见的一级菜单。
- 逐级发现二级、三级菜单。
- 记录菜单路径、截图和失败原因。
- 对“已发现但未覆盖目标”继续追踪，而不是误判完成。

### 7.2 阶段B：页面目标

目标：让系统稳定进入页面入口并确认页面状态。

要求：

- 识别页面标题、URL 和主内容区。
- 判断页面是否可达、白屏或受限。
- 对页面入口做去重和稳定化。
- 页面目标达成后，继续向功能点目标推进。

### 7.3 阶段C：功能点目标

目标：让系统从页面中识别真实功能点，并生成可执行用例。

要求：

- 识别查询、重置、详情、导出、Tab、弹窗、行级动作等功能点。
- 基于功能点类型生成执行型测试用例。
- 用例必须包含前置条件、步骤、反馈、断言和证据要求。
- 低风险用例优先自动执行。

### 7.4 阶段D：证据与报告目标

目标：让每个目标都能被复核。

要求：

- 记录目标级、用例级、步骤级证据。
- 输出结构化结果。
- 输出可阅读报告。
- 报告中必须说明未执行原因、失败归因和人工确认项。

### 7.5 阶段E：跨系统验证目标

目标：验证目标循环不是只适配单一界面。

要求：

- 接入新的真实业务系统。
- 复用同一套目标循环。
- 验证失败分类和套路动作在跨系统场景下是否仍有效。

### 7.6 阶段F：平台能力深化目标

目标：把稳定经验从项目级提升为平台级能力。

要求：

- 从多次 run 中识别通用规律。
- 汇总高频失败分类。
- 审核平台级候选。
- 决定哪些能力应该进入编程大模型驱动的代码修改流程。

## 8. 结构化产物

每个目标循环 run 至少应产出以下结构化信息。这里分为两类：目标循环核心产物，以及沿用的 run 级产物。

### 8.1 目标循环核心产物

| 文件 | 中文解释 | 作用 |
| --- | --- | --- |
| `goal_registry.json` | 目标注册表，记录本次 run 中有哪些目标、目标类型、成功条件、最大轮数和状态。 | 让系统知道“本轮要达成什么”。 |
| `goal_state.json` | 当前目标状态快照，记录目标正在进行到哪一步、剩余轮数、阻塞原因和下一步建议。 | 让人和模型都能快速看到“现在卡在哪里”。 |
| `goal_attempts.jsonl` | 目标尝试流水，逐条记录每次尝试的时间、动作、反馈和结果。 | 形成可追溯的目标时间线。 |
| `failure_classifications.jsonl` | 失败分类流水，记录每次失败被归入哪一类、依据是什么、置信度如何。 | 让失败归因稳定、可复用。 |
| `playbook_actions.jsonl` | 套路动作流水，记录针对某一失败分类实际采取了哪些固定动作。 | 让下一轮尝试可回放、可审计。 |
| `experience_updates.jsonl` | 经验更新流水，记录这次循环沉淀了什么成功经验、失败修正和候选规则。 | 让自进化结果落盘。 |
| `goal_summary.json` | 目标摘要，汇总当前目标是否达成、尝试次数、主要失败原因和经验结论。 | 供运行中心、报告和下一轮计划消费。 |

### 8.2 沿用的 run 级产物

| 文件 | 中文解释 | 作用 |
| --- | --- | --- |
| `menu_tree.json` | 菜单树结构，表示菜单之间的父子层级和遍历结果。 | 作为菜单目标与页面目标的入口依据。 |
| `menu_entries.json` | 菜单入口明细，记录每个菜单项的名称、层级、路径、状态和截图证据。 | 供菜单目标复核和去重。 |
| `page_entries.json` | 页面入口清单，记录每个页面入口的名称、URL、来源、可达性和截图证据。 | 供页面目标和功能点目标消费。 |
| `feature_points.json` | 功能点清单，记录每个功能点的名称、类型、风险等级、置信度和审核状态。 | 供用例生成和目标循环继续推进。 |
| `generated_test_cases.json` | 执行型测试用例清单，记录每个功能点对应的基础路径步骤、前置条件和断言。 | 供测试大模型和执行器执行。 |
| `execution_results.json` | 执行结果清单，记录每条用例的状态、页面反馈、截图引用和失败原因。 | 作为目标成败判断的核心证据。 |
| `screenshots_index.json` | 截图索引，记录每张截图的编号、阶段、来源和文件位置。 | 让证据可以快速定位。 |
| `human_tasks.json` | 人工任务清单，记录哪些目标需要人工确认、接管或审阅。 | 供人工介入与恢复使用。 |
| `round_analysis.json` | 本轮复盘结果，汇总覆盖情况、失败簇、证据质量和下一轮建议。 | 供目标继续推进和平台沉淀判断。 |
| `next_round_plan.json` | 下一轮计划，说明下一次应继续哪个目标、采用什么修正动作、是否需要人工批准。 | 供目标循环续跑。 |
| `run_report.md` | 可阅读的运行报告，用自然语言描述本轮发生了什么。 | 供人工阅读和交付。 |
| `run_report.json` | 结构化运行报告，供平台聚合、统计和前端展示。 | 供运行中心和后续模型消费。 |

这些产物的作用是让系统、人工和后续模型都能理解：

- 当前目标是什么。
- 已尝试几次。
- 为什么失败。
- 下一次准备怎么试。
- 这次循环留下了什么经验。

## 9. 验收标准

如果满足以下条件，可认为目标循环版需求成立：

1. 系统可以围绕单个菜单、页面或功能点建立目标。
2. 系统可以对失败原因进行稳定分类。
3. 系统可以根据失败分类选择套路动作并再次尝试。
4. 系统可以在目标达成或达到最大轮数时停止。
5. 系统可以沉淀成功经验和失败修正。
6. 系统可以把经验分为当前 run、项目级和平台级候选。
7. 系统不需要人工频繁改代码，也能完成常规目标循环。
8. 系统不把某一个页面的特殊适配误当成平台整体能力。
9. 系统能把菜单目标、页面目标和功能点目标串成连续的目标循环，而不是孤立地展示一组产物。
10. 系统能稳定产出步骤级证据，而不是只在 run 级别留下粗粒度截图。
11. 系统能在低风险功能点上形成可复用的基础路径验证能力。
12. 系统能把跨系统验证结果用于判断哪些经验值得晋升为平台级。

## 10. 暂不实现

本需求书暂不要求：

- 自动修改平台代码。
- 自动对复杂业务正确性下结论。
- 无限轮次重试。
- 全量页面穷举。
- 多角色权限矩阵自动覆盖。
- 把模板文件作为普通用户主入口。
- 把某个单页面的特例规则直接晋升为平台默认能力。

## 11. 说明

本需求书与现有第二阶段 v3 需求并不冲突，而是进一步把执行单元收敛到“目标循环”。

后续技术方案应围绕本需求书展开，但不在本轮文档中展开。
