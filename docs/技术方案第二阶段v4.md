# AI Agent 软件自动化评测平台技术方案（第二阶段 v4）

更新时间：2026-07-02

## 1. 方案目标

第二阶段 v4 的技术目标，是把“围绕明确目标的自进化循环”真正落成可运行、可验证、可回放的执行系统。

这里的核心变化是：

- 不再把整轮 run 当作唯一优化单元。
- 不再把某个样本页面的成功路径当作平台默认能力。
- 不再让测试大模型和编程大模型混在同一个循环里直接互相替代。

v4 的真实目标是：

**围绕单个菜单、页面或功能点建立目标循环，让系统在每次尝试中自动归因、自动选择套路动作、自动沉淀经验，并在目标达成或达到最大轮数时停止。**

## 2. 设计约束

### 2.1 以目标为主实体

技术方案必须以目标循环为主线，而不是以模板文件、单脚本或单个页面样本为主线。

### 2.2 以证据为接口

任何判断都必须能回到证据：

- 菜单路径
- 页面入口
- 功能点
- 执行动作
- 页面反馈
- 截图
- 网络事件
- 状态分类

### 2.3 以分层沉淀为边界

经验沉淀必须分层：

- 当前 run 临时修正
- 项目级沉淀
- 平台级候选

平台级候选不能自动生效。

### 2.4 以安全边界为硬约束

默认禁止高风险真实提交。  
只有经过策略授权的动作，才允许进入执行链路。

### 2.5 以可验证为第一标准

每个阶段都必须能被运行、被观察、被验证。  
任何不能被验证的能力，都不能进入主线。

### 2.6 以兼容现有运行中心为过渡约束

v4 的目标循环是新的主逻辑，但不要求一次性推倒现有 v3 运行中心和既有产物链路。

技术实现上，目标级核心状态可以先与现有 run 级产物并行存在：

- `progress_events.jsonl`
- `current_status.json`
- `phase_summary.json`
- `round_analysis.json`
- `next_round_plan.json`
- `human_tasks.json`
- `human_takeover.json`
- `human_takeover_resolution.json`
- `promotion_candidates.json`
- `candidate_template_review.json`

其中，目标级产物负责表达“当前目标怎么推进”，现有 run 级产物负责继续支撑运行中心、报告和人工接管消费。  
也就是说，v4 不是先造一套和现有平台隔离的新世界，而是在兼容现有证据层的前提下，把主循环升级为目标循环。

## 3. 总体架构

v4 采用“运行中心 + Python 执行子系统 + 目标循环内核 + 证据与复盘层 + 人工接管层”的结构。

```text
运行中心
  -> run 创建 / 配置 / 启停 / 复盘 / 报告 / 人工任务

Python 执行子系统
  -> preflight
  -> 目标循环编排
  -> Browser-use + Playwright 执行
  -> 结构化产物落盘

目标循环内核
  -> 目标注册
  -> 目标状态
  -> 尝试记录
  -> 失败分类
  -> 套路动作
  -> 经验更新

证据与复盘层
  -> 菜单 / 页面 / 功能点 / 用例 / 执行结果 / 失败簇 / 下一轮计划 / 报告

人工接管层
  -> 审核目标
  -> 处理阻塞
  -> 批准下一轮
  -> 审阅候选沉淀
```

## 4. 核心模块

### 4.1 运行中心

运行中心是普通用户的唯一主入口，负责：

- 创建和选择 run。
- 配置系统名称、入口 URL、CDP 地址、目标范围和安全策略。
- 查看目标循环状态、复盘结果和人工任务。
- 启动、暂停、恢复、终止和进入下一轮。
- 查看 artifact、截图和报告。

运行中心不应把模板文件作为普通用户主路径。

### 4.2 Preflight 与路由模块

该模块负责：

- 校验浏览器连接、执行环境和模型能力。
- 判断当前 run 是否可进入真实浏览器链路。
- 记录模型能力标签和路由结果。
- 决定本轮是否允许 Browser Use 参与，以及参与哪类任务。

它的接口重点不是“是否能回答一次”，而是“是否满足目标循环所需的能力边界”。

### 4.3 目标循环编排模块

这是 v4 的核心模块，负责管理一个目标的完整生命周期。

职责包括：

- 注册目标。
- 初始化目标状态。
- 推进每次尝试。
- 在失败后执行分类。
- 根据分类触发套路动作。
- 更新经验。
- 判断是否停止。

同一个 run 的目标编排应采用单 active goal 机制：允许目标队列存在，但同一时间只推进一个目标。  
这可以保证目标状态、证据和停止条件始终围绕一个清晰对象收敛。

目标循环的最小闭环是：

```text
目标 -> 尝试 -> 失败分类 -> 套路动作 -> 再尝试 -> 成功沉淀 -> 停止
```

### 4.4 菜单入口发现模块

该模块负责菜单目标的达成。

需要具备的能力：

- 发现一级、二级、三级菜单。
- 归一化折叠菜单壳。
- 过滤非业务 chrome。
- 处理可见但未覆盖目标。
- 输出可复核的菜单路径和截图证据。

### 4.5 页面入口探索模块

该模块负责页面目标的达成。

需要具备的能力：

- 进入菜单叶子或指定页面入口。
- 采集标题、URL、主内容区和白屏状态。
- 去重页面入口别名。
- 标记页面可达性和阻塞原因。

### 4.6 功能点识别模块

该模块负责从页面中提炼出真实功能点。

最重要的不是“识别更多 DOM”，而是“识别更有价值的业务动作”。

重点识别：

- 查询 / 筛选
- 重置
- 详情
- 导出 / 下载
- Tab / 切换
- 弹窗动作
- 行级动作
- 表单交互

输出必须包含：

- 功能点名称
- 功能点类型
- 风险等级
- 可自动验证性
- 置信度
- 审核状态

### 4.7 执行型测试用例生成模块

该模块把功能点转换成执行型测试用例。

用例必须表达的是基础路径，不是需求验收全集。

低风险优先覆盖：

- 导航进入
- 查询 / 筛选
- 重置
- 详情查看
- Tab 切换
- 导出入口确认

高风险动作默认只验证入口和前置条件，不默认执行最终提交。

### 4.8 执行模块

执行模块负责把测试用例落实为真实浏览器动作。

要求：

- 每个用例从标准起点开始。
- 执行前、关键反馈后、异常时都要采集证据。
- 动作必须可审计。
- 当 Playwright 不足时，可按策略启用 Browser Use 语义接管。

### 4.9 证据绑定模块

证据绑定模块负责把动作、截图、网络事件、反馈和结论绑定成同一条可追溯链路。

这是 v4 的关键深模块之一。

它的职责不是“收集更多文件”，而是：

- 把 case 级证据与 run 级证据隔离。
- 把 step 级证据固定下来。
- 把证据缺口显式暴露为 `evidence_incomplete`。

### 4.10 复盘与下一轮计划模块

该模块负责：

- 汇总覆盖、失败和跳过情况。
- 生成失败簇。
- 产出下一轮计划。
- 判断是否自动继续或等待人工确认。

这里的关键不是“生成一段总结”，而是“输出下一次可以直接执行的动作建议”。

### 4.11 人工任务模块

该模块承载所有必须人工处理的场景：

- 登录接管
- 高风险动作授权
- 发现结果审核
- 功能点审核
- 目标豁免
- 下一轮计划批准
- 候选沉淀审阅
- 人工接管包生成与恢复

人工接管不是异常旁路，而是目标循环中的标准状态之一。  
当系统被安全策略、权限、前置数据或执行器问题阻断时，应输出结构化接管包，并在人工处理后生成恢复结果，再继续下一轮。

### 4.12 报告模块

报告模块负责把目标循环转译成可阅读、可交付的文档。

报告必须按目标、页面、功能点和执行结果组织，而不是按日志堆叠。

## 5. 目标循环的数据模型

v4 的运行数据围绕以下几类结构展开：

### 5.1 目标注册

记录目标名称、类型、成功条件、最大轮数和状态。

### 5.2 目标尝试

记录每次尝试的动作、页面反馈、证据和结果。

### 5.3 失败分类

记录失败的结构化原因，以及它对应的后续策略。

### 5.4 套路动作

记录针对某一失败分类实际执行的固定动作序列。

### 5.5 经验更新

记录成功经验、失败修正和沉淀候选。

### 5.6 目标摘要

记录某个目标是否达成、尝试了几次、主要障碍是什么、下一步是否继续。

目标状态需要能区分“目标已被发现但尚未覆盖”和“目标真实不存在或不可达”。
前者属于继续推进的对象，不能误判为成功；后者才允许进入停止或跳过分支。

目标摘要还必须明确记录该目标是否已经成为当前 active goal 的结论对象，以及是否已经被后续目标接替。  
这样运行中心和复盘层才能知道“这是当前卡点”还是“这是已完成的历史目标”。

## 6. 关键接口约定

### 6.1 目标接口

目标至少要能表达：

- `goal_name`
- `goal_type`
- `success_criteria`
- `max_rounds`
- `status`
- `evidence_requirements`
- `allow_human_intervention`

### 6.2 失败分类接口

失败分类至少要能表达：

- `failure_reason`
- `reason_confidence`
- `evidence_refs`
- `suggested_playbook`
- `scope`

其中 `unknown` 只能作为临时兜底分类，不能作为长期停留的最终结论。  
如果连续出现 `unknown`，必须回到人工审阅或代码改进流程，补齐新的稳定分类。

### 6.3 套路动作接口

套路动作至少要能表达：

- `playbook_id`
- `trigger_reason`
- `action_steps`
- `expected_effect`
- `safety_constraints`
- `output_evidence`

### 6.4 经验更新接口

经验更新至少要能表达：

- `source_goal`
- `winning_pattern`
- `failed_pattern`
- `promotion_level`
- `confidence`
- `review_status`

### 6.5 下一轮计划接口

下一轮计划至少要能表达：

- `decision`
- `next_goal`
- `target_ids`
- `planned_playbooks`
- `risk_level`
- `requires_human_approval`

## 7. 模型分工与边界

### 7.1 测试大模型

测试大模型负责：

- 目标理解
- 页面语义理解
- 功能点归类
- 失败分类建议
- 套路动作选择建议
- 下一轮计划建议
- 成功经验沉淀
- 复盘摘要

测试大模型不直接修改代码。

### 7.2 编程大模型

编程大模型负责：

- 修复真实程序缺陷
- 调整模块接口
- 提升通用能力
- 处理重复出现的系统性缺口
- 把稳定经验上升为平台能力

编程大模型不应介入每一次普通目标失败的运行时循环。

### 7.3 边界原则

如果某类失败可以通过分类 + 套路动作解决，就不应优先进入代码修改。

只有当同一问题在同一目标上连续多轮无效，或跨多个目标、跨多个 run、跨多个系统反复出现，才应进入编程大模型修改流程。  
这里的“连续无效”应当以预置阈值为准，不能靠人工即时感觉拍板。

## 8. 安全与可观测性

### 8.1 安全控制

默认禁止高风险真实提交。  
安全策略必须在执行前生效，而不是执行后解释。

### 8.2 可观测性

运行过程中必须持续落盘：

- 当前阶段
- 当前目标
- 当前尝试
- 当前阻塞原因
- 最新事件
- 下一步动作

### 8.3 恢复性

中断后必须能从结构化产物恢复到目标循环上下文，不能只靠终端日志恢复。

## 9. 方案的关键结论

v4 的技术方案要点可以收束为四句话：

1. 目标循环是运行时自进化的基本单位。
2. 失败分类和套路动作是目标循环的稳定接口。
3. 证据绑定是所有判断的可信根基。
4. 编程大模型只处理系统性缺口，测试大模型负责运行时自进化。

## 10. 漏洞审查与修正循环

v4 方案必须经过至少三轮自审，审查顺序如下：

### 10.1 第一轮：目标偏差审查

检查是否又把模板、样本或单一界面当成主线。

### 10.2 第二轮：闭环完整性审查

检查是否存在“发现了但没生成用例”“生成了但没执行”“执行了但没证据”“有证据但没法复盘”的断点。

### 10.3 第三轮：泛化性审查

检查是否把某个单页面的特例动作写成了平台默认能力。

### 10.4 第四轮：安全审查

检查是否把高风险动作、业务正确性判断或平台级晋升交给了测试大模型直接决策。

### 10.5 第五轮：可验证性审查

检查每个阶段是否都能独立运行、独立验证、独立退出。

## 11. 方案成立条件

如果以下条件满足，可认为技术方案成立：

1. 目标循环有独立数据模型。
2. 菜单、页面、功能点三类目标可以串联。
3. 失败分类可以驱动套路动作。
4. 证据绑定能支撑人工复核。
5. 运行中心能查看目标级状态。
6. 测试大模型与编程大模型职责清晰。
7. 方案不会把某个项目样本误写成平台能力。
