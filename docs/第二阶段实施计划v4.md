# AI Agent 软件自动化评测平台第二阶段实施计划 v4

更新时间：2026-07-02

## 1. 计划目标

本计划用于把第二阶段 v4 技术方案拆成可运行、可验证、可回退的阶段性任务。

计划遵循两个原则：

1. 每个阶段都必须有明确输入、明确产物和明确验证方式。
2. 任何阶段完成后，都能独立运行或独立验证，不依赖后续阶段先完成。

## 2. 计划总则

### 2.1 先通主线，再做深做强

实施顺序必须优先建立以下链路：

```text
目标注册 -> 目标尝试 -> 失败分类 -> 套路动作 -> 经验更新 -> 下一轮计划
```

然后再逐步加深：

- 菜单目标
- 页面目标
- 功能点目标
- 低风险执行
- 证据绑定
- 跨系统验证

### 2.2 每阶段都可运行

每个阶段都必须能单独运行，并输出可验证产物。

### 2.3 每阶段都可回退

如果某阶段引入新问题，必须能回退到上一个稳定阶段，不允许把整个链路一起推倒。

### 2.4 每阶段都可审计

每阶段必须能通过 artifact、日志、截图或报告被复核。

### 2.5 先兼容再收敛

本计划以现有 v3 运行中心和产物链路为承载面，不要求在同一阶段内同时完成全部重构。

因此，阶段产物分两类：

- 目标循环新增产物：用于表达目标、尝试、失败分类、套路动作、经验更新
- 现有 run 级产物：用于继续支撑运行中心展示、人工接管、复盘和报告

阶段验收的标准不是“是否已经完成最终架构”，而是“是否已经形成可运行、可验证、可回退的增量闭环”。

## 3. 阶段A：目标循环骨架接通

### 3.1 阶段目标

把“目标循环”做成运行时最小闭环，使系统能注册目标、记录尝试、记录失败分类、记录套路动作和记录经验更新。

### 3.2 可运行内容

- 创建目标。
- 更新目标状态。
- 记录一次尝试。
- 为失败写入分类。
- 记录一次套路动作。
- 写入一次经验更新。
- 输出目标摘要。

### 3.3 交付物

- 目标注册与状态结构
- 目标尝试记录结构
- 失败分类记录结构
- 套路动作记录结构
- 经验更新记录结构
- 目标摘要结构
- 与现有 `current_status.json` / `phase_summary.json` 的最小兼容映射

### 3.4 可验证标准

阶段A 完成后，必须能够验证：

1. 单个目标能从 planned 进入 running。
2. 单个目标能记录至少一次失败分类。
3. 单个目标能记录至少一次套路动作。
4. 单个目标能输出可读摘要。
5. 目标摘要可以映射回运行中心可消费的状态视图。

### 3.5 阶段风险

- 风险是只做成一个记录器，而没有真正形成可推进的目标状态机。

### 3.6 阶段修正

- 如果只会记日志但不会推进状态，则阶段A 不算完成。

## 4. 阶段B：菜单目标闭环

### 4.1 阶段目标

让系统围绕菜单目标形成完整闭环，稳定发现菜单入口树，并对发现失败进行分类和修正。

### 4.2 可运行内容

- 从已登录首页启动菜单目标。
- 扫描菜单候选。
- 归一化菜单壳。
- 展开、点击、恢复。
- 记录菜单树、菜单入口、遍历日志和截图。
- 对未发现、展开失败、点击失败进行分类和套路动作。

### 4.3 交付物

- `menu_tree.json`
- `menu_entries.json`
- `menu_traversal_log.jsonl`
- `screenshots_index.json`
- `goal_summary.json`（菜单目标）
- `progress_events.jsonl` 中的菜单目标事件

### 4.4 可验证标准

阶段B 完成后，必须能够验证：

1. 首页可见一级菜单能够被稳定发现。
2. 菜单展开失败能被稳定分类。
3. 菜单壳归一化能够改善菜单扫描成功率。
4. 菜单目标可以在达到成功条件后停止。
5. 菜单目标的关键结论可以同步到 run 级摘要。

### 4.5 阶段风险

- 风险是把单一页面截图误认为完整菜单覆盖。

### 4.6 阶段修正

- 如果没有菜单树、菜单路径和遍历日志，就不能判定菜单目标完成。

## 5. 阶段C：页面目标闭环

### 5.1 阶段目标

让系统围绕页面目标稳定进入页面入口，识别页面状态，并把白屏、不可达和权限阻塞区分开。

### 5.2 可运行内容

- 基于菜单目标进入页面入口。
- 采集标题、URL、主内容区和页面状态。
- 进行页面去重和别名归一化。
- 输出页面入口清单和页面级探索日志。
- 对白屏、超时、权限不足做分类和修正。

### 5.3 交付物

- `page_entries.json`
- `page_exploration_log.jsonl`
- `screenshots_index.json`
- `goal_summary.json`（页面目标）
- `progress_events.jsonl` 中的页面目标事件

### 5.4 可验证标准

阶段C 完成后，必须能够验证：

1. 页面入口可达性可以被稳定记录。
2. 白屏与不可达可以被稳定区分。
3. 页面目标能在有足够证据时达成。
4. 页面目标未达成时，下一轮计划能继续追踪已发现但未覆盖目标。
5. 页面目标的状态可以回写到运行中心的当前状态视图。

### 5.5 阶段风险

- 风险是只记录 URL 变化，不记录页面状态。

### 5.6 阶段修正

- 如果没有主内容区与截图证据，不能把页面入口当作已完成。

## 6. 阶段D：功能点目标闭环

### 6.1 阶段目标

让系统从页面中识别真实功能点，并将功能点转换为执行型测试用例。

### 6.2 可运行内容

- 在页面上识别查询、重置、详情、导出、Tab、弹窗、行级动作等功能点。
- 对功能点类型进行规则和模型合并判定。
- 为低风险功能点生成执行型测试用例。
- 对高风险入口生成入口确认类用例。

### 6.3 交付物

- `feature_points.json`
- `generated_test_cases.json`
- `discovery_review.json`
- `goal_summary.json`（功能点目标）
- `progress_events.jsonl` 中的功能点目标事件

### 6.4 可验证标准

阶段D 完成后，必须能够验证：

1. 功能点不再只停留在页面可见性。
2. 至少能稳定识别若干低风险功能点。
3. 功能点可以转换成可执行用例。
4. 用例可以被后续阶段执行。
5. 功能点类型与风险等级可以落到结构化产物。

### 6.5 阶段风险

- 风险是功能点类型全部退化成 `view`。

### 6.6 阶段修正

- 如果无法识别出查询、详情、Tab 等类型，就说明功能点目标没有真正完成。

## 7. 阶段E：执行、证据与复盘闭环

### 7.1 阶段目标

让系统执行低风险用例，并把执行证据、失败簇和下一轮计划稳定输出。

### 7.2 可运行内容

- 执行低风险或已授权用例。
- 记录动作、页面反馈、截图和网络事件。
- 识别失败簇。
- 输出下一轮计划。
- 形成可阅读报告。

### 7.3 交付物

- `execution_results.json`
- `action_log.jsonl`
- `network_events.json`
- `screenshots_index.json`
- `round_analysis.json`
- `next_round_plan.json`
- `run_report.md`
- `run_report.json`
- `human_tasks.json` 或 `human_takeover.json`（如触发人工接管）

### 7.4 可验证标准

阶段E 完成后，必须能够验证：

1. 用例执行结果可追溯。
2. 证据可按用例和步骤定位。
3. 失败原因可以聚类成失败簇。
4. 下一轮计划可以由复盘结果驱动。
5. 报告能解释未执行原因和人工确认项。
6. 人工接管和恢复动作可以独立验证，不阻断已完成的目标结果。

### 7.5 阶段风险

- 风险是只有 run 级截图，没有 case 级证据。

### 7.6 阶段修正

- 如果无法定位到 step 级证据，就不能把结果判成稳定通过。

## 8. 阶段F：跨系统验证与平台深化

### 8.1 阶段目标

验证目标循环不是只适配单一界面，并把跨系统复用成功的经验提升为平台候选。

### 8.2 可运行内容

- 接入新的真实业务系统。
- 使用同一套目标循环推进菜单、页面和功能点。
- 比较不同系统的失败分类与套路动作。
- 审阅项目级和平台级候选。

### 8.3 交付物

- `promotion_candidates.json`
- `promotion_candidate_summary`
- `failure_classifications.jsonl`
- `experience_updates.jsonl`
- `goal_summary.json`
- `human_takeover_resolution.json`（如阶段内发生人工恢复）

### 8.4 可验证标准

阶段F 完成后，必须能够验证：

1. 同一套目标循环能在新系统上运行。
2. 失败分类和套路动作能跨系统复用。
3. 能识别哪些经验仅适合当前项目。
4. 能识别哪些经验值得晋升为平台级。
5. 新系统接入不会破坏既有 run 级产物读取。

### 8.5 阶段风险

- 风险是把单系统经验误判为平台能力。

### 8.6 阶段修正

- 只有跨多个功能点、跨多次 run 仍稳定有效的经验，才允许进入平台级候选。

## 9. 阶段间依赖

### 9.1 依赖顺序

```text
阶段A -> 阶段B -> 阶段C -> 阶段D -> 阶段E -> 阶段F
```

### 9.2 允许并行的部分

以下工作可以在不破坏主线的前提下并行推进：

- 运行中心展示与 artifact 读取
- 报告格式化
- 人工任务展示
- 候选沉淀审阅

但这些并行工作不能替代目标循环主线。

## 10. 阶段验收总表

| 阶段 | 验收关键词 | 最小可验证结果 |
| --- | --- | --- |
| 阶段A | 目标循环骨架 | 能注册目标、记录尝试、分类失败、记录套路动作 |
| 阶段B | 菜单目标 | 能发现菜单树并区分展开失败与未覆盖 |
| 阶段C | 页面目标 | 能进入页面并判断可达、白屏、受限 |
| 阶段D | 功能点目标 | 能识别真实功能点并生成执行型测试用例 |
| 阶段E | 执行与复盘 | 能输出执行结果、失败簇、下一轮计划和报告 |
| 阶段F | 跨系统验证 | 能把同一套目标循环复用于新系统，并输出平台候选 |

## 11. 计划中的漏洞审查循环

为了满足“不断重复检查和修正”的要求，每个阶段结束后都要进行一次最小审查。

### 11.1 目标偏差检查

确认本阶段是否又退回到模板、样本或单页面特例。

### 11.2 证据缺口检查

确认本阶段的结果是否有足够证据支撑。

### 11.3 泛化性检查

确认本阶段的经验是否只能在一个系统上成立。

### 11.4 安全边界检查

确认本阶段没有把高风险动作默认化。

### 11.5 下一阶段可运行性检查

确认下一阶段可以在本阶段产物基础上直接启动。

## 12. 暂不纳入本计划的内容

以下内容仍然不作为本期主线：

- 自动修改平台代码的完整闭环
- 复杂业务正确性的自动裁决
- 多角色权限矩阵自动覆盖
- 全量页面穷举
- 将模板文件暴露给普通用户作为主入口

## 13. 结论

第二阶段 v4 的实施计划不是“做更多功能”，而是：

1. 先把目标循环做厚。
2. 再把菜单、页面、功能点三类目标串起来。
3. 再把证据、复盘和下一轮计划接稳。
4. 最后把跨系统复用和平台候选做实。

这样才能保证每个阶段都能独立运行、独立验证，也才能让后续技术方案的演进有稳定基座。
