# AI Agent 软件自动化评测平台需求分析书（第二阶段 v3）

更新时间：2026-06-25

## 1. v3 修订目的

本文档是对《需求分析第二阶段.md》的纠偏版，不覆盖旧文档。

本次修订来自“追本溯源管理系统”接入试跑后的复盘。试跑暴露出一个核心问题：

**当前原型和运行中心已经偏向“模板接入与样本回归工程”，但用户最初期望的是“给定已登录 Web 系统后，自动发现页面入口、识别功能点、生成并执行执行型测试用例、输出证据和总体报告”。**

因此 v3 的目标是重新校准第二阶段需求边界、用户体验和验收标准，避免继续沿着“让用户填写大量模板文件”或“只用 raw CDP 扫当前页 DOM”的方向前进。

本次补充修订进一步明确：

- 第二阶段 v3 的探索和测试技术主线应是 **Browser-use + Playwright**，参考 `demo/suyuan_test3.py` 中“Browser-use Agent + CDP 浏览器连接 + 自定义页面工具”的工作方式。
- raw CDP 只能作为浏览器连接、预检、截图或低层兼容兜底能力，不能被误当成完整探索能力。
- v3 首要目标是搭建“每轮探索/测试 -> 错误分析 -> 经验沉淀 -> 下一轮扩大成功范围”的循环闭环，从而形成可控自进化。
- 操作界面化建立在闭环能力之上。运行中心是总控界面，但不能用漂亮界面掩盖底层探索能力不足。

## 2. 复盘结论

### 2.1 结论摘要

对照旧版需求和当前实现，结论是：

1. **原始需求基本准确**  
   旧版需求中提出的 5 个目标仍然是第二阶段应达成的核心目标，并没有因为追本溯源系统试跑而失效。

2. **方案后续执行发生偏移**  
   后续实现把大量精力投入到了模板骨架、模板修订清单、项目级动作族、样本验证矩阵、运行中心操作按钮等工程化能力上。这些能力有价值，但它们不是用户第一次接入新系统时最需要看到的主流程。

3. **功能闭环没有完成到可用状态**  
   当前已有若干底层能力，例如 CDP 连接、受控 Playwright discovery、模板执行器、截图、报告模型、运行中心聚合。但它们没有组成一个“输入少、自动跑、可复核”的新系统自动评测闭环。

4. **用户被迫承担了系统应承担的工作**  
   让用户理解并填写 `template.json`、`locator_hints.json`、`baseline.json`、`data_schema.json`，说明系统把内部执行资产暴露成了主操作界面。这违背了“自动化、智能化”的初衷。

### 2.2 旧目标与当前状态对照

| 旧版目标 | v3 判断 | 当前主要缺口 |
| --- | --- | --- |
| 对当前测试账号权限范围内的 Web UI 页面入口进行自动发现 | 目标正确，未达可用状态 | 发现能力存在，但运行中心主流程没有以“系统级自动遍历”为第一入口；容易退化成模板播种 |
| 对页面入口下默认可见功能点和轻量交互后显式出现的功能点进行识别 | 目标正确，部分实现 | DOM 候选扫描和分类有雏形，但没有形成用户可审核、可继续执行的功能点清单工作流 |
| 基于功能点类型生成并执行执行型测试用例 | 目标正确，基本未形成端到端 | 当前以手写/修订模板为主，不是从功能点自动生成执行型测试用例 |
| 记录测试过程日志、关键步骤截图和状态结果 | 目标正确，分散实现 | 截图和进度事件有能力，但与自动发现、用例生成、执行结果没有稳定串成统一证据链 |
| 输出结构化产物和可阅读总体报告 | 目标正确，未面向新系统闭环 | 报告模型存在，但新系统接入后不能自然生成“页面入口-功能点-执行结果-截图-结论”的总体报告 |

## 3. 第二阶段 v3 定位

第二阶段 v3 定位为：

**面向已登录 Web UI 系统的自动探索与基础验证子系统。用户提供目标系统入口和测试账号上下文后，系统应自动发现页面入口、识别功能点、生成安全范围内的执行型测试用例并执行，最终输出可复核证据和总体报告。**

v3 不再把“模板编写”作为用户主流程。

模板、定位器提示、基线数据和数据 schema 可以继续作为内部资产存在，但默认应由系统自动生成、自动修订建议、自动引用；只有在高级调试或人工干预时才暴露给用户。

## 4. v3 产品原则

### 4.0 运行中心总控优先

第二阶段 v3 必须明确：

**运行中心是第二阶段的查询、操作、监控、人工介入和报告查看的集中平台。**

CLI、脚本、模板文件和原始 JSON 产物仍可存在，但它们只能作为后台能力、调试能力或高级模式存在，不能成为普通用户完成第二阶段评测的主路径。

运行中心至少承担以下职责：

- 新建和配置一次自动评测运行。
- 启动、暂停、恢复和终止运行。
- 展示当前阶段、当前对象、当前轮次、事件列表和阻塞原因。
- 展示页面入口、功能点、执行型测试用例、执行结果和报告。
- 承载人工审核、人工录制、数据更正和策略授权。
- 提供所有结构化产物和截图证据的安全访问入口。

如果某个关键动作只能通过命令行或手工改文件完成，则该动作在 v3 中视为“未完成产品化”。

### 4.1 用户目标优先

用户进入第二阶段运行中心时，看到的主问题应是：

- 我要测哪个系统？
- 当前是否已登录？
- 自动发现到了哪些页面入口？
- 识别到了哪些功能点？
- 哪些功能点可以安全自动验证？
- 哪些需要人工确认或前置数据？
- 最终报告和证据在哪里？

运行中心切换或新建 run 时，操作反馈必须明显标识当前 run 上下文；新建 run 不应沿用上一 run 的反馈信息。

用户不应首先面对：

- 该建几个模板目录
- `template.json` 怎么写
- `locator_hints.json` 怎么配
- `baseline.json` 和 `data_schema.json` 有什么区别
- 哪个 CLI 命令应该先跑

### 4.2 自动化优先，人工审核兜底

v3 默认主流程为：

1. 系统自动发现。
2. 系统自动识别和分类。
3. 系统自动生成执行型测试用例。
4. 系统按安全策略自动执行可执行部分。
5. 人工只审核不确定项、高风险项和失败归因。

人工参与应是“审核和补充”，不是“替系统写执行模板”。

### 4.2.0 探索能力优先于界面化

v3 的建设顺序必须是：

```text
Browser-use + Playwright 探索/测试闭环
  -> 每轮结果分析和经验沉淀
  -> 下一轮扩大成功范围
  -> 运行中心操作界面化
```

不能先把 raw CDP 的当前页扫描包装成运行中心界面，再把它称为“自动发现”。如果底层没有完成菜单遍历、页面探索和低风险验证，界面只能展示“不具备能力”的真实状态。

### 4.2.0.1 技术主线

第二阶段 v3 的探索和测试技术主线为：

- **Browser-use**：负责页面语义理解、目标寻找、多步导航决策、复杂菜单/弹窗/动态页面的探索辅助。
- **Playwright**：负责确定性动作执行、菜单点击、页面跳转、表单输入、截图、等待、网络事件采集和状态恢复。
- **raw CDP**：仅作为连接、预检、截图、浏览器 target 枚举等底层能力，不承担完整菜单发现或页面探索主线。

`demo/suyuan_test3.py` 中的实践应作为 v3 能力目标参考：Browser-use Agent 负责高层任务，Playwright/CDP 浏览器对象和自定义工具负责可复核动作与证据采集。

### 4.2.1 自动执行与 AI 闭环

v3 的目标不是只跑一轮自动化，而是形成循环：

```text
自动发现 / 自动执行
  -> AI 分析错误、风险、覆盖缺口和可沉淀成果
  -> 生成下一轮测试计划、修正建议和资产候选
  -> 运行中心展示并允许人工确认关键决策
  -> 自动进入下一轮或等待人工处理
```

每轮完成后，AI 必须至少产出：

- 本轮覆盖了哪些页面入口和功能点。
- 本轮新增发现了什么。
- 哪些执行失败，失败原因初判是什么。
- 哪些失败可以自动修正后重试。
- 哪些需要人工确认或补前置数据。
- 哪些定位器、动作策略、测试数据或用例可以沉淀。
- 下一轮建议测试哪些页面入口和功能点。

AI 复盘结果必须是结构化产物，并在运行中心可见。系统不得只把 AI 分析写到日志里。

### 4.2.2 自进化边界

自进化不是让系统自由改代码或自由提交风险动作。

v3 中允许自动沉淀：

- 更稳定的定位器候选。
- 功能点类型修正。
- 低风险测试数据生成策略。
- 执行型测试用例改进建议。
- 页面入口去重规则。
- 失败归因规则。
- 已发现但未覆盖目标的下一轮继续策略。

v3 中必须人工确认后才能生效：

- 高风险动作白名单。
- 会改变真实业务数据的提交动作。
- 平台级通用规则晋升。
- 影响多个系统的全局模板或策略。
- AI 对业务正确性的判断。

### 4.3 证据链优先

每个结论必须能回溯到：

- 页面入口
- 功能点
- 执行型测试用例
- 执行动作
- 页面反馈
- 截图或日志
- 状态分类
- 人工确认标记

### 4.4 安全边界硬约束

v3 仍保持旧版安全原则：

- 默认禁止高风险真实提交。
- 查询、查看、导航、打开弹窗、触发前端校验等低风险动作可自动执行。
- 新增、编辑、删除、审批、发布、提交、外部通知等动作必须经过策略判断和人工确认。

## 5. v3 用户主流程

### 5.1 新系统接入

用户最少输入：

1. 系统名称
2. 首页 URL
3. 已登录 Chrome CDP 地址，默认 `http://localhost:9222`
4. 测试账号/角色说明
5. 探索范围说明，可为空
6. 高风险动作策略，默认禁止

用户点击“开始自动评测”后，系统进入自动流程。

### 5.2 环境与登录检查

系统自动检查：

- Python 执行环境
- Playwright/浏览器连接
- CDP 是否可用
- 当前页面是否已登录
- 当前 URL 是否与输入入口一致
- 是否能读取 DOM、标题、导航候选和主要内容区

如果未登录，系统进入登录接管提示，而不是继续生成空模板。

### 5.3 第一轮：菜单入口完全遍历

第一轮探索的目标不是测试某个功能点，也不是寻找单个用户指定页面，而是完整发现当前测试账号权限范围内的菜单入口树。

第一轮必须自动发现：

- 顶部菜单、侧边菜单和折叠菜单容器。
- 一级菜单。
- 二级菜单。
- 三级菜单。
- 可展开但尚未进入的菜单分组。
- 叶子菜单项及其点击后的 URL、route 或页面状态变化。

以“追本溯源管理平台”为例，如果首页可见 3 个一级菜单，第一轮必须发现这 3 个一级菜单；如果“线上备案申请”是某个一级菜单下的二级菜单，第一轮必须在菜单树中发现它。第一轮只返回首页 1 个页面入口，视为 `menu_discovery_incomplete`，不能判定为完成。

第一轮输出：

- `menu_tree.json`
- `menu_entries.json`
- `menu_traversal_log.jsonl`
- `navigation_tree.json`
- 每个一级菜单展开后的截图
- 未展开、疑似遗漏、被权限阻断或识别不确定的菜单清单

第一轮不做：

- 表单填写
- 提交、删除、审批等副作用动作
- 页面内功能点深度验证
- 把当前页 DOM 快照误当成完整系统地图

### 5.3.1 第二轮：按菜单入口逐页探索

第二轮基于第一轮 `menu_tree.json` 和 `menu_entries.json`，逐个进入叶子菜单或被选中的菜单入口，探索页面内容和功能点。

第二轮自动发现：

- 页面标题、URL、route 和主要内容区。
- 查询/筛选/重置区域。
- 表格、详情入口、行级动作。
- 新增、编辑、删除、导出、审批、提交等按钮入口。
- Tab、弹窗、抽屉、下拉菜单等轻量交互后显式出现的功能点。
- 页面入口截图和关键交互截图。

第二轮输出：

- `page_entries.json`
- `feature_points.json`
- `page_exploration_log.jsonl`
- `screenshots_index.json`
- 页面级跳过原因和风险标记

### 5.3.2 后续轮次：测试执行与范围扩大

第三轮及后续轮次基于前一轮的页面入口和功能点清单，生成执行型测试用例并执行低风险或已授权动作。

每轮结束后，系统必须进入 AI/规则复盘，分析：

- 本轮新增覆盖了哪些菜单、页面和功能点。
- 哪些菜单或页面仍未探索。
- 哪些功能点因定位、权限、前置数据或安全策略未执行。
- 哪些成功路径可以沉淀为项目级经验。
- 下一轮应扩大到哪些菜单、页面或功能点。

### 5.4 功能点自动识别

系统对每个页面入口识别：

- 查询/筛选
- 重置
- 详情查看
- 新增入口
- 编辑入口
- 删除入口
- 导出/下载
- Tab/切换
- 弹窗动作
- 行级动作
- 其他可见业务动作

功能点识别必须包含：

- 功能点名称
- 所属页面入口
- 功能点类型
- 风险等级
- 可自动验证性
- 建议验证策略
- 证据来源
- 置信度

### 5.5 人工审核开关

发现后系统应显示“发现结果审核”界面。

用户可：

- 勾选/取消页面入口
- 勾选/取消功能点
- 修改系统识别出的名称
- 调整功能点类型
- 标记前置数据要求
- 允许或禁止特定高风险动作
- 选择“自动继续验证”或“先保存清单”

审核结果必须结构化落盘，并被后续验证消费。

### 5.5.1 人工介入必须界面化

人工可以介入，但介入必须通过运行中心界面完成，不能要求用户直接修改 JSON 或模板文件。

允许的人工介入场景包括：

- 选择优先测试的页面入口或模块。
- 在页面入口树中排除不测范围。
- 审核和更正功能点名称、类型、风险等级。
- 对系统识别错误的字段、按钮、表格列做界面化更正。
- 在浏览器中进行实际操作，让系统录制为候选路径。
- 更正测试数据、输入样例或前置数据说明。
- 对高风险动作进行授权或拒绝。
- 审核 AI 生成的下一轮计划。
- 审核项目级沉淀和平台级晋升候选。

运行中心应为这些介入提供明确任务卡、表单、选择器、录制按钮和确认按钮。

以下做法在 v3 主流程中不允许作为正常用户操作：

- 要求用户直接编辑 `template.json`。
- 要求用户直接编辑 `locator_hints.json`。
- 要求用户直接编辑 `baseline.json`。
- 要求用户直接编辑 `data_schema.json`。
- 要求用户通过复制 CLI 命令来推进常规阶段。
- 要求用户在文件系统中寻找产物后再判断下一步。

### 5.6 执行型测试用例生成

系统基于功能点类型自动生成执行型测试用例。

每条测试用例至少包含：

- 用例 ID
- 页面入口
- 功能点
- 功能点类型
- 风险等级
- 前置条件
- 执行动作
- 期望页面反馈
- 终止条件
- 截图点
- 自动判定规则
- 人工确认项

用户不需要手写模板文件。

### 5.7 安全执行

系统按策略执行：

- 导航类：自动进入页面并确认加载
- 查询类：使用默认条件或低风险输入执行查询
- 重置类：执行重置并检查表单状态变化
- 详情类：打开详情并确认关键内容可见
- Tab/切换类：切换并确认内容变化
- 导出类：默认只确认入口和下载响应，是否实际下载由策略控制
- 新增/编辑/删除/提交类：默认只验证入口、表单、确认弹窗或前端校验，不做最终真实提交

### 5.8 结果与报告

执行完成后必须输出：

- 页面入口汇总
- 功能点汇总
- 执行型测试用例清单
- 执行结果清单
- 未执行原因
- 失败原因初分
- 截图索引
- 结构化 JSON 产物
- Markdown 或 HTML 总体报告

报告不能声称“业务验收通过”，只能给出探索式状态分类和人工确认提示。

### 5.9 AI 复盘与下一轮计划

每轮执行结束后，系统必须自动进入 AI 复盘阶段。

AI 复盘输入：

- 本轮页面入口和功能点清单。
- 执行型测试用例。
- 执行动作日志。
- 页面反馈。
- 截图索引。
- 网络事件。
- 失败原因。
- 人工审核和人工录制结果。

AI 复盘输出：

- `round_analysis.json`
- `failure_clusters.json`
- `improvement_candidates.json`

## 9. 运行中心 v3 需求

第二阶段运行中心应是独立页面，并以任务流程为主，而不是模板文件为主。

### 9.1 首页布局

运行中心首页至少包含：

1. 新建运行
2. 当前运行状态
3. 第一轮菜单树遍历结果
4. 第二轮页面入口探索结果
5. 功能点发现结果
6. 生成的执行型测试用例
7. 执行证据与截图
8. 报告入口
9. 人工审核/接管任务
10. AI/规则复盘与下一轮计划
11. 沉淀成果审阅

### 9.1.1 总控视图

运行中心必须提供总控视图，让用户无需打开命令行即可完成：

- 创建 run。
- 查看 run 列表和历史轮次。
- 查看当前 run 正在执行的阶段。
- 查看菜单遍历、页面探索和执行统计。
- 查看实时事件流。
- 查看 AI/规则复盘。
- 审核下一轮计划。
- 启动下一轮探索或测试。
- 查看报告和证据。

总控视图不应混入第一阶段项目设置、需求解析、用例设计等无关内容。

### 9.2 新建运行表单

新建运行只要求少量用户可理解字段：

- 系统名称
- 首页 URL
- 测试账号说明

- 探索范围
- 安全策略

- 第一轮菜单树
- 菜单项展开/点击状态

- 哪些通过？
- 哪些失败？
- 哪些因为安全或前置数据未执行？
- 哪些需要人工确认？

## 10. v3 验收标准

第二阶段 v3 原型只有满足以下条件，才视为达成最初需求的最小可用版本。

1. 用户不手写模板文件，也能从已登录系统首页发起一次运行。
2. 第一轮必须自动产出菜单树，覆盖当前首页可见的一级菜单及其可展开的二级、三级菜单；只发现首页 1 个页面入口不合格。
3. 对于“线上备案申请”这类二级菜单，第一轮必须在 `menu_tree.json` 或 `menu_entries.json` 中发现其菜单路径，否则标记 `menu_discovery_incomplete`。
4. 第二轮必须基于第一轮菜单树逐个进入叶子菜单或选中菜单入口，并产出页面入口清单。
5. 系统能对每个已访问页面产出候选功能点。
6. 系统能把功能点分类为导航、查询、详情、表单、导出、切换等类型。
7. 系统能基于功能点自动生成执行型测试用例。
8. 系统能执行低风险测试用例，并跳过高风险最终提交。
9. 系统能记录进度事件、菜单遍历日志、页面探索日志、动作日志、截图和状态结果。
10. 系统能输出结构化产物和可阅读总体报告。
11. 系统能明确列出未执行原因和人工确认项。
12. 运行中心能完整展示菜单遍历、页面探索、审核、执行、报告和复盘阶段。
13. 运行中心能作为查询、操作、监控总入口完成一次 run，不要求用户执行 CLI。
14. 每轮执行后 AI/规则分析能生成失败分析、沉淀候选和下一轮计划。
15. 人工介入可通过界面完成，不要求用户直接修改 JSON 或模板文件。
16. 探索和测试主链路必须使用 Browser-use + Playwright；仅使用 raw CDP 当前页快照不满足 v3 验收标准。

## 11. 当前实现差距

当前实现中可复用的部分：

- CDP 连接和环境检查
- 部分 Playwright 受控 discovery 能力
- 模板执行器
- 通用动作 registry
- 截图采集
- 进度事件与当前状态产物
- 报告模型和运行中心聚合
- 人工录制和候选模板审阅包

当前实现中需要纠偏的部分：

- v3 run 的真实浏览器路径仍停留在 raw CDP 当前页 DOM 快照，不能发现完整菜单树。
- 没有把 Browser-use + Playwright 作为探索和测试主链路；Browser-use 没有参与菜单理解、目标寻找和多步探索。
- 第一轮没有单独完成菜单入口完全遍历；首页存在多个一级菜单时，仍可能只产出 1 个首页页面入口。
- 第二轮没有基于菜单树逐个进入页面做探索，导致“线上备案申请”这类二级菜单无法稳定被发现。
- `进入下一轮` 目前可能只创建 `planned` round，没有自动执行下一轮探索，状态语义对用户不清晰。
- 运行中心主流程仍围绕模板步骤，而不是自动评测任务。
- 系统地图探索没有成为稳定的默认主流程。
- 功能点发现结果没有直接转成执行型测试用例。
- 模板文件暴露过多，用户需要理解内部格式。
- 报告不是新系统自动评测的自然终点。
- 当前 `artifacts/stage2` 可能只留下 operation 会话，而没有完整 run 产物。

## 12. v3 暂不实现

v3 仍不承诺：

- 自动判断复杂业务规则是否正确
- 自动完成所有登录验证码
- 多角色权限矩阵自动覆盖
- 高风险真实提交默认自动执行
- 复杂业务前置数据自动构造
- 正式 Word/PDF 验收报告
- API、移动端、桌面端自动化测试

## 13. v3 下一步建议

下一步不应继续优先增加模板编辑界面，而应优先实现：

1. Browser-use + Playwright 的真实探索执行器，替代 raw CDP 当前页快照作为主发现能力。
2. 第一轮菜单入口完全遍历，生成 `menu_tree.json`、`menu_entries.json` 和菜单遍历证据。
3. 第二轮基于菜单树逐页探索，生成 `page_entries.json` 和 `feature_points.json`。
4. 每轮探索/测试后的 AI/规则复盘、失败分析、经验沉淀和下一轮计划。
5. `进入下一轮并执行` 的 run 状态机，避免停留在用户无法理解的 `planned` 状态。
6. discovery 结果到执行型测试用例的自动生成器。
7. 功能点审核界面和低风险用例自动执行器。
8. run 级总体报告生成。
9. 模板资产内部化和高级模式隐藏。

## 14. 漏洞审查与修正闭环

本节用于把需求继续“拧紧”，避免 v3 再次跑偏。

### 14.1 第一轮漏洞审查：入口形态

漏洞：继续把 CLI 和模板文件当作实际主流程，只在运行中心外面包一层按钮。

修正：运行中心必须成为唯一普通用户入口。所有常规动作都必须可通过运行中心完成；CLI 只能是后台执行方式或高级调试入口。

### 14.2 第二轮漏洞审查：自动化不足

漏洞：系统只完成页面扫描，不生成和执行测试用例。

修正：验收标准必须要求 `feature_points -> generated_test_cases -> execution_results -> report` 全链路产物都存在，并且运行中心可展示。

### 14.3 第三轮漏洞审查：AI 只做总结，不驱动下一轮

漏洞：AI 分析停留在报告文本，没有形成下一轮可执行计划。

修正：每轮结束必须生成 `next_round_plan.json`，并在运行中心提供“自动继续 / 人工确认后继续 / 停止”的决策入口。

### 14.4 第四轮漏洞审查：自进化失控

漏洞：系统把 AI 建议直接写入全局规则或执行高风险动作。

修正：自进化分层。运行时临时修正可自动使用；项目级沉淀需运行中心审阅；平台级晋升必须人工批准；高风险动作永远不能由 AI 单独授权。

### 14.5 第五轮漏洞审查：人工介入又退回文件编辑

漏洞：用户为了修正识别错误，仍要打开 JSON 文件。

修正：所有人工介入都必须有界面形态，包括选择页面、修正功能点、录制操作、更正测试数据、授权策略和审阅沉淀。

### 14.6 第六轮漏洞审查：报告不可复核

漏洞：报告只有自然语言总结，无法追溯证据。

修正：报告中的每个页面入口、功能点、测试用例和结论都必须关联截图、动作日志和状态结果。

### 14.7 第七轮漏洞审查：追求“全自动”导致遇阻即失败

漏洞：登录、验证码、前置数据、复杂业务动作导致系统中断。

修正：运行中心必须提供人工接管任务队列。人工处理后，系统从结构化上下文恢复，不丢失 run 状态。

### 14.8 第八轮漏洞审查：探索范围不可控

漏洞：系统无限探索、误入无关页面或触发危险动作。

修正：运行中心提供探索范围、排除范围、深度、数量和风险策略配置；执行层硬拦截危险动作。

### 14.9 第九轮漏洞审查：低质量数据污染沉淀

漏洞：一次失败或误判被沉淀为项目级/平台级规则。

修正：沉淀候选必须带证据、置信度、适用范围、来源 run 和审阅状态。平台级沉淀必须多轮验证或人工批准。

### 14.10 第十轮漏洞审查：需求与实现验收脱节

漏洞：实现完成了很多工程组件，但没有满足用户可感知目标。

修正：v3 验收以用户可完成的端到端任务为准，而不是以 CLI 命令数量、模板数量或 artifact 数量为准。

### 14.11 当前信心结论

经过上述漏洞审查后，v3 需求层面暂无已知阻塞性缺口。后续若真实系统试跑暴露新问题，应继续通过本节补充漏洞和修正项，而不是把修正散落到实现细节里。
