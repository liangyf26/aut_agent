# 第二阶段原型采用单主进程、多逻辑代理与受控 worker，而不采用全量多进程架构

## Status

Accepted

## Context

第二阶段技术方案引入了主代理循环、多角色协作、失败簇分析、人工介入和项目级沉淀。如果一开始就把这些角色全部实现为真实多进程或多服务架构，会立刻引入以下复杂度：

- 跨进程消息编排
- 状态同步与任务恢复
- 日志汇聚与可观测性设计
- 人工介入节点的跨进程暂停与恢复
- 浏览器执行、证据采集和报告渲染之间的调度协调

当前阶段的首要目标不是验证多进程基础设施，而是验证以下能力边界：

- 页面入口发现
- 功能点识别
- 单功能点基础路径验证
- 失败簇归因与循环优化
- 项目级沉淀与平台级候选项提炼

因此，需要一个更利于调试、状态追踪和快速迭代的原型运行时形态。

## Decision

第二阶段原型采用：

**单主进程 + 多逻辑代理 + 受控 worker**

具体含义如下：

- 主代理在单一主进程内运行，负责阶段切换、循环调度、判停、人工介入路由和沉淀分层决策
- 页面探索代理、功能点分析代理、失败分析代理、技能沉淀代理和报告整理代理在原型阶段先实现为逻辑角色，而非独立操作系统进程
- 浏览器执行、证据采集、报告渲染等高副作用或高成本任务，可按需实现为受控 worker
- 原型阶段不引入完整的多进程消息总线或分布式任务编排系统

## Consequences

正面影响：

- 降低原型阶段基础设施复杂度
- 更容易复现实验结果和调试循环逻辑
- 更容易观察“发现 -> 验证 -> 失败簇 -> 修正 -> 再验证”是否真正有效
- 可在后续稳定后按需拆分真实 worker 或服务

负面影响：

- 进程级隔离能力较弱
- 某些重任务的资源竞争需通过受控 worker 局部缓解
- 后续若演进为正式平台，仍需规划进程拆分与任务编排

## Rejected Alternatives

### 1. 一开始就实现为全量多进程 / 多服务架构

不采用。原因是这会把第二阶段原型拖入基础设施工程，模糊执行子系统能力验证的主目标。

### 2. 完全单线程单角色串行执行

不采用。原因是这会削弱循环优化、失败簇分析、沉淀分层和人工介入等多角色协作能力，不利于验证第二阶段方案的关键思想。
