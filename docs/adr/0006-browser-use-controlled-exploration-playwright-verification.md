# Browser Use 用于受控探索，验证阶段默认由 Playwright 确定性执行

## Status

Accepted

## Context

第二阶段需要在不依赖需求说明书和功能说明书的前提下，对 Web UI 系统进行页面入口发现、功能点识别和单功能点基础路径验证。

Browser Use 能帮助 Agent 理解页面、推断入口和规划动作，但其自由端到端执行存在以下风险：

- 本地小模型能力不稳定时，容易偏离既定测试模板
- 工具调用自由度高，失败归因困难
- 验证阶段结果难以稳定复现
- 成功路径难以直接沉淀为可回放脚本

Playwright 更适合承担确定性执行、截图、下载监听、等待、状态恢复和脚本回放。

因此需要明确 Browser Use 与 Playwright 的职责边界。

## Decision

第二阶段采用以下执行边界：

- 发现阶段允许 Browser Use + Playwright 协同，用于页面理解、页面入口推断和功能点推断
- 验证阶段默认采用 Playwright 确定性执行
- Browser Use 不作为验证阶段的默认端到端执行器
- 仅当功能点被规则或人工明确标记为视觉兜底场景时，允许 Browser Use 参与验证
- 视觉兜底输出必须转化为可复核的动作路径、执行证据或 Playwright 可回放脚本候选

## Consequences

正面影响：

- 提升验证阶段的可复现性
- 降低模型自由执行带来的不确定性
- 让失败归因更清晰
- 更利于后续沉淀 Playwright 脚本和项目级技能

负面影响：

- 发现阶段与验证阶段需要维护两套不同的执行策略
- 对复杂视觉页面，Playwright 默认路径可能覆盖不足
- 需要额外设计视觉兜底候选标记和人工确认机制

## Rejected Alternatives

### 1. 全程由 Browser Use 自由执行

不采用。原因是验证阶段要求稳定、可复现、可回放，而 Browser Use 自由执行更适合探索，不适合作为默认验证执行器。

### 2. 完全禁用 Browser Use

不采用。原因是第二阶段的页面入口发现和功能点推断需要一定页面语义理解能力，完全依赖 Playwright 结构化定位会降低自动发现能力。

### 3. 验证阶段默认 Browser Use，失败后再转 Playwright

不采用。原因是这会让验证结果和失败原因先被模型自由执行污染，再尝试恢复确定性，不利于形成稳定脚本和报告。
