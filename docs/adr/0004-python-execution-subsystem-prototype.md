# 第二阶段采用独立 Python 执行子系统原型，而不替换现有 Node.js 主线

## Status

Accepted

## Context

第一阶段已经形成 Node.js 主线 MVP，覆盖项目管理、需求解析、用例生成、模拟执行和报告输出。第二阶段的新目标发生了明显变化：

- 不再以需求说明书和功能说明书为主要输入
- 转向对 Web UI 系统进行自动发现、功能点识别和单功能点基础路径验证
- 技术上更依赖 Python 生态中的 Browser Use 与 Playwright 协同
- 部署环境大多位于内网，优先考虑本地大模型或 OpenAI 兼容接口接入

如果直接把现有主线整体切换为 Python，会把两类风险耦合在一起：

1. 第二阶段执行能力本身是否成立
2. 平台主线是否应整体迁移

这会显著增加交付风险，并削弱我们对执行子系统能力边界的验证效率。

## Decision

第二阶段采用**独立 Python 执行子系统原型**，用于验证页面入口发现、功能点识别、单功能点基础路径验证、结构化产物输出、失败簇分析和技能沉淀等能力。

现有 Node.js 主线平台暂时保留，不在第二阶段立即替换。

当前阶段采取以下策略：

- Node.js 主线继续作为现有平台壳和第一阶段能力承载体
- Python 原型专注执行子系统能力验证
- Python 原型通过结构化产物目录输出结果，为未来平台接入预留边界
- 待第二阶段原型能力稳定后，再评估与现有 Node.js 平台的集成方式

## Consequences

正面影响：

- 将“执行子系统能力验证”与“平台主线迁移”拆开，降低整体风险
- 能更快验证 Browser Use + Playwright + 本地小模型的实际适配边界
- 保留未来多种集成路径：
  - 继续双栈
  - Python 执行子系统作为独立服务
  - 后续部分或整体迁移

负面影响：

- 短期内形成 Node.js 主线与 Python 原型并存的双栈状态
- 需要额外定义结构化产物格式和未来集成边界
- 原型阶段的部分工作可能在平台接入时需要二次整理

## Rejected Alternatives

### 1. 直接将主线整体改为 Python

不采用。原因是这会把平台重构与执行能力验证绑定，风险过高，且当前尚无足够证据证明第二阶段执行子系统能力已成熟到可以承担整个平台主线。

### 2. 继续坚持 Node.js 全栈，不引入 Python 原型

不采用。原因是第二阶段明确要求以 Python、Browser Use、Playwright 为核心验证执行能力边界，继续只走 Node.js 会偏离当前阶段目标。

### 3. 暂不实现原型，仅继续完善第一阶段平台

不采用。原因是第二阶段的核心价值就在于验证“无需求文档输入、依赖页面自动发现和单功能点验证”的执行能力，如果不做原型，无法获得真实结论。
