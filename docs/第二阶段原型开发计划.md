# AI Agent 软件自动化评测平台第二阶段原型开发计划

更新时间：2026-06-21

## 1. 文档目的

本文档用于把《需求分析第二阶段》和《技术方案第二阶段》进一步收敛成可执行的原型开发计划，作为进入编码开发前的评审稿。

本计划重点吸收本次“线上备案申请”实战验证的经验，回答以下几个核心问题：

1. 如何把当前跑通的成功路径抽象成平台可复用的“执行模板 + 基线数据”。
2. 如何把第二阶段原型补成完整闭环，而不只是一个可跑通的单脚本。
3. 当没有人工成功样本可复用时，平台如何借助人工录制和自动探索建立首条成功路径。
4. 如何避免反复复用同一批业务数据，改为生成可变化、可追踪、可回收的测试样例数据。
5. 每轮执行后如何生成可复核的分析报告，并对模型表现做横向点评。
6. 如何对成功经验、失败修正和脚本资产做分层沉淀。
7. 如何用这次真实场景的经验反向修正第二阶段技术方案。

## 2. 计划结论

本轮原型开发建议采用以下总路线：

**以单主进程 Python 原型为主体，建立“配置层 + 两段式执行层 + 循环分析层 + 人工介入层 + 产物沉淀层”的最小闭环。**

落地时不再把 `tools/suyuan_submit_loop.py` 继续扩成大脚本，而是把它拆成平台原型的首个项目级执行模板样本。

本轮开发完成后，原型至少应具备以下能力：

1. 对单个 Web UI 系统完成登录接管、发现、验证、归因、沉淀、报告的完整闭环。
2. 支持“执行模板 + 基线数据 + 测试数据生成策略”三层解耦。
3. 支持人工录制一遍操作，平台据此生成项目级执行基线。
4. 支持对 AI-tester 与 Qwen 两个本地模型执行同一任务，并输出结构化对比结果。
5. 支持生成项目级技能沉淀清单和平台级晋升候选清单。

## 3. 本次原型验证带来的关键修正

### 3.1 对需求边界的修正

本次“线上备案申请”实战说明：

- 真实页面可通过 Playwright + 已登录会话稳定执行。
- 真正帮助我们快速走通的，不是“自由探索”，而是“已验证路径 + 稳定回放 + 必填项补全 + 网络反馈判定”。
- 如果直接把这一条样本路径误当成平台通用能力，会高估原型成熟度。

因此第二阶段原型要明确拆成两类能力：

1. **平台骨架能力**
   - 发现
   - 验证
   - 归因
   - 报告
   - 沉淀

2. **项目级样本能力**
   - 当前项目的定位器
   - 当前项目的页面规则
   - 当前项目的执行模板
   - 当前项目的基线数据

### 3.2 对技术路线的修正

本次原型说明，验证阶段不能依赖 Browser Use 自由执行。

修正后的执行边界应更清晰：

- Browser Use：
  - 用于发现阶段的受控探索
  - 用于视觉兜底
  - 用于页面语义理解
- Playwright：
  - 用于所有稳定执行动作
  - 用于截图、等待、下载、网络监听、状态恢复
- LLM：
  - 用于规划、分类、归因、总结、候选修正建议
  - 不直接承担所有浏览器底层动作

### 3.3 对“成功固化”的修正

本次成功路径已经证明：

- “执行模板”与“样例数据”必须分离。
- “样例数据”还要继续拆成：
  - 参考基线数据
  - 生成规则
  - 本轮运行实际使用的数据

否则平台后续无法适配别的系统，也无法避免“拿旧数据作弊”的问题。

## 4. 原型范围

## 4.1 本轮要完成

1. 原型目录骨架与模块边界。
2. 执行模板机制。
3. 基线数据与测试数据生成机制。
4. 发现阶段最小闭环。
5. 验证阶段最小闭环。
6. 失败簇归因与循环调度。
7. 人工录制与人工接管入口。
8. 结构化产物目录与可阅读报告。
9. 技能沉淀与晋升候选输出。
10. 用“线上备案申请”作为首个项目级模板样本完成回归验证。

## 4.2 本轮暂不完成

1. 替换现有 Node.js 主平台。
2. 多项目并发调度和分布式 worker。
3. 正式 Web 管理后台。
4. 复杂业务链自动造数。
5. 多角色权限矩阵自动编排。
6. 标准 Word/PDF 验收文档导出。
7. 性能压测和安全测试。

## 5. 原型目标架构

建议把第二阶段原型落成如下目录和模块边界：

```text
prototype/
  stage2/
    app/
      config/
      orchestration/
      discovery/
      verification/
      fallback/
      reporting/
      knowledge/
      human_loop/
      data_factory/
      progress/
      runtime/
    templates/
      suyuan_online_apply/
        template.json
        baseline.json
        data_schema.json
        locator_hints.json
    runs/
      <run_id>/
```

> 说明：目录名可在编码时微调，但“平台代码”和“项目模板”必须分开。

### 5.1 配置层

负责：

- 被测系统地址
- 登录模式
- 模型配置
- 风险白名单
- 停止条件
- 截图规则
- 数据生成规则

建议产物：

- `project_config.json`
- `model_profiles.json`
- `run_policy.json`

### 5.2 调度层

负责完整循环：

1. 初始化 run
2. 模型能力预检
3. 登录接管或自动登录
4. 发现阶段
5. 人工审核开关
6. 验证阶段
7. 失败归因
8. 沉淀与报告
9. 判定是否进入下一轮

此外，调度层还负责在以下时机发出标准化进度事件：

- 阶段开始
- 阶段完成
- 步骤开始
- 步骤完成
- 进入等待人工
- 命中停止条件
- 发生异常或切换到下一轮

### 5.3 发现层

负责：

- 遍历页面入口
- 构建页面地图
- 推断功能点
- 给功能点套用类型模板

本轮最低要求：

- 能产出页面入口清单
- 能产出功能点清单
- 能记录页面入口截图
- 能标记来源是“自动发现”还是“人工录制补充”

### 5.4 验证层

负责：

- 执行模板装载
- 标准起点恢复
- 基线数据装载
- 测试样例数据生成
- Playwright 确定性执行
- 步骤日志、网络日志、截图采集
- 状态分类

### 5.5 循环分析层

负责：

- 失败项聚类为失败簇
- 归因为环境问题、模型问题、定位问题、前置数据问题、页面变化问题等
- 输出下一轮修正建议
- 输出哪些修正只用于当前 run，哪些应固化到项目级模板

### 5.6 人工介入层

负责：

- 登录接管
- 人工录制
- 发现结果审核
- 功能点清单审核
- 高风险动作审批
- 平台级晋升审批

原型阶段不要求复杂界面，但至少要有可实际使用的入口。

### 5.7 产物与报告层

负责：

- 结构化 artifacts
- Markdown 报告
- HTML 报告
- 模型对比结果
- 技能沉淀总结
- 日报

### 5.8 运行态流程可视化层

这是本轮新增的明确要求。

目标不是展示浏览器点击细节，而是让人直观看到平台当前运行到了哪个流程环节、卡在了哪里、下一步准备做什么。

建议把它定义成：

**运行态编排进度视图**

负责展示：

- 当前阶段：预检 / 登录接管 / 发现 / 人工审核 / 验证 / 失败归因 / 沉淀 / 报告
- 当前轮次：第几轮发现、第几轮验证、第几轮归因
- 当前步骤：例如“自动遍历页面入口”“构建页面地图”“识别功能点”“生成执行型测试用例”
- 当前对象：当前页面入口、当前功能点、当前失败簇
- 当前状态：运行中 / 已完成 / 失败 / 等待人工 / 已跳过
- 开始时间、已耗时、最近心跳时间
- 本阶段累计统计：已发现页面入口数、已识别功能点数、已完成验证数、失败簇数
- 下一动作提示：完成当前步骤后即将进入的下一个平台环节

原型阶段不要求一开始就做成完整 Web 看板，但至少应同时提供：

1. 控制台友好的阶段进度输出
2. 结构化进度事件流
3. 可供后续前端消费的当前状态快照

建议最小产物：

- `progress_events.jsonl`
- `current_status.json`
- `phase_summary.json`

## 6. 执行模板 + 基线数据设计

这是本轮开发的第一优先级。

### 6.1 设计目标

把“写死在脚本里的成功路径”改造成：

1. **执行模板**
   - 关注步骤结构
   - 关注页面动作
   - 不携带具体业务值

2. **基线数据**
   - 记录当前项目已验证成功的参考值
   - 记录字段约束
   - 记录可复用枚举

3. **测试数据生成策略**
   - 基于基线约束产生本轮测试数据
   - 保证每轮尽量不同
   - 保留可追溯种子

### 6.2 建议数据结构

建议模板最少拆成四份：

1. `template.json`
   - 页面入口
   - 功能点
   - 执行步骤
   - 成功判定
   - 失败判定

2. `baseline.json`
   - 已验证成功的参考字段值
   - 附件要求
   - 页面/接口稳定特征

3. `data_schema.json`
   - 字段类型
   - 是否必填
   - 取值范围
   - 生成策略
   - 唯一性要求

4. `locator_hints.json`
   - 首选定位器
   - 备用定位器
   - 组件级定位说明

### 6.3 对现有“线上备案申请”的改造要求

以当前成功样本为例，首轮重构要做到：

- 把 `SUCCESS_BASELINE` 移出主脚本。
- 把“同意签署 -> 初始表单 -> 完整育苗表单 -> 上传附件 -> 最终提交”的执行步骤抽成模板。
- 把成功判定中的 UI 文案和关键接口特征写入模板判定规则。
- 把“育苗”场景作为第一个可回归模板样本，不再继续堆新的硬编码分支。

## 7. 循环闭环补全计划

第二阶段原型要从“单次脚本成功”升级为“可循环优化的原型”。

### 7.1 单轮循环定义

每一轮固定包括：

1. 发现
2. 选择待验证目标
3. 验证
4. 失败归因
5. 沉淀修正
6. 报告输出
7. 停止条件判断

每个环节都要可视化呈现为平台流程步骤，而不是只体现在内部日志里。

### 7.2 失败簇分类

建议原型阶段先固定以下一级分类：

1. 环境问题
2. 模型响应问题
3. 定位失败
4. 页面加载超时
5. 权限不足
6. 缺少前置数据
7. 页面规则变化
8. 提交被业务校验拦截
9. 人工接管
10. 未知异常

### 7.3 循环修正分层

每轮修正必须强制落到以下三层之一：

1. 当前运行临时修正
2. 项目级沉淀
3. 平台级候选

这样才能避免“调通一次就把项目特例写进平台基线”。

### 7.4 停止条件

沿用已确认的五类停止条件，并把它们落成程序级规则：

1. 目标完成
2. 改进无收益
3. 安全边界命中
4. 人工接管命中
5. 资源预算耗尽

## 8. 无人工成功样本时的首条路径获取方案

这是本轮必须正面解决的问题。

### 8.1 方案结论

支持“人工演示一遍，平台负责记录和整理”，但不要把它理解成传统 RPA 录制器。

更准确的定位应该是：

**人工引导式基线采集器**

### 8.2 采集内容

当人工手工操作时，平台应同步记录：

- 页面 URL 变化
- 点击目标
- 输入字段
- 选项选择
- 文件上传
- 关键网络请求
- DOM 快照
- 截图
- 人工备注

### 8.3 采集结果

人工录制完成后，平台不直接产出“最终自动化脚本”，而是产出：

1. 候选页面入口
2. 候选功能点
3. 候选执行步骤
4. 候选定位器
5. 候选测试数据 schema
6. 候选成功判定规则

再由平台转为项目级执行模板。

### 8.4 边界提醒

人工录制能显著缩短首条路径建立时间，但也有边界：

- 只能录到可见交互，录不到全部业务语义。
- 录到的字段值不能直接长期复用，仍需转换成数据 schema。
- 录制过程中要对密码、验证码、敏感 token 做脱敏。

## 9. 测试样例数据生成计划

### 9.1 设计目标

解决“重复使用已有成功数据像作弊”的问题。

平台需要引入三层数据来源：

1. **参考基线数据**
   - 只用于理解字段约束和成功结构

2. **自动生成测试数据**
   - 作为每轮实际提交或填写的数据

3. **运行时实际数据记录**
   - 记录本轮真正使用了什么值

### 9.2 生成策略

本轮原型建议先支持以下几类策略：

1. 常量继承
   - 对不可变化的枚举值沿用基线

2. 轻量扰动
   - 文本后缀
   - 日期平移
   - 数值区间微调

3. 规则生成
   - 手机号
   - 统一社会信用代码样式
   - 批次号
   - 备注文本

4. 唯一性生成
   - run_id
   - 时间戳
   - 序号

### 9.3 数据治理要求

每次运行都必须记录：

- 数据来源
- 生成规则
- 实际取值
- 是否命中唯一性约束
- 是否可回收

并在报告中明确标注：

- 本轮是否使用人工参考数据
- 本轮是否使用自动生成数据
- 哪些字段仍因业务限制沿用固定值

## 10. 报告与性能评价计划

### 10.1 每轮运行报告必须包含

1. 页面入口清单与状态
2. 功能点清单与状态
3. 成功项摘要
4. 失败项摘要
5. 失败簇归因
6. 关键步骤截图索引
7. 关键网络事件摘要
8. 本轮数据来源与样例生成摘要
9. 本轮沉淀的项目级资产
10. 建议晋升的平台级候选

此外，运行过程中还应同时生成一份“运行态进度快照”，用于实时展示，不必等整轮报告结束后才能看到状态。

### 10.1.1 运行态进度视图建议展示内容

建议把“运行态流程可视化”与最终报告区分开：

- 最终报告：面向复盘、交付、阅读
- 运行态进度视图：面向监控、调试、人工接管

运行态进度视图建议最少展示以下栏目：

1. 当前 run 基本信息
2. 当前阶段与当前步骤
3. 当前轮次
4. 当前处理对象
5. 已完成/总数进度条
6. 最新 20 条平台级步骤事件
7. 当前等待原因或阻塞原因
8. 下一步计划动作

### 10.2 时间与速度评价

本轮可以做，但要调整表述，避免误称为正式性能测试。

建议报告中只输出以下“交互执行指标”：

- 总运行时长
- 每个功能点耗时
- 关键步骤耗时
- 页面加载等待耗时
- 接口响应耗时摘要
- 重试次数
- 人工接管次数

报告中明确写为：

**执行效率观察**

而不是：

**系统性能结论**

### 10.3 模型点评

在当前选型阶段，同一任务需输出模型对比摘要，至少包含：

- 预检能力标签
- 是否参与发现
- 是否参与归因与总结
- 任务完成率
- 响应稳定性
- 平均响应时长
- 结构化输出稳定性
- 适合承担的角色建议

### 10.4 运行日报

新增日报，用于汇总：

- 新增模板
- 新增定位器策略
- 新增失败修正策略
- 候选平台级技能
- 仍待观察的问题

## 11. 技能沉淀计划

### 11.1 本轮沉淀目标

不把“技能”只理解成 prompt，还包括：

- 页面级定位器策略
- 组件级交互策略
- 执行模板
- 成功判定规则
- 失败修复策略
- 数据生成策略
- 人工录制转模板规则

### 11.2 沉淀载体

建议原型阶段至少生成：

- `project_knowledge/locators/`
- `project_knowledge/templates/`
- `project_knowledge/failure_fixes/`
- `project_knowledge/data_rules/`
- `promotion_candidates.json`

### 11.3 晋升规则

只有满足以下条件，才允许推荐晋升到平台级：

1. 至少跨多个功能点复用成功
2. 不依赖当前项目特有字段名或业务值
3. 有明确证据链
4. 通过人工审核

## 12. 分阶段开发里程碑

## 12.1 P0：脚本固化重构

目标：

- 将现有 `tools/suyuan_submit_loop.py` 从大脚本重构为“模板执行样本”。

输出：

- 执行模板定义
- 基线数据文件
- 数据 schema 文件
- 运行目录统一结构
- 简化后的执行入口

验收：

- 不依赖内联 `SUCCESS_BASELINE`
- AI-tester 与 Qwen 可复跑该模板
- 成功判定仍保持一致

- W4 现在已经能给出 `next_round_decision.json`，下一步最值得投入的是把它真正接成自动续跑，而不是继续只停留在结果文件层。
- human loop 与 data factory 现在已经能为模板草稿提供更强输入，后续重点会转到真实站点复杂组件打磨和项目级字段映射，而不是再做一轮纯骨架建设。
- 统一入口、日报和基线冻结 manifest 已经跑通，因此后续应该把精力更多放在“扩样本、扩模板、扩可复用性”，而不是重复补报告壳子。

这是当前围绕“泛化闭环”的四步执行计划。它不替代第 12 节的 P0-P6 里程碑，也不替代第 16 节的 W1-W7 工作包，而是作为当前验证模板泛化阶段的专门推进顺序。

   - 已把录制草稿 DSL 对齐到可执行模板动作

   - `suyuan_online_apply` 已抽出 wizard / drawer 共享动作族

   - 当前 registry 已开始转向“共享动作 + 少量项目胶水”的结构

   - 需要锁定模板动作名到 registry 的 contract

- `fill_success_template` 仍是明显的项目级耦合点
- 它属于 G2 之后剩余的“继续泛化候选”
- 但它不优先于 G3；当前正确节奏应是“先补护栏，再继续拆剩余耦合，再进入 G4”

## 14. 主要风险与修正建议

### 14.1 风险：过早追求“泛化智能”，导致原型重新变成难调的大 Agent

修正：

- 先固定平台骨架和项目模板边界
- 先把成功路径模板化
- 再扩探索与泛化

### 14.2 风险：人工录制被误当作最终自动化能力

修正：

- 录制只用于生成候选模板和基线
- 最终执行仍由模板引擎 + Playwright 承担

### 14.3 风险：数据生成失控，造成脏数据或不合法数据

修正：

- 数据 schema 强制声明约束
- 默认采用轻量扰动，不造复杂业务链条
- 高风险提交仍受白名单控制

### 14.4 风险：报告里混入“性能测试”口径

修正：

- 只输出执行效率观察
- 明确这不是压测结论

### 14.5 风险：项目级沉淀和平台级沉淀混淆

修正：

- 强制走沉淀分层
- 平台级必须人工审批

## 15. 编码开发前的审核点

建议你审核时重点看以下六项：

1. 是否同意“平台代码”和“项目模板”分仓内分层。
2. 是否同意优先做“执行模板 + 基线数据 + 数据工厂”，而不是先追求完整智能探索。
3. 是否同意把“人工录制”定义为“人工引导式基线采集器”。
4. 是否同意报告里的时间与速度部分只表述为“执行效率观察”。
5. 是否同意技能沉淀按“运行时 / 项目级 / 平台级候选”三层落盘。
6. 是否同意先用“线上备案申请”作为第一号项目模板样本推进编码。
