## 概述

本文档围绕 aut_agent 第二阶段 v4 的**目标循环（Goal Loop）**架构，从六个方向探索如何让测试程序通过循环实现自我进化。其中③④⑥三个方向做了深度展开。

核心命题来自需求分析 v4：

> 围绕单个菜单、页面或功能点建立目标循环，让系统自己在失败中收敛，在成功中沉淀。

## 方向①：逆向依赖推导 → 替代全量正向遍历

### 现状

当前 v4 菜单→页面→功能点的发现流程是**正向遍历**：从首页出发，逐级展开菜单，遍历页面，再识别功能点。

### 借鉴：小红书逆向链式推导

小红书的做法：从测试目标出发，反向推导依赖链。

```
正向（当前）：首页 → 逐级菜单 → 所有页面 → 识别功能点 → 挑低风险验证
逆向（借鉴）：目标功能点 → 反推页面入口 → 反推菜单路径 → 验证可达性
```

### 三步法

**第一步：意图三元组拆解**。将测试需求结构化为：
- Actions：Execute（执行什么操作）、Verify（验证什么结果）
- Entities：涉及哪些业务实体
- Attributes：实体需要满足的属性约束

**第二步：迭代式逆向推导**。从实体与属性出发，逐轮按需加载业务域知识：
- 能力预检：每个动作是否有对应接口/页面入口？
- 深度依赖构建：逐级倒推每个入参的来源，形成闭环依赖树

**第三步：生成调用计划**。将依赖树反转排序输出：前置构造 → 链路执行 → 结果验证。

### 对 aut_agent 的价值

- 不需要遍历整个菜单树，直接从测试意图出发
- 减少 80% 无效页面探索
- 与现有 `menu_goal → page_goal → feature_goal` 链兼容（逆向规划出的路径仍通过正向 goal 执行）

### 实施建议

在 preflight 阶段增加逆向推导步骤，优先查 L2 沉淀库中的历史路径，未命中再正向探索。

## 方向②：渐进式知识加载 → 替代 RAG/全量加载

### 问题

RAG（检索增强生成）在测试场景下有两个致命缺陷：
1. **场景不匹配**：RAG 擅长海量非结构化语料，但测试知识库文档有限、结构固定、按子域天然隔离
2. **召回不准代价过高**：一般问答漏召可二次追问，但规划阶段一旦召回错误片段，执行计划偏差无法中途纠正，只能整轮重跑

### 借鉴：小红书渐进式加载

按业务线与子域组织知识库目录树：

```
被测系统/
  ├── knowledge.md      （业务规则、跨域依赖、名词解释）
  ├── tool_list          （可用接口/页面入口描述）
  └── pitfall.md         （历史踩坑沉淀）
```

加载方式：从目标子域出发 → 发现依赖 → 追加加载对应子域 → 直至依赖链闭合。

### 对 aut_agent 的映射

```
被测系统A/
  ├── knowledge.md       — 业务规则、页面层级说明、名词解释
  ├── locator_hints.json — 稳定定位器、菜单壳特征、页面签名
  └── pitfall.md         — 隐性坑（如 Element Plus 固定列重复渲染）
```

每次目标循环只加载当前目标涉及的子域知识，不是把所有系统的全部知识一次性塞进上下文。

## 方向③：Debug-first 探测 → 提升首次命中率 ★ 已深挖

### 问题诊断

v4 套路动作表中以下失败与"定位不准"直接相关：`locator_unstable`、`feature_not_identified`、`menu_not_found`、`page_blank`。

当前流程："选定位器 → 执行 → 失败 → 分类 → 套路 → 换定位器 → 重试"。每轮失败都消耗 `max_rounds` 配额。

### 核心方案：四层探测金字塔

```
         ┌─────────────┐
         │  L4: 语义接管 │  ← Browser Use 兜底（最贵，~5s）
         ├─────────────┤
         │ L3: ARIA 探测 │  ← 无障碍树 + 角色语义（~500ms）
         ├─────────────┤
         │ L2: DOM 指纹  │  ← 元素特征签名 + 候选排序（~100ms）
         ├─────────────┤
         │ L1: 静态快照  │  ← aria_snapshot 全页元素清单（~200ms）
         └─────────────┘
```

- **L1**：一次 `page.accessibility.snapshot()` 获取全页可交互元素清单
- **L2**：对关键元素生成多维度指纹（文本、ARIA label、CSS class、DOM path、相邻文本、视口位置），按优先级排序候选定位器
- **L3**：利用 ARIA snapshot 提取 role=button/menu/link 等结构化语义
- **L4**：仅当前三层都无法定位时才启用 Browser Use，且带上探测上下文

### 与传统方法的区别

不是"单策略→失败→换策略"，而是**执行前把候选池排好序，执行时按序尝试，第一个命中即停**。

### 集成到目标循环

在每次 Attempt 前插入 `preflight_probe` 步骤，探测结果落盘到 `goal_attempts.jsonl`，失败时把探测上下文传给分类器。

### 参考技术

| 技术 | 用途 |
|------|------|
| Playwright `page.accessibility.snapshot()` | L1/L3 ARIA 探测 |
| Playwright `page.getByRole()` | L2 语义定位 |
| Healenium | 定位器自愈学习 |
| Browser Use 结构化输出 | L4 语义接管 |

## 方向④：失败分类的两层经验沉淀 ★ 已深挖

### 小红书两层沉淀的启发

| 层 | 粒度 | 存储位置 | 复用方式 |
|----|------|---------|---------|
| 工具级 | 单个接口的隐性坑 | 写回 toolPrompt | 下次调用自动带上 |
| 链路级 | 完整调用链的成功路径 | 脚本+元信息→向量库 | RAG 召回 |

关键设计：**工具级是"绕坑"（下次不踩），链路级是"抄作业"（下次复用）**。

### aut_agent 的三层映射

```
┌─────────────────────────────────────────────┐
│  L1 战术沉淀（目标级 / 当前 run）              │
│  作用域：同一个目标的后续尝试                   │
│  内容：本次探测结果、失败的定位器、有效替代策略   │
│  存储：goal_attempts.jsonl + goal_state.json  │
│  生命周期：目标结束即归档                       │
├─────────────────────────────────────────────┤
│  L2 项目沉淀（被测系统级）                     │
│  作用域：同一被测系统的所有后续目标              │
│  内容：稳定定位器库、菜单壳特征、页面签名、踩坑记录│
│  存储：locator_hints.json + pitfall.md        │
│  生命周期：被测系统生命周期内持续累积            │
├─────────────────────────────────────────────┤
│  L3 平台沉淀（跨系统级）                       │
│  作用域：所有被测系统的通用能力                 │
│  内容：通用失败模式库、通用套路动作、类型化模板   │
│  存储：平台级 skill / 套路动作注册表            │
│  生命周期：需人工审核晋升                       │
└─────────────────────────────────────────────┘
```

### L1 → L2 → L3 晋升条件

| 条件 | 阈值建议 |
|------|---------|
| 跨系统出现 | ≥ 2 个不同被测系统 |
| 稳定有效 | 成功率 ≥ 90%，使用次数 ≥ 10 |
| 人工审核 | 必须（平台级不能自动生效） |

### L2 三类核心资产

**A. 定位器偏好库**（`locator_hints.json`）：
```json
{
  "stable_locators": {
    "查询按钮": {
      "primary": "button[name=查询]",
      "confidence": 0.98,
      "failed_alternatives": [".search-btn", "#queryBtn"],
      "notes": "页面有两个同名按钮，优先用工具栏的"
    }
  }
}
```

**B. 踩坑录**（`pitfall.md`）：把隐性坑（Element Plus 固定列重复渲染、日期选择器联动限制等）写回被测系统文档。

**C. 成功动作片段**（`reusable_snippets`）：完整可复用的操作序列，带成功率统计。

### L2 消费方式

执行前先查 L2 沉淀 → 命中且置信度高则跳过探测直接用 → 成功则 confidence+1 → 失败则降权并触发重新探测。

## 方向⑤：Plan-and-Execute（宏观）+ ReAct（微观）

### 思路

当前 v4 目标循环接近纯 ReAct（尝试→失败→分类→套路→再尝试）。

改进：加一层目标级 Plan，形成**双层执行框架**。

```
Plan 层（目标级 / 宏观）：
  当前目标是什么 → 需要哪些证据 → 最大轮数 → 预期出口
  → 一次性规划，不因单次失败而重规划

Execute 层（尝试级 / 微观）：
  ReAct: 观察 → 动作 → 反馈 → 再观察
  → 有韧性，单步失败自动修正
```

### 参考

小红书采用相同模式：宏观用 Plan-and-Execute 保证方向不跑偏，微观用 ReAct 保证执行有韧性。

### 对 aut_agent 的映射

- Plan 层 = 目标注册 + 成功/停止条件 + `next_round_plan.json`
- Execute 层 = 目标尝试 + 失败分类 + 套路动作 + L1 战术修正

## 方向⑥：双向沉淀飞轮 ★ 已深挖

### 本质定义

小红书的核心飞轮："用得越多 → 脚本沉淀越多 → 召回命中率越高 → 执行越快"。

但这只是正向回路。真正的**双向**飞轮包含两条对称路径：

```
正向回路（成功加速）：
  执行成功 → 沉淀经验 → 下次更快命中 → 更多执行 → 更多成功 → 更多沉淀

反向回路（失败修正）：
  执行失败 → 分类归因 → 套路修正 → 沉淀教训 → 下次绕坑 → 成功率提升
```

**关键洞察**：两次失败不只是"重试"——同时贡献了 L1 战术修正、L2 项目沉淀、分类器训练数据、套路有效性统计。

### MAPE 控制循环（来自论文）

飞轮不能靠"自然积累"运转，必须有显式的 **MAPE** 控制循环：

```
┌─────────────────────────────────────────────┐
│  Monitor  →  Analyze  →  Plan  →  Execute   │
│    ↑                                   ↓     │
│    └────────── Knowledge Base ─────────┘     │
└─────────────────────────────────────────────┘
```

| 阶段 | aut_agent 对应 | 飞轮中的作用 |
|------|---------------|-------------|
| Monitor | 目标状态观测（goal_state, progress_events） | 采集"当前发生了什么" |
| Analyze | 失败分类器 + 涌现聚合器 | 判断"为什么会这样" |
| Plan | 套路动作选择 + 下一轮计划 | 决定"下一步怎么做" |
| Execute | Playwright / Browser Use 执行 | 验证"这样做对不对" |
| KB | L1/L2/L3 三层沉淀 | 记住"什么有效/什么无效" |

### 飞轮的三层嵌套

```
L3 平台飞轮（周/月级）：跨系统经验 → 人工审核 → 晋升平台能力 → 新系统受益
  └─ L2 项目飞轮（小时/天级）：同系统经验 → 自动沉淀 → 下次目标命中 → 置信度更新
       └─ L1 目标飞轮（秒/分钟级）：尝试 → 失败 → 分类 → 套路 → 再尝试
```

### 飞轮健康指标

| 指标 | 计算方式 | 健康阈值 |
|------|---------|---------|
| 首次命中率 | 执行前查 L2 命中的比例 | > 60% |
| 套路有效率 | 套路动作执行后目标推进的比例 | > 70% |
| 沉淀腐化率 | L2 沉淀中置信度持续下降的比例 | < 10% |
| 平均尝试次数 | 每个目标达成所需的平均 attempt 数 | 递减趋势 |

### 飞轮退化的四种防御

| 退化模式 | 表现 | 防御机制 |
|---------|------|---------|
| 腐化 | 系统升级后旧定位器失效 | 每次命中验证，失败降置信度 |
| 膨胀 | 沉淀库无限增长 | 按置信度+最后使用时间 LRU 淘汰 |
| 过拟合 | 过度适配单一系统 | 晋升 L3 前强制跨系统验证 |
| 噪声 | 偶发成功被当作稳定经验 | 置信度 ≥3 次成功才进入 L2 |

### 飞轮启动策略（冷启动）

新系统首次接入（飞轮为空）：
- **种子轮**：不做任何跳过，完整正向遍历 + 完整探测，每个成功动作都沉淀
- **加速轮**：先查 L2，命中则跳过探测直接复用，未命中才完整执行

**关键认知**：第一轮不是"浪费"，而是**投资**。每一轮都比上一轮更快。

### 热/温/冷数据分层

```
热数据（同 session / 近 3 天） → 自动复用，不验证 → 内存 + SQLite
温数据（3-30 天、同系统）      → 轻量验证（aria 对比）→ locator_hints.json
冷数据（>30 天、跨系统候选）    → 完整 preflight probe  → pitfall.md + skill
```

### 复盘时刻

每个目标完成后，增加显式复盘步骤：

1. 这次有效的是什么？→ 存入 L2 热数据
2. 这次踩了什么坑？→ 写入 pitfall.md
3. 这个经验只对这个系统有效，还是对所有系统有效？→ 决定 L3 晋升

### 参考论文

| 论文 | 关键启示 |
|------|---------|
| Adaptive Data Flywheel: MAPE Control Loops | 飞轮需要显式控制循环，不能靠自然积累 |
| Skill-as-Pseudocode | 沉淀的不是脚本片段，而是带条件的决策规则 |
| EVOR: Evolving Retrieval | 检索库需持续进化：淘汰过时、更新权重、合并重复 |
| AR-RAG: Safe Cache Routing | 缓存复用需要安全路由：判断当前上下文是否仍有效 |

## 实施优先级建议

基于 aut_agent 当前阶段（v4 阶段A-F，B/C/D/E/F 均已接入 main.py）：

```
第一优先：⑥ 飞轮 MAPE 骨架
  → 把 L1 目标循环的 M→A→P→E 跑通
  → 当前已有 goal_loop/resolution_writer.py，可在此基础上加 MAPE 显式循环

第二优先：③ Debug-first 探测
  → 在 menu_goal / page_goal 的 orchestrator 中加 preflight_probe
  → 降低 locator_unstable / menu_not_found 的触发频率

第三优先：④ L2 项目沉淀
  → 让 locator_hints.json 和 pitfall.md 从"手动维护"变成"自动累积"
  → 对接现有的 locator_hints 解析链

后续：
  ① 逆向推导 → 等阶段E执行闭环稳定后再引入
  ② 渐进式加载 → 当被测系统 ≥3 个时收益明显
  ⑤ P&E + ReAct → 当前 ReAct 模式已足够，等复杂目标增多再分层
```
