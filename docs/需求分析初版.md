# AI Agent 软件自动化测试项目需求分析初版

## 1. 项目背景

本项目面向第三方软件项目验收评测场景。平台作为独立评测方工具，用于辅助甲方验收乙方交付的软件系统，重点判断软件项目是否满足合同、需求说明书、功能说明书、性能指标与安全合规要求。

项目核心目标不是替代测试人员，而是利用 AI Agent 提升需求理解、测试设计、自动化执行、问题归因和报告生成效率，形成可复核、可追溯、可沉淀的评测过程。

## 2. 项目定位

本项目定位为 Web 平台形态的 AI Agent 软件自动化测试与验收评测系统。

第一阶段优先聚焦 Web UI 系统的自动化评测，围绕自然语言需求、需求说明书和功能说明书生成测试任务，驱动浏览器执行功能验证，并输出结构化评测报告。后续可扩展 API 测试、性能测试、安全扫描、移动 App 测试和 Windows 独立应用测试。

## 3. 目标用户

- 第三方测评机构测试工程师
- 甲方项目验收负责人
- 乙方交付团队或整改负责人
- 项目管理人员

## 4. 典型使用场景

1. 甲方或第三方上传被测系统的需求说明书、功能说明书、验收标准等资料。
2. 平台利用 AI Agent 解析资料，提取功能点、验收项、测试场景、约束条件和风险点。
3. 测试人员确认或调整 AI 生成的测试计划和测试用例。
4. 平台基于 Web UI 自动化能力执行测试。
5. 平台记录执行过程、页面截图、失败证据、操作路径和 Agent 判断依据。
6. 平台生成验收评测报告，说明功能符合情况、缺陷问题、性能表现、安全风险和整改建议。
7. 对于可自动化沉淀的场景，平台生成可维护的测试代码，供后续回归测试或压力测试复用。

## 5. 第一阶段范围

### 5.1 优先实现

- Web 平台作为用户使用入口。
- 支持上传或录入自然语言需求、需求说明书、功能说明书等资料。
- 从文档中提取功能点、验收项和测试场景。
- 面向 Web UI 系统生成测试计划和测试用例。
- 基于浏览器自动化执行 Web UI 测试。
- 记录测试执行证据，包括操作日志、截图、页面状态、失败原因等。
- 输出结构化评测报告。
- 对失败项给出缺陷描述和整改建议。
- 对部分场景生成自动化测试代码，为后续回归测试和压力测试复用做准备。

### 5.2 第一阶段需要评估但不一定完整实现

- API 测试：如果需求说明书中包含接口文档或 OpenAPI 文档，可先设计扩展点。
- 性能测试：第一阶段可输出性能测试建议或基础指标采集，完整压测能力可后置。
- 安全测试：第一阶段可集成基础安全风险检查或生成安全测试清单，完整漏洞扫描能力可后置。
- 移动 App 测试：需要额外引入 Appium、真机/模拟器管理、安装包管理、移动端控件识别和设备兼容性矩阵，建议推后。
- Windows 独立应用测试：需要额外引入桌面自动化框架、控件树识别、窗口管理和图像识别能力，建议推后。

### 5.3 第一阶段暂不实现

- 自动修改乙方系统代码。
- 自动提交修复补丁。
- 无人工确认的生产环境破坏性操作。
- 完整移动端 App 自动化测试平台。
- 完整 Windows 桌面应用自动化测试平台。
- 完整安全渗透测试平台。
- 完整性能压测平台。

### 5.4 关于移动 App 和 Windows 独立应用的范围决策

经讨论确认，第一阶段不将移动 App 和 Windows 独立应用的完整自动化执行能力纳入核心交付范围。

第一阶段对移动 App 和 Windows 独立应用的支持范围为：

- 支持在需求解析阶段识别目标系统是否包含移动 App 或 Windows 独立应用。
- 支持为移动 App 和 Windows 独立应用生成功能测试点、验收项和手工测试用例。
- 支持人工上传移动 App 或 Windows 独立应用的测试证据，如截图、录屏、日志和缺陷描述。
- 支持在最终评测报告中汇总移动 App 和 Windows 独立应用的测试范围、执行结果、风险项和整改建议。
- 在架构上预留移动端自动化执行器和桌面端自动化执行器扩展点。

第一阶段不承诺：

- 移动 App 的完整自动化执行。
- Windows 独立应用的完整自动化执行。
- 移动端多设备、多系统版本兼容性矩阵。
- iOS 签名、证书、真机设备池等复杂环境管理。
- Windows 桌面控件树、窗口句柄、系统弹窗、托盘程序、安装包和运行环境的完整自动化管理。

若甲方强制要求第一阶段覆盖移动 App 或 Windows 独立应用，建议仅作为专项可行性验证或冒烟测试能力处理，不应承诺完整验收自动化能力。

## 6. 输入来源

第一阶段主要输入包括：

- 自然语言需求描述。
- 项目需求说明书。
- 功能说明书。
- 验收标准。
- 用户手册或操作说明。
- 被测系统访问地址。
- 测试账号、角色和权限说明。
- 可选的接口文档、数据库说明、部署说明和历史缺陷资料。

后续可扩展输入包括：

- OpenAPI/Swagger 文档。
- 原型图或页面截图。
- 现有测试用例。
- 现有自动化测试代码。
- 线上日志或监控数据。

## 7. 输出产物

第一阶段主要输出：

- 验收评测报告。
- 功能符合性矩阵。
- 测试计划。
- 测试用例清单。
- 测试执行记录。
- 缺陷清单。
- 失败原因分析。
- 整改建议。
- 测试证据包，包括截图、操作日志、页面快照和执行 trace。

需要预留能力：

- 自动化测试代码生成。
- 回归测试脚本。
- 性能测试场景脚本。
- 标准化报告导出，如 HTML、PDF、Word、Excel 或 JUnit XML。

## 8. 核心功能需求

### 8.1 项目管理

- 创建评测项目。
- 配置项目基本信息、被测系统地址、测试范围、验收标准和评测周期。
- 管理测试环境、测试账号和访问凭据。
- 维护甲方、乙方、第三方测评人员信息。

### 8.2 文档解析与需求理解

- 上传需求说明书、功能说明书、验收标准等文档。
- 自动提取功能模块、功能点、业务规则、角色权限、异常场景和验收条件。
- 支持人工确认、编辑和补充 AI 提取结果。
- 生成需求到测试项的追踪关系。

### 8.3 测试计划与用例生成

- 基于需求和验收标准生成测试计划。
- 生成正常流程、异常流程、边界条件和权限相关测试用例。
- 为每个测试用例生成测试目标、前置条件、操作步骤、预期结果和判定标准。
- 支持人工审核、修改、启用或禁用测试用例。

### 8.4 Web UI 自动化执行

- 基于浏览器自动化访问被测系统。
- 支持登录、导航、表单填写、按钮点击、搜索、筛选、提交、下载等常见操作。
- 支持页面元素识别、等待、断言和失败重试。
- 记录每一步操作、页面状态、截图和错误信息。
- 支持多角色、多账号、多流程测试。

### 8.5 结果分析与报告生成

- 汇总测试执行结果。
- 标记通过、失败、阻塞、未执行和需人工确认等状态。
- 分析失败原因，包括功能缺失、页面异常、权限问题、数据问题、环境问题、定位失败或断言不满足。
- 生成验收结论、风险评级和整改建议。
- 支持报告导出。

### 8.6 测试代码生成

- 对稳定且可自动化的 Web UI 测试场景生成测试代码。
- 生成代码应包含清晰的步骤、断言和必要注释。
- 生成代码应可用于后续回归测试。
- 为后续性能测试或批量执行预留接口。

### 8.7 整改建议

- 对失败或风险项提供自然语言整改建议。
- 建议内容包括问题表现、可能原因、影响范围、建议修复方向和复测建议。
- 第一阶段不自动修改被测系统代码。

## 9. 非功能需求

### 9.1 可追溯性

- 需求、测试用例、执行结果、缺陷和报告之间应建立追踪关系。
- 每个评测结论应能追溯到对应需求、测试步骤和执行证据。

### 9.2 可复核性

- AI 生成的测试计划、测试用例、评测结论和整改建议应支持人工审核。
- 报告中的关键结论应保留依据。

### 9.3 安全性

- 测试账号、密码、Token 等敏感信息应加密存储或安全托管。
- 平台应限制 Agent 执行高风险操作。
- 对生产环境测试应有明确确认机制。

### 9.4 稳定性

- Web UI 自动化执行应具备等待、重试、失败截图和错误恢复能力。
- 应识别由于环境不稳定、网络波动或页面加载延迟导致的非功能缺陷失败。

### 9.5 可扩展性

- 架构应预留 API 测试、性能测试、安全测试、移动端测试和桌面端测试扩展点。
- Agent 工具层和测试执行层应解耦。

## 10. 技术方案初步设想

### 10.1 总体架构

建议采用分层架构：

1. Web 平台层：负责项目管理、资料上传、用例审核、任务执行和报告查看。
2. 文档理解层：负责解析需求说明书、功能说明书和验收标准。
3. Agent 编排层：负责规划、探索、生成、执行、分析和报告。
4. 测试执行层：负责调用浏览器自动化、接口测试、性能测试或安全检查工具。
5. 测试资产层：负责存储需求、用例、脚本、截图、日志、trace 和报告。
6. 报告层：负责生成验收报告、缺陷清单、风险清单和证据包。

### 10.2 Agent 分工

第一阶段可设计以下 Agent：

- Requirements Analyst Agent：解析需求文档，提取功能点和验收项。
- Test Planner Agent：生成测试计划和覆盖策略。
- Test Case Designer Agent：生成测试用例。
- Web Explorer Agent：探索被测 Web 系统页面和操作路径。
- Test Executor Agent：执行自动化测试。
- Result Analyst Agent：分析失败原因和风险等级。
- Report Writer Agent：生成评测报告。

后续可扩展：

- API Test Agent。
- Performance Test Agent。
- Security Test Agent。
- Mobile Test Agent。
- Desktop App Test Agent。

### 10.3 推荐技术栈

Web UI 自动化执行层优先考虑 Playwright。原因是 Playwright 对现代 Web 应用支持较好，具备定位器、自动等待、trace、截图、录制和跨浏览器能力，适合与 AI Agent 结合。

Web 平台可根据现有团队技术栈选择：

- 前端：React、Vue 或现有前端框架。
- 后端：Python FastAPI、Node.js/NestJS 或现有后端框架。
- 数据库：PostgreSQL 或 MySQL。
- 任务队列：Celery、BullMQ、RQ 或其他异步任务框架。
- 文件存储：本地对象存储、MinIO 或云对象存储。
- 报告：HTML 报告、PDF/Word 导出、Excel 缺陷清单、JUnit XML。

### 10.4 移动 App 和 Windows 应用扩展评估

范围结论：第一阶段以 Web UI 自动化评测为核心能力。对于移动 App 和 Windows 独立应用，第一阶段支持需求解析、测试计划生成、人工测试用例生成、测试证据管理和报告汇总，不承诺完整自动化执行；系统架构应预留移动端和桌面端自动化执行器扩展能力。

如果目标系统包含手机端 App，需要增加：

- Appium 或同类移动自动化框架。
- Android/iOS 真机或模拟器管理。
- 安装包上传、安装、卸载和版本管理。
- 移动端控件树识别、手势操作、权限弹窗处理。
- 多设备、多系统版本兼容性测试矩阵。
- 短信、定位、相机、文件、推送、权限弹窗等系统能力模拟。
- iOS 场景下的签名、证书、开发者账号和 WebDriverAgent 环境管理。

建议：移动 App 测试复杂度明显高于 Web UI，第一阶段只做架构预留和调研，不作为核心交付。

如果目标系统包含 Windows 独立应用，需要增加：

- Windows UI Automation、WinAppDriver、FlaUI、pywinauto 或图像识别方案。
- 窗口句柄、控件树、菜单、弹窗、文件对话框处理。
- 安装包管理、运行环境管理和桌面会话管理。
- 与 Web UI 不同的交互模型和稳定性处理机制。
- 分辨率、缩放比例、输入法、权限、杀毒软件干扰等运行环境差异处理。
- 对不可访问控件的图像识别或人工辅助处理。

建议：Windows 独立应用自动化与 Web UI 技术路线差异较大，第一阶段不纳入核心交付，仅在需求说明书中作为后续扩展方向。

## 11. 初步里程碑

### 阶段一：需求分析与方案设计

- 明确项目边界和验收场景。
- 明确第一阶段测试对象和输入输出。
- 完成需求说明书。
- 完成技术方案设计。
- 完成原型流程设计。

### 阶段二：MVP 原型

- 实现项目创建和资料上传。
- 实现需求文档解析和功能点提取。
- 实现测试用例生成与人工审核。
- 实现基础 Web UI 自动化执行。
- 实现基础评测报告生成。

### 阶段三：测试资产沉淀

- 增加测试代码生成。
- 增加执行 trace、截图和证据包。
- 增加用例复用和回归测试。
- 增加结果对比和趋势分析。

### 阶段四：能力扩展

- 扩展 API 测试。
- 扩展基础性能测试。
- 扩展基础安全风险检查。
- 评估移动端和 Windows 应用自动化。

## 12. 待确认问题

1. 第一阶段是否只支持 Web UI，还是必须包含 API 测试的最小能力。
2. 被测系统是否通常能提供测试环境和稳定测试账号。
3. 需求说明书和功能说明书的主要格式是 Word、PDF、Excel，还是在线文档。
4. 报告是否需要符合某个固定模板或行业规范。
5. 是否需要对接甲方现有项目管理、缺陷管理或 CI/CD 系统。
6. 是否需要支持离线部署或私有化部署。
7. 是否允许平台调用外部大模型 API，还是必须本地化部署模型。
8. 评测过程中是否允许 Agent 自动提交、删除、审批等高风险业务操作。
9. 性能测试在第一阶段的最低要求是什么，是指标采集、压测脚本生成，还是完整压测执行。
10. 安全风险评估在第一阶段的最低要求是什么，是安全清单、基础扫描，还是漏洞验证。
