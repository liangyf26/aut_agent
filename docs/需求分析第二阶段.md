# AI Agent 软件自动化评测平台需求分析书（第二阶段）

## 1. 文档目的

本文档用于定义本项目第二阶段的需求边界、原型目标和技术方案约束。

第二阶段的目标不是替代第一阶段现有 Node.js 主线平台，也不是直接形成正式验收平台，而是先完成一个以 Python 为核心的执行子系统原型，用于验证以下能力是否成立：

- 在不依赖需求说明书和功能说明书的前提下，自动发现当前测试账号权限范围内的 Web UI 导航结构。
- 基于页面入口、页面元素和操作路径推断出可测功能点。
- 按固定模板对单个功能点执行基础路径验证。
- 产出可复核的结构化测试证据和可阅读的测试报告。

## 2. 第二阶段定位

### 2.1 定位结论

第二阶段定位为：

**面向 Web UI 系统的 Python 执行子系统原型，用于进行页面入口发现、功能点识别、单功能点基础路径验证和结构化测试证据输出。**

### 2.2 与第一阶段的关系

- 第一阶段以需求文档、功能说明书和验收项为输入，构建需求驱动的评测闭环。
- 第二阶段不依赖需求说明书和功能说明书作为主要输入，而以目标系统的实际页面结构和当前测试账号权限范围为主要探索对象。
- 第二阶段当前不替换现有 Node.js 主线和平台层，仅验证 Python 执行子系统的能力边界。

### 2.3 第二阶段不等于自动验收

第二阶段默认不自动给出“业务正确性”和“验收符合性”的结论。

第二阶段自动判定的重点仅限于：

- 页面入口是否可见、可达、可进入
- 页面是否可稳定加载
- 关键控件是否可识别、可操作
- 单功能点基础路径是否可走通
- 页面反馈是否与当前动作一致
- 是否存在明显异常、权限阻断、前端报错或接口失败

对业务规则是否正确、数据结果是否满足实际验收要求等问题，默认标记为：

**需人工确认**

## 3. 第二阶段目标

第二阶段原型需要实现以下目标：

1. 对当前测试账号权限范围内的 Web UI 页面入口进行自动发现。
2. 对页面入口下的默认可见功能点和轻量交互后显式出现的功能点进行识别。
3. 基于功能点类型生成并执行执行型测试用例。
4. 记录测试过程日志、关键步骤截图和状态结果。
5. 输出结构化产物和一份可直接阅读的总体测试报告。

## 4. 第二阶段输入

第二阶段必须输入：

- 被测系统访问地址
- 测试账号
- 登录方式说明

第二阶段可选输入：

- 测试环境说明
- 可自动执行的高风险操作白名单
- 前置测试数据说明
- 人工登录接管说明
- 系统特殊提示，如必须先进入某首页、必须先关闭某弹窗、菜单层级说明

### 4.1 测试账号范围

第二阶段仅验证**所提供测试账号**权限范围内可见、可达、可操作的页面入口和功能点。

当前阶段不默认构建多角色矩阵，不推断缺失角色的覆盖范围。

### 4.2 登录模式

第二阶段支持两种登录模式：

1. 自动登录模式  
   适用于账号密码直登，或项目方提供了可自动完成的验证码/测试登录方式。

2. 登录接管模式  
   当登录过程无法自动完成时，由人工先完成登录，Agent 从已登录首页或当前已登录页面开始执行自动遍历与测试。

## 5. 第二阶段输出

第二阶段输出拆为两层：

1. 页面入口层
2. 功能点层

### 5.1 页面入口层输出

- 页面入口清单
- 页面入口层级关系
- 页面入口可达性状态
- 页面入口对应的关键截图

### 5.2 功能点层输出

- 功能点清单
- 功能点所属页面入口
- 功能点类型
- 功能点执行型测试用例
- 功能点执行结果
- 功能点关键截图
- 需人工确认标记

### 5.3 结构化产物

原型阶段至少需要完整产出以下结构化产物：

- 页面入口清单
- 功能点清单
- 执行型测试用例和测试过程日志
- 截图索引
- 汇总结果数据

### 5.4 可阅读报告

原型阶段需要产出一份可直接阅读的总体测试报告，格式可为 Markdown 或 HTML。

当前阶段暂不追求：

- Word/PDF 正式模板
- 行业规范报告版式
- 多格式导出矩阵

此外，原型阶段还必须提供一套运行时可视化产物，用于让人工直观看到平台当前运行到哪个流程环节、是否等待人工、下一步准备做什么。

## 6. 核心术语和边界

第二阶段以 `CONTEXT.md` 中的 glossary 为准，关键术语包括：

- 页面入口
- 功能点
- 执行型测试用例
- 探索边界
- 高风险操作
- 前置测试数据
- 两段式执行
- 人工审核开关
- 运行态进度视图
- 人工接管恢复包
- 探索式状态分类
- 关键步骤截图

## 7. 第二阶段核心原则

### 7.1 仅聚焦 Web UI

第二阶段只聚焦 Web UI 系统，不覆盖：

- API 自动化测试
- 移动端 App 自动化测试
- Windows 桌面应用自动化测试
- 性能压测
- 安全渗透测试

### 7.2 有限深度探索

第二阶段采用**有限深度探索**策略。

系统仅在当前测试账号权限范围内，对以下对象进行探索：

- 菜单入口
- 页面入口
- 默认可见功能点
- 轻量交互后显式出现的功能点

系统不做：

- 深层数据穷举
- 无限递归探索
- 全分页遍历
- 全筛选条件穷举
- 全树节点展开
- 为追求覆盖率而自主构造复杂业务链路数据

### 7.3 高风险操作默认禁止

第二阶段默认禁止高风险真实提交。

默认允许：

- 导航
- 查询
- 筛选
- 分页
- 查看详情
- 打开弹窗
- 表单填写但不最终提交
- 触发前端校验
- 非破坏性下载

默认禁止：

- 删除
- 审批
- 支付
- 发布
- 出库
- 结算
- 作废
- 批量操作
- 真实落库提交
- 触发外部通知、短信、邮件、工作流的动作

仅在项目级白名单显式允许时，才可在测试环境中自动执行高风险真实提交。

### 7.4 不负责复杂造数

第二阶段默认不负责自主构造复杂业务前置数据。

系统仅验证当前环境和已有测试数据下可见、可达、可操作的功能点。

对于因缺少前置数据而无法验证的功能点，需要在报告中明确标记。

### 7.5 原型阶段默认自动继续，但预留人工审核

第二阶段支持“默认自动继续”，但必须保留人工审核开关。

发现阶段结束后，系统应支持人工：

- 审核页面入口清单
- 审核功能点清单
- 禁用功能点
- 补充功能点
- 重命名功能点

原型阶段默认可采用“发现后直接进入验证”的自动模式。

## 8. 两段式执行模式

第二阶段采用：

**先发现，后验证**

### 8.1 发现阶段

发现阶段的目标是：

- 自动遍历页面入口
- 构建页面地图
- 识别功能点
- 产出页面入口清单和功能点清单

发现阶段的输出必须可增量落盘。

### 8.2 验证阶段

验证阶段的目标是：

- 按功能点逐个执行基础路径验证
- 生成执行型测试用例
- 记录日志和截图
- 产出探索式状态结果

### 8.3 每个功能点的起点规则

验证阶段在同一登录会话内执行，但每个功能点验证前应尽量回到所属页面入口的标准起点。

标准起点至少包括：

- 关闭弹窗
- 返回功能点所属页面入口
- 清理明显筛选条件或临时 UI 状态
- 确认页面加载完成

## 9. 页面入口和功能点识别规则

### 9.1 页面入口识别

页面入口指可被导航访问的菜单项、页面入口或功能入口。

第二阶段应识别：

- 一级菜单
- 二级菜单
- 三级菜单
- 页面内显式导航入口

### 9.2 功能点识别

功能点定义为：

**用户可感知、可独立触发、可独立验证结果的一次业务交互目标**

不将以下对象单独算成功能点：

- 输入框本身
- 按钮图标本身
- 标签或展示字段
- 列标题

### 9.3 功能点示例

例如在“用户管理”页面：

- 页面入口：用户管理
- 功能点：查询用户、新增用户、编辑用户、删除用户、导出用户

## 10. 功能点类型模板

第二阶段按功能点类型定义固定的最小验证模板。

每类功能点只验证一条基础路径。

### 10.1 第一批支持的功能点类型

建议原型阶段优先支持以下类型：

1. 导航类
2. 查询/筛选类
3. 详情查看类
4. 新增类
5. 编辑类
6. 删除类
7. 导出/下载类
8. Tab/切换类

### 10.2 最小验证模板建议

#### 导航类

- 进入目标页面
- 确认页面加载成功

#### 查询/筛选类

- 输入条件或使用默认条件
- 触发查询
- 确认页面反馈发生变化

#### 详情查看类

- 打开详情页或详情弹窗
- 确认关键内容可见

#### 新增类

- 打开新增界面
- 填写最小必填字段
- 触发前端校验或非高风险提交前验证

#### 编辑类

- 打开编辑界面
- 修改一个低风险字段
- 触发前端校验或非高风险提交前验证

#### 删除类

- 验证删除入口存在
- 验证确认弹窗或安全拦截存在
- 默认不执行最终删除

#### 导出/下载类

- 触发下载动作
- 确认浏览器响应或下载事件出现

#### Tab/切换类

- 执行切换
- 确认内容区域发生变化

## 11. 探索式状态分类

第二阶段采用探索式状态分类，而不是直接输出“验收通过/失败”。

### 11.1 页面入口状态

- 已发现
- 可进入
- 进入失败
- 受限不可访问
- 未验证完成

### 11.2 功能点状态

- 已发现未执行
- 执行通过
- 执行失败
- 受前置数据限制未执行
- 受安全策略限制未执行
- 需人工确认

### 11.3 说明

这里的“执行通过”仅表示：

- 可达
- 可操作
- 基础路径可走通

不表示业务验收通过。

## 12. 关键步骤截图策略

第二阶段采用固定的截图触发规则，不由 Agent 自由决定。

### 12.1 必须截图的场景

1. 首次进入每个页面入口时
2. 首次开始执行每个功能点时
3. 功能点执行后出现关键反馈时
4. 发生异常时

### 12.2 关键反馈包括

- 成功提示
- 失败提示
- 校验错误
- 页面跳转结果

### 12.3 不要求的情况

- 不要求每一步都截图
- 不要求对重复重试的同一步骤全部截图
- 同一功能点重复重试时，仅在页面状态明显变化时补充截图

## 13. 测试过程日志与执行型测试用例

第二阶段的“用例”不是从需求文档预先生成的标准验收用例，而是探索与执行过程中即时生成的执行型测试用例。

每条执行型测试用例至少应包含：

- 所属页面入口
- 功能点名称
- 功能点类型
- 测试目标
- 前置条件
- 执行动作步骤
- 页面反馈
- 自动判定结果
- 是否需人工确认

## 14. 结构化产物建议

原型阶段建议采用结构化文件落盘，供后续平台集成。

建议最小产物目录如下：

```text
artifacts/
  page_entries.json
  feature_points.json
  executions.json
  screenshots/
  report.md
  report.html
```

可根据实现需要增加：

- run_meta.json
- downloads/

- trace/

## 15. 原型阶段推荐技术方案

### 15.1 技术栈

第二阶段原型采用 Python 技术栈，核心依赖建议为：

- Browser Use：用于高层探索、页面理解、功能点推断
- Playwright Python：用于稳定的浏览器控制、截图、下载监听、页面状态获取和必要的确定性操作

### 15.2 角色分工建议

为降低 Browser Use 完全自由执行带来的不可控性，建议采用以下分工：

- Browser Use 负责发现阶段的页面理解、入口推断、功能点推断、执行意图规划
- Playwright 负责低层确定性执行，如导航、点击、输入、等待、截图、下载监听、页面状态回收

### 15.3 原型形态建议

第二阶段原型建议优先采用：

**Python CLI/任务执行器 + 结构化产物目录 + 可阅读报告**

暂不要求：

- 完整 Web 平台接入
- 完整前后端统一调度
- 正式多用户任务管理

## 16. 非功能需求

### 16.1 可复核性

- 每个页面入口和功能点都要有来源和执行证据
- 人工能够通过日志、截图和页面反馈理解系统为何给出当前状态

### 16.2 稳定性

- 探索和验证过程应具备超时、重试、去重、循环检测和页面状态恢复能力
- 失败必须能够落盘，不允许因中途中断而丢失全部结果

### 16.3 安全性

- 默认禁止高风险真实提交
- 测试账号凭据需从环境变量或安全配置中读取，不写死在代码中
- 对人工登录接管模式，应避免在日志中泄露敏感会话信息

### 16.4 可恢复性

- 中断后应能基于已有结构化产物继续执行
- 页面入口清单和功能点清单应支持增量续跑
- 命中人工审核或人工接管时，应能生成结构化恢复包并支持按恢复命令继续下一轮执行

### 16.5 运行态可观测性

- 运行过程中应持续输出平台级进度事件，而不只是浏览器操作日志
- 人工应能看到当前阶段、当前步骤、当前轮次、当前对象、阻塞原因和下一步动作
- 运行态进度视图应支持 CLI 观察与结构化落盘，供后续前端接入

## 17. 第二阶段原型的主要漏洞与修正建议

以下问题是第二阶段方案中最可能导致原型失败或结果失真的点。

### 17.1 漏洞：无需求文档时容易误把“可操作”当“验收通过”

修正建议：

- 报告层面严禁把“执行通过”解释为“业务通过”
- 对业务正确性统一标记为需人工确认

### 17.2 漏洞：页面入口和功能点识别容易过粗或过细

修正建议：

- 固定“页面入口”和“功能点”的定义边界
- 保留人工审核开关

### 17.3 漏洞：Agent 自由度过高，可能偏离既定验证模板

修正建议：

- 使用固定功能点类型模板
- 高风险动作不只靠 prompt 约束，必须在执行层硬拦截

### 17.4 漏洞：发现阶段和验证阶段混在一起会污染状态

修正建议：

- 严格采用两段式执行
- 功能点验证前尽量回到标准起点

### 17.5 漏洞：同一账号下的结果可能被当前测试数据状态误导

修正建议：

- 对缺失前置数据导致的未验证结果明确标记
- 不让 AI 自主造复杂业务数据

### 17.6 漏洞：登录过程可能成为真实系统中的第一阻塞点

修正建议：

- 只承诺支持可自动完成登录的系统
- 对不支持自动登录的系统使用登录接管模式

### 17.7 漏洞：截图过多会导致报告冗长、存储膨胀

修正建议：

- 使用固定关键步骤截图规则
- 不做全量逐步截图

### 17.8 漏洞：本地或第三方 LLM 兼容性不稳定

修正建议：

- 在执行前增加 LLM Provider 预检
- 明确记录模型是否支持结构化输出、视觉能力和响应超时
- 对不兼容模型提前失败，不进入正式任务

### 17.9 漏洞：Browser Use 直接操作文件、待办清单或自由工具调用可能污染原型流程

修正建议：

- 对可用工具做白名单约束
- 尽量将浏览器外部副作用操作下沉为受控执行器能力

### 17.10 漏洞：报告如果只保留流水日志，不利于人工确认

修正建议：

- 同时保留结构化产物和可阅读报告
- 报告中显式列出页面入口、功能点、状态、截图索引和未执行原因

### 17.11 漏洞：人工接管如果没有结构化恢复上下文，容易在下一轮续跑时丢失状态

修正建议：

- 命中人工接管时生成结构化恢复包
- 恢复包中显式列出待处理动作、恢复命令、下一轮目标阶段和备注
- 恢复续跑结果继续写回同一套结构化产物体系

## 18. 原型阶段建议里程碑

### 阶段 2.1：执行子系统原型

- 完成单账号登录/接管
- 完成页面入口发现
- 完成功能点识别
- 完成最小验证模板执行
- 完成截图和日志落盘
- 完成 Markdown/HTML 汇总报告

### 阶段 2.2：稳定性增强

- 增加循环检测、超时恢复、重试与状态恢复
- 增加人工审核开关
- 增加前置数据限制标记

### 阶段 2.3：平台接入准备

- 规范结构化产物格式
- 设计与 Node.js 平台的数据交换边界
- 评估后续集成方式

## 19. 第二阶段原型验收标准建议

如果以下条件同时满足，可视为第二阶段原型成立：

1. 能在至少一个真实 Web UI 系统上完成页面入口发现。
2. 能识别并落盘功能点清单。
3. 能对至少三类功能点完成基础路径验证。
4. 能输出关键步骤截图。
5. 能输出结构化测试日志和执行型测试用例。
6. 能生成一份可阅读的总体测试报告。
7. 报告中的状态分类、未执行原因和人工确认标记清晰可见。
8. 运行过程中能提供平台级进度视图或等价结构化进度产物。
9. 命中人工接管时，能生成恢复包并支持继续下一轮执行。

## 20. 本阶段暂不实现

- 基于需求说明书和功能说明书的需求解析
- 标准化验收结论自动生成
- 多角色账号矩阵自动编排
- 复杂业务数据自动构造
- 正式 Word/PDF 报告模板导出
- API、移动端、桌面端自动化能力
- 直接替换现有 Node.js 主平台
